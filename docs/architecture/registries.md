# Canonical registries + flavor registry (implementation)

Code-level implementation rules for the canonical registry system (`lib/*-registry.ts` validation mirrors, alias maps, write-time enforcement, migration provenance) + the flavor-notes registry. **Claude-Code-facing reference — read on demand when touching a registry; not MCP-registered.** The *glossary* cut (term definitions + cardinality, claude.ai-facing) lives at [docs/reference/canonical-registries.md](../reference/canonical-registries.md); this file is the implementation companion it points to. Extracted from CLAUDE.md 2026-06-02 (root-doc compaction) — move-never-delete.

- **Extraction strategies (Axis 1):** `Suppression`, `Clarity-First`, `Balanced Intensity`, `Full Expression`, `Extraction Push`, `Hybrid` — 6 canonical post v8.4 Hybrid promotion (2026-05-06; predecessor: Extraction Strategy v2 / migration 034 / 2026-04-27 shipped the first 5). Canonical list in [lib/brew-import.ts](lib/brew-import.ts). Mechanics-vs-intent symmetry: Suppression and Clarity-First share coarse/low-temp/low-agitation mechanics but differ in intent (Suppression holds an over-expressive co-ferment back; Clarity-First protects a delicate cup). Full Expression and Extraction Push share fine-grind + high-temp mechanics but differ in agitation: Full Expression is high agitation (develop heavy co-ferments), Extraction Push uses Melodrip / low agitation (push yield on a clean coffee while preserving transparency — Wölfl/Tran/Giachgia pattern). Hybrid is structurally distinct — sits on the structural-composition axis with 5 enum sub-forms (sequential / phase_mapped / selective_bloom / intensity_clarity_split / temperature_staged) and operates with intensity-strategy choices INSIDE each phase. Hybrid requires `hybrid_subform` set on the brew; the other 5 leave `hybrid_subform` null. All 55 brews populated; Inmaculada reclassified to Suppression in migration 034 (Basha Bekele Kokose lands when the bean uploads). `extraction_confirmed` is optional free-text recorded only when the planned strategy diverged from what was tasted.
- **Extraction modifiers (Axis 2):** `brews.modifiers jsonb` array of `{type, ...subfields}` entries. **5 canonical types post Sub-sprint 4c (2026-05-28):** `output_selection` (form: early_cut/late_cut/both/dilution, brew_weight, cup_yield, dilution_g, notes), `thermal_staging` (phases — free-text covering BOTH the kettle thermal stance "kettle off after bloom, natural cooling" / "on-base constant" AND active ramps "86°C → 92°C across two phases"), `aroma_capture` (application), `role_based_pulse` (roles — v8.5 promotion 2026-05-08 from the WBC Time Distribution Playbook's percolation sub-case), `equipment` (name + scope — persistent/timed gear beyond brewer+filter: Melodrip / booster / Paragon ball; scope is free-text "throughout" / "bloom + P1"; 4c addition). Optional + stackable; default empty (most brews have none — modifiers are deliberate additions). Canonical shape in [lib/extraction-modifiers.ts](lib/extraction-modifiers.ts) with `cleanModifiers()` validator + `composeModifierLabel()` display helper + `emptyModifier(type)` for picker seeding. Modifiers fold four of the WBC taxonomy's five axes (Output Selection / Time Distribution / External Control / Immersion-Hybrid) into one heterogeneous bucket — Chris's single-origin context doesn't combine them in ways that warrant per-axis sub-bucketing yet. **Lineage note**: `immersion` was a 4th modifier added Sprint 2.1 (2026-04-27) but was **retired in v8.4 (2026-05-06)** when Hybrid was promoted to a full strategy — the immersion modifier's semantics absorbed into Hybrid via `hybrid_subform` set; net-new immersion-shaped brews now persist as `extraction_strategy: "Hybrid"` with a `hybrid_subform`. `role_based_pulse` was added in v8.5 (2026-05-08). **4c (2026-05-28):** `inverted_temperature_staging` RENAMED to `thermal_staging` (ALIAS-SAFE — `cleanModifiers` normalizes the legacy name via `MODIFIER_TYPE_ALIASES` + `push_brew` still accepts it, so stored rows + stale claude.ai payloads round-trip with no data migration); `equipment` added as the 5th type. See [docs/skills/wbc-brewing-archivist/cluster/wbc-reference.md](docs/skills/wbc-brewing-archivist/cluster/wbc-reference.md) for the comprehensive 5-axis + 8-family map + "consciously not pursuing" appendix (migrated from BREWING.md § WBC Reference in Wave 2 PR 1).

- **Canonical-registry factory:** [lib/canonical-registry.ts](lib/canonical-registry.ts) exports `makeCanonicalLookup(names, aliases?)` returning a `CanonicalLookup` bundle (`list` / `isCanonical` / `findClosest`) used by flavor / producer / roaster / terroir / cultivar registries. ## Flavor notes registry (`lib/flavor-registry.ts`)
- **3-axis composable taxonomy** post sprint 1g (2026-04-27): `BASE_FLAVORS` (182 canonicals across 12 categories — Rosemary added Sprint 2.1) + `FLAVOR_MODIFIERS` (43 across 10 categories with `priority` ordering Transformation > Form > State > Processing > Intensity/Sweetness/Texture/Blending/Other > Structure) + `STRUCTURE_TAGS` (29 across 7 axes: Acidity / Body / Clarity / Finish / Sweetness / Balance / Overall). Authoritative authored content in [docs/taxonomies/flavors.md](docs/taxonomies/flavors.md) with 17 numbered rules.
- **Storage shape:** `brews.flavors jsonb` (array of `{base, modifiers[]}`) for structured form + `brews.structure_tags text[]` for per-coffee descriptors as `"Axis:Descriptor"` keys. Legacy `brews.flavor_notes text[]` retained as denormalized display through 1g.2 follow-up — write paths populate all three.
- **Tea-base rule (Rule 11):** when base is in `TEA_BASES` (Tea / Black Tea / Green Tea / Earl Grey / etc.), the modifier slot accepts BOTH canonical modifiers AND base flavor names — `Peach Tea` decomposes to `Tea + Peach (modifier)`, `Jasmine Milk Tea` to `Tea + Milky + Jasmine`. The composer's modifier dropdown surfaces 212 options for Tea bases and 43 for all others.
- **Fallback anchors (Rule 4):** `FALLBACK_ANCHORS` map suggests a default specific base when an abstract category-name is typed: `Floral → Jasmine`, `Citrus → Lemon`, `Stone Fruit → Peach`, `Tropical → Mango`, `Herbal → Mint`, `Berry → Strawberry`. Override immediately if a real note exists.
- **Composition helpers:** `composeFlavorNote(chip)` renders `"Blueberry (Baked)"`. `decomposeFlavorNote(raw)` resolves a raw input via `ALIAS_MAP` (112 entries) or canonical base; returns null otherwise. `composeFlavorNotes(chips[])` for array form.
- **Validation helpers:** `cleanFlavors(input)` + `cleanStructureTags(input)` return `CleanResult<T> = {ok: true, value} | {ok: false, error}`. Used by both `app/api/brews/[id]/route.ts` PATCH + `lib/brew-import.ts` persistBrew (rule-of-2 extraction in /simplify pass). Single source of truth for the per-chip canonicalize + 2-modifier clamp + Tea-base rule.
- **Module-level O(1) lookups (hot-path):** `ALIAS_LOWER_MAP: Map`, `BASE_LOWER_SET: Set`, `BASE_ENTRY_BY_NAME: Map`, `MODIFIER_ENTRY_BY_NAME: Map` — built once at import. Composer keystroke + chip render hit these instead of `Array.find` (was O(n) on 181 bases per chip render).
- **Family render (back-compat):** 8-family palette (Citrus / Stone Fruit / Berry / Tropical / Grape & Wine / Floral / Tea & Herbal / Sweet & Confection + Other) preserved verbatim. `getFlavorFamily(note)` strips parenthetical modifier ("Blueberry (Baked)" → "Blueberry") then routes via `getCategoryFamily(category, subCategory)` — Fruit splits across 4 families by sub-category (Citrus / Berry / Stone Fruit / Tropical / Grape & Wine), other 11 categories collapse to their direct family or Other (Defect / Vegetative / Roasted / Nut / Spice / Savory / Acid / Beverage all go to Other).
- **`<FlavorComposer>` + `<StructureTagsPicker>`** components consume the registry. Replaces deprecated `FlavorNotesInput`. Soft warnings (not save-gating) per Rule 11 (2-4 flavors) + Rule 8 (2-3 structure tags).
- **Aggregation:** `aggregateFlavorNotes(brews)` → `[note, count][]` sorted desc. Reads from legacy `flavor_notes` (composed display) for back-compat through 1g.2; new aggregator reading from `flavors` jsonb is deferred to 1g.2.
- **Adding entries:** alias additions are encouraged (edit `ALIAS_MAP`); new bases are deliberate edits to `BASE_FLAVORS` + the markdown sub-section (per Rule 1 of alias maintenance: "Only add aliases, not new canonicals" — promote only when repeated across many coffees AND not expressible via modifiers). Net-new bases beyond Chris's authored 176: `Pomegranate`, `Baklava`, `Haw Flake` (in legacy registry) + `Tea`, `Sugar` (generic catch-alls surfaced during Bucket C decomposition).
