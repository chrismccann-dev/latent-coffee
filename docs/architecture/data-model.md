# Data model reference (per-column histories + relationship detail)

Per-column schema histories (migration provenance, populate-side notes) for the core entities, plus the full relationship-pattern detail. **Claude-Code-facing reference — read on demand when touching a column / migration; not MCP-registered.** The tight entity roster + key FK rules live in CLAUDE.md § Data Model. Canonical-registry implementation rules live in [registries.md](docs/architecture/registries.md). Extracted from CLAUDE.md 2026-06-02 (root-doc compaction) — move-never-delete.

## Core entities
- **brews** — individual coffee tastings (purchased or self-roasted). 79 records as of 2026-05-09. Has distinct `roaster` and `producer` columns; don't conflate them. Self-roasted brews carry `roaster = 'Latent'`.
- **terroirs** — geographic/ecological zones. Hierarchy: Country → Admin Region → Macro Terroir. 121 canonical macros across 38 countries post-Region-sprint (2026-04-22, sprint 1d.1). See [docs/taxonomies/regions.md](docs/taxonomies/regions.md) for authored content and [lib/terroir-registry.ts](lib/terroir-registry.ts) for the validation mirror. Meso and Micro are free-text annotations on terroir rows — retained in the schema, not in the canonical registry, not validated, not autocompleted (every behavioral attribute is macro-scoped; producer-level locality is the right home for farm detail).
- **cultivars** — coffee varieties. Hierarchy: Species → Genetic Family → Lineage → Cultivar. 63 canonical cultivars post-Variety-sprint (2026-04-22); see [docs/taxonomies/varieties.md](docs/taxonomies/varieties.md) for authored content and [lib/cultivar-registry.ts](lib/cultivar-registry.ts) for the validation mirror.
- **green_beans** — raw coffee lots (self-roasted only, 13 records as of 2026-05-13). Has roasts, experiments, cuppings, roast_recipes. Lifecycle state is derived per row, not stored — see § Green for the 4-state rules.
- **roast_recipes** (Sub Pages 6.1, migration 052, 2026-05-13) — first-class entity for per-batch design intent. One row per Roest profile pushed (each batch execution = one recipe, even when curves are identical — matches Roest tablet semantics). ~32 fields covering curves (`temperature_bezier` / `fan_bezier` / `rpm_bezier` / `power_bezier`) + design specs (`charge_temp` / `hopper_load_temp`) + design-time predictions (`predicted_fc_temp` / `predicted_cup` etc.) + drop rules (`drop_rule_if_fast` / `drop_rule_if_slow`) + per-batch `rationale` (Hypothesis prose distinct from `notes`) + Roest linkage (`roest_profile_id` / `roest_share_url`) + provenance (`was_backfilled boolean` + `backfill_notes text`, Schema sprint S4 / migration 057 — flag distinguishes design-time pushes from STAGE-0-inline-backfill / log-roast.md STAGE 1(b) backfills / migration-052 legacy shells; 112 surviving migration-052 rows marked `was_backfilled=true` with a canonical note in 057). UPSERT on (user_id, experiment_id, batch_slot) when both supplied (V-set framing — v1a/v1b/v1c are three recipes), otherwise (user_id, green_bean_id, recipe_name). Backfilled 129 rows from existing roasts (1:1) in migration 052. Roast execution links via `roasts.recipe_id` FK. Recipe lineage via nullable `parent_recipe_id` self-FK ("v3a replicates v2b"). See [docs/roasting/redesign.md](docs/roasting/redesign.md) § 4.3 for full field rationale.
- **roasts.is_reference_candidate** (Schema sprint S2, migration 056, 2026-05-18) — forward-looking quality flag set during V-set iteration via `patch_roast(is_reference_candidate: true|false)` in `log-cupping.md` STAGE 3 on the leading slot. **Decoupled from `is_reference` AND `worth_repeating`**: candidate is mid-flight ("the V_n leading slot looks reference-quality"); `is_reference` is final ("yes, this is the lot reference"); `worth_repeating` is a recipe-replay judgment. Multiple V-sets can each have a candidate; only one batch ultimately gets `is_reference=true` at close-out. Candidate does NOT auto-flip — `close-lot.md` / `one-shot-closeout.md` STAGE 2 makes the promotion explicit. See [CONTEXT-roasting.md § Reference candidate](CONTEXT-roasting.md) for terminology. Net-new forward; no historical backfill.
- **roasts.ror_at_2_30 / ror_at_4_00 / ror_at_fc_minus_30s + inlet_curve_recorded** (Sprint 3.5, migration 070, 2026-05-26) — 4 net-new columns populated server-side by `pull_roest_log` from the Roest `/datapoints/?log={id}` endpoint discovered during the Sprint 3.5 audit (raw bt + inlet_temp time-series; data_type 0=event / 1=sensor + event_type 0-8 incl. auto-detected FC/Dryend/Drop — see [docs/skills/roest-knowledge/cluster/api/read-surface.md](docs/skills/roest-knowledge/cluster/api/read-surface.md)). RoR values are °C/min via a 30-second centered window on the bt curve (2:30 = drying-handoff check, 4:00 = approach-to-FC, FC-30s = cross-lot comparable post-hoc anchor). `inlet_curve_recorded` is the as-recorded inlet temp sampled at the same msec keys as the as-designed `inlet_curve` (RoestProfile.temperature_bezier) so the two display strings line up for visual diffing — they diverge when the operator overrode the designed inlet mid-roast. Same migration also populates the existing-but-NULL-only `roasts.tp_time` + `roasts.tp_temp` + `roasts.yellowing_temp` columns (added migration 039) via server-side compute from the same bt curve. **R57 rewire same migration**: legacy `roasts.roest_notes` content coalesces into `roasts.color_description` where the latter was NULL (Chris uses Roest UI Notes for post-CM200 color descriptors; `pull_roest_log` payload field renamed `roest_notes → color_description`; `push_roast.roest_notes` deprecated for back-compat). The `roasts.roest_notes` column remains in place for back-compat; future cleanup sprint may drop it. Net-new forward; existing rows get RoR + inlet_curve_recorded populated on next re-pull via `pull_roest_log` if the original log still exists on Roest. **Lessons from sprint**: Roest API documentation gap — `/datapoints/` had been there the whole time, never surfaced because no prior audit pulled the OpenAPI schema; standing tripwire: pull the schema cheap-and-early on any API-coverage audit.
- **roasts.fc_audibility** (Sprint 11, migration 061, 2026-05-20) — 4-value enum (`audible` / `subtle` / `silent` / `ambiguous`) on each batch's first-crack audibility state. Three of four values (subtle / silent / ambiguous) trigger the same downstream protocol (bean-temp end condition + drop-ceiling-primary + Agtron as proxies); the distinction matters for cause attribution and for predicting audibility on future similar lots. `push_roast` + `patch_roast` Zod schemas expose the field; ResolvedView surfaces it inside `<PerRoastReflections>` (per-roast drill-down rather than per-row column on RoastLogTable). Historical 135 roasts left NULL — populate going forward per RO-CP-3 (4.4% had `fc_total_cracks` populated, too sparse to backfill reliably from). See [CONTEXT-roasting.md § FC audibility state](CONTEXT-roasting.md) for the full 4-value semantics + protocol-stack notes.
- **cuppings.wb_agtron + wb_to_ground_delta** (Schema sprint S1, migration 055, 2026-05-18) — snapshots `roasts.agtron` onto cuppings at `push_cupping` time + generates the WB→Ground delta from the same row. `persistCupping` joins `roasts.agtron` by `roast_id` and writes `wb_agtron`; `wb_to_ground_delta` is a Postgres generated column (`wb_agtron - ground_agtron` STORED). Solves the Round 3 #6 cross-lot delta query gap; the resolved-view's inline computation at [page.tsx:1297-1308](app/%28app%29/green/%5Bid%5D/page.tsx) remains for back-compat (column-read refactor is a deferred follow-up). Cross-lot delta range on existing data: -4.3 to +11.9 (sign-inverted cases like Fazenda Um v1b -0.8 are now queryable without SQL gymnastics). 99/99 cuppings backfilled wb_agtron at migration time; the generated column populates wherever ground_agtron is also present (59 rows). Drift on post-cupping Agtron re-measurement is realigned via `patch_cupping(wb_agtron=...)`.
- **cuppings.sweetness + cuppings.temperature_behavior** (migration 046 / 2026-05-07; reachable via MCP from Schema sprint S3 / 2026-05-18) — two prose axes that landed in the schema in May 2026 but were unreachable via MCP for ~10 days. `push_cupping` + `patch_cupping` Zod schemas now expose both; `persistCupping` passes them through; `/green/[id]` ResolvedView's REFERENCE CUP card + All Cuppings collapsible render both. Sweetness is a distinct axis from acidity / body (don't fold in); temperature_behavior parallels brews.temperature_evolution.
- **cuppings.aromatic_behavior + cuppings.structural_behavior** (Sprint 11, migration 062, 2026-05-20) — two prose axes relocated from `roast_learnings` per ADR-0008. They describe what a cup IS (per-tasting observation), not what a lot TAUGHT (lot-aggregate carry-forward, which stays on `roast_learnings.cultivar_takeaway` / `general_takeaway` / etc.). `push_cupping` + `patch_cupping` Zod schemas expose both; `persistCupping` passes them through; `/green/[id]` ResolvedView's REFERENCE CUP pourover sub-card renders them in the descriptor cascade below `overall` (or the labeled-rows fallback). 7 closed lots backfilled in the migration to the canonical pourover cupping on each lot's `best_roast_id` (priority `eval_method ILIKE '%pourover%'` DESC, then `cupping_date` DESC, then `created_at` DESC, LIMIT 1). The corresponding columns dropped from `roast_learnings` in the same migration; `push_roast_learnings` + `patch_roast_learnings` Zod schemas + Tool descriptions document the move.
- **taxonomy_overrides_queue** (Phase 3, migration 045) — canonical-promotion queue. Mirror of `doc_proposals` (037) for canonical entries. Populated at push time when an `*_override:true` flag bypasses canonical validation, or via the `propose_canonical_addition` Tool. Chris-as-arbiter walks it via `list_taxonomy_queue` + `resolve_queue_entry` and edits `lib/{axis}-registry.ts` + `docs/taxonomies/{axis}.md` per row. See [docs/features/taxonomy-overrides-queue.md](docs/features/taxonomy-overrides-queue.md) for design + [ARBITER.md](ARBITER.md) § Taxonomy queue arbitration for the playbook.
- **green_beans + brews provenance columns** (Phase 3, migration 045) — `terroir_provenance` + `cultivar_provenance` enum (`canonical` | `auto_created`) flags whether a push materialized the FK row for the first time vs. resolved against an existing row. `green_beans.canonicals_updated_at` timestamp bumps when terroir_id / cultivar_id changes via patch_green_bean or future canonical re-resolution.

## Relationship patterns (IMPORTANT)
- **Brew → Terroir:** FK via `brews.terroir_id`. All 55 brews linked (backfilled in migrations 005-006).
- **Brew → Cultivar:** FK via `brews.cultivar_id`. All 55 brews linked (backfilled in migration 006).
- **Brew → Green Bean:** `brews.green_bean_id` — set for all 4 self-roasted brews (backfilled in migration 006).
- **Green Bean → Terroir/Cultivar:** `green_beans.terroir_id` and `green_beans.cultivar_id` — all 13 populated.
- **Roast → Roast Recipe:** `roasts.recipe_id` FK to roast_recipes (Sub Pages 6.1, migration 052). All 129 backfilled rows 1:1 linked. Nullable through Phase 3; future roasts populate via push_roast.recipe_id alongside push_roast_recipe.
- **Roast Recipe → Experiment:** `roast_recipes.experiment_id` nullable FK. Set when the recipe is part of a V-set; null for one-off / calibration recipes.
- **Roast Learnings → Roast:** `roast_learnings.best_roast_id` typed UUID FK to the winning execution (Sub Pages 6.1, migration 052). All 6 closed lots backfilled cleanly from the legacy text `best_batch_id`. Legacy text column stays through Phase 3 for back-compat.
- **New brews must set `terroir_id` and `cultivar_id`** on insert. The `push_brew` MCP Tool handles this for both purchased + self-roasted paths via `findOrCreateTerroir` + `findOrCreateCultivar` in [lib/brew-import.ts](lib/brew-import.ts) — same helpers the deleted form code consumed.
