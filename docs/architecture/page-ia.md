## Cultivars (`app/(app)/cultivars/`)
- **Index:** Groups by Species (header) → Genetic Family (header w/ swatch + visible-lineage count) → Lineage (sub-header, non-link) → Cultivar (linked row w/ brew count). 0-brew cultivars hidden (Sub Pages 3, 2026-05-11) — matches event-driven workflow rule. Lineage cascades hide when all its leaves hide; family cascades hide when all its lineages hide. Brew counts via `cultivar_id` FK join. Lineage labels render even for 1-cultivar lineages (e.g. Gesha lineage > Gesha) — structurally correct per Chris's taxonomy. **Re-skinned to the v2 grouped-row-list (Redesign Sprint 6 PR2, 2026-05-29) — kept the Species→Family→Lineage→Cultivar nesting, NOT the v2 genealogical text-tree, per Chris's call:** `<IndexCap>` + `<GrlCap label="CULTIVARS">` + Species as a mono `text-xs` muted spine label + Family as `<GrlGroupHeader>` (family-color swatch) + Lineage as the `.grl-lineage` indented mono sub-label (aligned under the row name column via `calc(44px + 16px)`) + Cultivar leaves as `<GrlRow>` (family-color tile + name + brew count + 5-block bar). The pre-Sprint-6 heading-weight hierarchy (Sub-sprint 4d's `text-lg` species header) is superseded by the grl register.
- **Detail:** **Re-skinned to `Ssp*` lab-document chrome in Redesign Sprint 5 PR2 (2026-05-29) — chrome-only, IA + 9-section order preserved.** `<SspTopBar>`+`<SspNamePlate>` (family-color cover, Species/Family/Lineage meta) → Genetic Background `.ssp-prose` → the three `<GridField>` grids (Cultivar Context / Brewing & Cup Profile / Roasting Characteristics) now render `SspProseRows` via a local `field()` mapper + `compactRows` (array fields join with `, `) → `<SynthesisCard>` (≥2 brews) → shared `<CoffeesList showProcessBadge>` → `.ssp-coll` Additional Information → shared `<ConfidenceCard>`. Original Sub Pages 3 IA: Single-cultivar destination post-2026-05-11 (Sub Pages 3 inverted the data model — lineage is no longer a destination, just a breadcrumb segment). Re-ordered around the user job ("about to brew/roast this specific cultivar"). Three-act framing: General Cultivar Info → What I've Learned → My Experiences. 9 sections top-to-bottom: (1) **Hero** — color swatch (family color) + `cultivar.cultivar_name` h1 + `species → genetic_family → lineage` breadcrumb subtitle. Drops the sibling Tags row (redundant on single-cultivar page). (2) **Genetic Background** — `<SectionCard>` prose. (3) **Cultivar Context** — 2-col grid (1-col mobile) of 4 fields: Typical Origins / Altitude Sensitivity / Limiting Factors / Market Context. (4) **Brewing & Cup Profile** — 2-col grid of 8 fields: Common Processing Methods / Typical Flavor Notes / Acidity Style / Body Style / Aromatics / Terroir Transparency / Extraction Sensitivity / Brewing Tendencies. (5) **Roasting Characteristics** — 2-col grid of 4 fields: Roast Tolerance / Roast Behavior / Resting Behavior / Common Pitfalls. (6) **What I've Learned About This Cultivar** — synthesis card, **conditional on `brewList.length >= 2`** (1-brew cultivars hide the card; LOW confidence carries the message). (7) **Coffees I Have Brewed In This Cultivar** — brews list filtered by `cultivar_id` (single, not `.in(siblings)`). (8) **Additional Information** — `<CollapsibleBlock>` containing 4 bare-prop sub-sections: Flavor Notes I Have Experienced + Terroirs I Have Explored + Processes I Have Explored + Roasters Who Roasted This Cultivar. (9) **Confidence** — dark card. Local `<GridField label value>` helper renders each grid cell (label above value, scalar or array-joined string). **Intentionally-internal columns (audited Sub-sprint 4d, 2026-05-28):** `cultivar_notes` (28/32 populated) + `cultivar_confidence` (26/32, `high`/`medium`) have no render path *by design*. `cultivar_notes` is normalization/authoring guidance ("treat as a field blend; do not separate into component cultivars"), not user-facing reference; `cultivar_confidence` is the registry's genetic-classification confidence, semantically distinct from the page's brew-count Confidence card (rendering both would read as two competing confidence signals). Do not surface either without a deliberate decision.
- **Synthesis:** `/api/cultivars/synthesize` — accepts `cultivarIds[]` (caller now sends `[singleId]`). Finds brews via `cultivar_id` FK. `entityName` = `primary.cultivar_name` (no longer lineage-fallback). Cache keyed on `cultivars.synthesis` per-row; existing lineage-aggregated content invalidated to NULL at Sub Pages 3 ship time. Re-generation triggers per-cultivar on first ≥2-brew visit via SynthesisCard's brewCount-changed trigger.
- **Adapter:** [lib/synthesis/adapters/cultivar.ts](lib/synthesis/adapters/cultivar.ts) — `CultivarAnchorContext` is now `{ primary: Cultivar }` only (dropped `cultivarNamesInLineage` field at Sub Pages 3 — dead after the lineage→cultivar inversion). All 28 cultivar fields render verbatim into the anchor when populated; the 6-tier weighting + outputFormat + extraRules + formatLearningRow are unchanged.

Original Sub Pages 4 IA below.

Sub Pages 4 (2026-05-11) replaced the flat text-keyed `/processes/{slug}` aggregation with a 3-tier composable architecture grounded in the existing `base_process` + modifier-axis registry. Subsumed the deferred 1e.4 facet redesign in the same sprint.
- **Index:** 3 navigation surfaces. **Re-skinned to the v2 register (Redesign Sprint 6 PR2, 2026-05-29):** `<IndexCap>` header; each surface gets a `.label` section header; Core Process Portals stay richer portal cards (kept — the pure/combo chips are navigationally load-bearing); Modifier Index + Signature Methods render `<GrlRow>` rows (axis/signature-color tile + name + meta + count + 5-block bar — absolute thresholds via `barBlocks`, NOT normalized within the list).
  1. **Core Process Portals** — 4 base hubs (Washed / Natural / Honey / Wet-hulled) sorted by brew count desc. 0-brew bases hidden per event-driven rule.
  2. **Modifier Index** — cross-base modifiers with ≥3 brews. 8 entries today (Anaerobic 16, Yeast Inoculated 9, Dark Room Dried 6, Cold Fermentation 4, Raised Bed 4, Slow Dry 4, Double Anaerobic 3, Thermal Shock 3). Driven by `eligibleModifierIndexEntries(brews)` — list grows as new modifiers cross the threshold. Row meta = `byBaseLine(entry.byBase)`.
  3. **Signature Methods** — 15 canonicals post Sprint T1 / BR-1 2026-05-18 (Moonshadow / TyOxidator / Alchemy / TIM / XO / Enzyflow / Bio-innovation / Sous-vide / Amazake / Anti-maceration / Dynamic cherry / Dry fermentation / Splash / Symbiotic / Wave Hybrid). - **Sub-page eligibility** — Chris's brainstorm Rules 1-3:
  - Rule 1 (≥3 brews): per-base modifier-combo earns `/processes/{base}/modifiers/{combo-slug}`
  - Rule 2 (signature exemption): always earns `/processes/signatures/{name}` even at 1 brew
  - Rule 3 (Honey color tier structural): White Honey + Generic Honey always earn `/processes/honey/{subprocess}`
  - Below threshold = chip-only on parent hub, no destination
- **URL surface (~21 pages from current DB):**
  - `/processes` (index)
  - `/processes/{base}` × 3 populated (Washed / Natural / Honey)
  - `/processes/honey/{subprocess}` × 2 (white / generic)
  - `/processes/{base}/modifiers/{combo}` × 3 (washed/anaerobic, natural/anaerobic, washed/double-anaerobic-thermal-shock-yeast-inoculated)
