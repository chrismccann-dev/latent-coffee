## Context

These are the small cupping/roast schema guardrails that keep recurring in roasting
workflow feedback — each one is a place where a real signal currently lives only in prose
(so it can't be queried across lots) or where a transcript slip can write nonsense with no
guard. They're additive, mechanical, and independent of the bigger lifecycle-state rework
(which is deliberately OUT of scope — see below).

## Goal

Make the recurring cupping/roast signals first-class + add the cheap validation guards, so
cross-lot queries stop being regex-on-prose and bad transcript data gets caught at write
time.

## Scope — IN

1. **`cooling_arc_pattern` enum on `cuppings`** (backlog: "cooling_arc_pattern enum", master log #27, med)
   - Values: `degrade / hold / improve / flat`. Independent of the `temperature_behavior`
     prose. It's now differentiating on a 2nd lot, so cross-lot "which lots cooling-arc
     degrade vs hold" should be canonical-queryable.
2. **Server-side `cupping_date ≥ roast_date + 1` + rest_days consistency check** (backlog: "cupping_date ≥ roast_date+1 validation", master log #31, med)
   - Voice-to-text date slips are a real transcript-drift class (a bare "March 31" with no
     day-of-week would write nonsense rest_days like −23). Validate that cupping_date follows
     roast_date AND that computed rest_days matches the supplied value (modulo timezone).
3. **Derived `roast.execution_diverged_from_recipe`** (backlog: "recipe↔roast intentional-divergence signal", master log #26, low)
   - `recipe.end_condition_target` set vs `roast.end_condition_type='manual'` (target NULL)
     is an intentional, lore-dependent divergence. A derived/computed boolean lets analytics
     read it correctly without knowing the lore.

## Scope — OUT
