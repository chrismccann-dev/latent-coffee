## Header

- **Doc pruned:** `docs/skills/wbc-brewing-archivist/cluster/wbc-recipes.md` — 73.8 KB → 35.7 KB; per-recipe rows extracted to new sibling `docs/skills/wbc-brewing-archivist/cluster/wbc-recipes-by-family.md` (40.1 KB). Cluster total 121.4 / 200 KB (unchanged net; content moved, not removed).
- **Trigger:** `npm run check:doc-sizes` flagged the doc 🔴 at 73.8 / 60 KB (single-doc cap), [doc-tripwires.md](docs/architecture/doc-tripwires.md) largest-doc row. Crossed the cap when the 2026 WBC drop (PR #522) added 52 routines + the Water Chemistry Systems family.
- **Shape(s) used:** `split` (the only shape) — along the doc's own stated taxonomy↔per-recipe seam. Not `extract`-to-pointer (the per-recipe tables aren't a self-contained appendix you point at once; they're a parallel reference surface with its own lookup life), not `consolidate`/`delete` (every row is live + unique competition data).
- **Judgment calls:** (1) **Split axis = taxonomy vs per-recipe, NOT by year.** The year-split option was surfaced and rejected: the recipe tables are grouped *by strategy family* with year as a row attribute, so a year split would scatter each family across two files and fight the dominant lookup ("show me all Hybrid recipes"). The taxonomy/per-recipe seam is the doc's own self-description (intro line 5) and lands both halves ~36/~40 KB. (2) **Raise-the-cap was a live alternative** (cluster had 81.5 KB headroom) but only defers — at +12-15 KB/WBC-year the combined doc re-crosses 100 KB in ~2 drops. Chris chose the split. (3) **Which half keeps the canonical name.** Kept `wbc-recipes.md` as the slow-growing taxonomy parent (most stable referent) and gave the fast-growing rows the new descriptive name `wbc-recipes-by-family.md` — so the growth pressure lives in the file with the more disposable identity.
- **Heuristic learned:** **For an inherently-growing reference corpus, split off the *fast-growing* sub-surface into its own file and keep the slow-growing taxonomy/index as the canonically-named parent that routes to it.** This puts the next cap-trip on the half that's actually accumulating, and the stable taxonomy half never re-trips. The split axis should be the doc's own organizing seam (here: definitions vs rows), never a cross-cutting attribute (year) that fractures the dominant lookup grouping.

## Shape-coverage note

Stayed `split` — and that was genuinely correct, not reflex-extract. The cut-test classes don't apply (no no-op / sediment / duplication / relevance failures — it's a deliberately exhaustive archive that *should* keep every row). The only honest lever for an all-live, monotonically-growing corpus is structural division. `split` precedent already exists (CONTEXT zone split, filters.md ownership split), but this is the **first split of an Archivist-cluster reference corpus** and the first where the split axis was chosen explicitly to isolate the growth vector rather than an ownership or zone boundary. No new shape; a new *split-axis-selection* heuristic.
