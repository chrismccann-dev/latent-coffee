## Header

- **Doc pruned:** `docs/skills/brewing-equipment-expert/cluster/filters.md` — **70.4 KB → 22.4 KB**. The brewing-equipment-expert cluster dropped **157.8 KB → 109.9 KB** as a result, clearing its 150 KB cap — so the companion **case 005 (cluster over-cap) is MOOT**, no session needed. New sibling: `docs/taxonomies/filters-not-owned-archive.md` (~21 KB, outside the measured cluster dir).
- **Trigger:** single-doc 60 KB cap on the EquipmentReference cluster ([doc-tripwires.md](../../architecture/doc-tripwires.md)) + the cluster's own 150 KB cap, both surfaced by the first `npm run check:doc-sizes` run (2026-06-03). filters.md was the lever for both.
- **Shape(s) used:** **`split` (primary)** — owned (23) stay in `filters.md`; not-owned (44) move to a `docs/taxonomies/` archive. Plus **`extract`** (the four `## Measured Drawdown Reference` research blocks + per-entry RP/lesson citation chains → a 1-paragraph appendix + pointers; full data already in `docs/research-projects/*.md`) and **`consolidate`** (owned entries trimmed to spec + one measured-drawdown line). No content deleted from the system; `lib/filter-registry.ts` kept intact as the full 67-entry validator.
- **Judgment calls:** (1) **Owned-only split, not in-doc compaction.** Chris's audio + two prototype files (`filters_refactored.md` + `filters_not_owned_archive.md`) steered "only carry what I own into the live brewing surface; not-owned is a promotion pool." I went further than my pre-read proposal (which kept non-owned as compact rows) because the agent-clarity win (no owned/not-owned ambiguity at selection time) is real. (2) **Registry stays the full validator (deviation from prototype).** The prototype implied registry merges (pack-size, cup-1/cup-4, duplicate-SKU); those change canonical identity and could orphan `brews.filter` rows, and this session had no raw-SQL surface to cross-check. Per AskUserQuestion → "flag as follow-up." So the .ts keeps all 67 rows; the archive is the prose home for the 44 not-owned, preserving doc↔registry consistency across two files. (3) **Kept registry-true canonical names** (e.g. `UFO FAST`, not the prototype's `Sibarist UFO FAST`) — the doc heading is what canonicalizes brews, so it must equal `lib/filter-registry.ts` `name`. (4) **Fit grouping** = V60 cone / Flat-bottom + wave / Specialty + paired-brewer (brewer-fit is the first selection gate). (5) **April Paper Filter fit corrected to April-only** (Chris: "that's actually wrong") — in both doc and registry; the registry's own paperShape audit note already said it fits only April, so `fitsBrewers` was internally inconsistent.
- **Heuristic learned:** **For a registry-catalog doc with an owned/not-owned distinction, the highest-leverage prune is `split` on the ownership axis, NOT consolidate.** A live equipment doc only needs what the operator owns; the long tail of catalog completeness is a promotion-pool reference that belongs outside the loaded surface. Critically — **the validation mirror (`lib/*-registry.ts`) stays whole; the split happens in the docs.** Two doc files + one intact validator preserves the canonical-consistency invariant with zero brew-orphan risk, and defers the heavier identity-merge work to a dedicated reconciliation pass. **Corollary for measured/researched registries:** the *research narrative* extracts cleanly to the research docs while the *measured result* stays as a single field-line in the entry — provenance is a pointer, not inline weight.
