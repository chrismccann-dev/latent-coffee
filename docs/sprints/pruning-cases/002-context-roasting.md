- **Heuristic learned:** **On a living glossary, the dominant bloat is not stale *terms* — it is resolved decisions narrated through their entire deliberation history** (grill-session provenance + "originally framed... the audit resolved..." narrative + superseded intermediate reasoning). `consolidate` = keep the resolved state, delete the journey to it. The cross-reference web makes `extract` mostly wrong (you can't pointer-ize a term without snapping its relationships) **except** where an entry has drifted into operational detail that already has an operational home (FC → fc-marking) — there, extract is exactly right. And: **always run a concept-set diff after a structural reorg** — restructuring silently drops concepts, and the dangling-prompt-reference is the bug that proves it.

## Shape-coverage note

This case **deliberately hunted the missing shapes and did not reflex-extract.** Coverage delta vs the brainstorm's checklist:

- **consolidate** — ✅ first worked example. Operator cluster-regroup + Claude Code definition-tightening + duplicate-paragraph removal (the doubled Multi-factor weighting entry). Synthesis, not move.
- **delete** — ✅ first worked example, **two distinct sub-categories**, both operator-authorized (not autonomous): (a) **provenance shrapnel** (grill-session breadcrumbs + deliberation-history narrative); (b) **speculative/deferred-future TODO notes** (deferred-badge "future UI surface", "earns strict validation later", "a future pass may add a state"). Both are low-risk delete classes: git-recoverable, zero downstream consumers, no concept value. Distinguishing them matters for systematization — both are candidates for the *first autonomous-delete rule*, separate from concept-bearing deletes which must stay flag-only.
- **extract** — used once, correctly (FC operational detail → its existing operational home). Not the primary shape.
- **split** — operator added navigation scaffolding (Purpose / Reading order / High-risk pairs / Agent term index), which is split-adjacent (one flat glossary → a navigable lookup graph).

Open question this case raises for systematization: **how autonomous can `delete` of provenance-class content become?** It's the lowest-risk delete class (audit breadcrumbs, git-recoverable, zero downstream consumers) — a candidate for the first autonomous-delete rule, distinct from concept-bearing deletes which must stay flag-only.
