## Context

When a lot closes, `close-lot.md` STAGE 5 (V-set) and `one-shot-closeout.md` STAGE 5
(one-shot) route the close-out narrative to **net-new per-lot files**:
- `docs/skills/roasting-historian/cluster/learnings/<lot-slug>.md`
- `docs/skills/roasting-historian/cluster/one-shot-calibrations/<lot-slug>.md`

But `propose_doc_changes` validates every `skills/` target against the registered
`SKILL_FILES` allow-list in [lib/mcp/docs.ts](../../lib/mcp/docs.ts) (via `isKnownDoc`),
so a brand-new per-lot file **rejects** with `"Unknown skills target: '...'. Register
the path in lib/mcp/docs.ts SKILL_FILES first, then re-attempt."` Every closed lot
therefore orphans its canonical-home content. This is the **root cause of the "doc
proposal couldn't land" friction** — recurrence 3 (UGA-MH-ELGON, RWA-NOVA Bukure,
COS-HIG Higuito), criticality high, and it stops a surface (the per-lot learnings /
calibrations doc-write path).

Today both prompts document a **manual three-step recovery** (draft content → emit a
deferred artifact + note to Chris → Chris hand-registers the path in a Claude Code
session). Option (b) formalizes that recovery into a standardized **arbiter ticket** the
existing operator-run arbiter consumes — keeping the human-in-the-loop gate (this is why
Chris chose (b) over (a) auto-register).

## Goal

Make a closed lot's per-lot learnings / calibrations file land through a standardized,
documented path: the close-out prompt emits a **per-lot-file-registration ticket**; an
ARBITER.md playbook section registers the path (`SKILL_FILES` + `DOC_DESCRIPTIONS` +
glob check) and seeds the file with the ticketed content. No more orphaned close-out
narrative; no ad-hoc "note to Chris."

## Scope — IN

1. **Define the ticket format** — a machine-scannable fenced block the close-out prompt
   emits at STAGE 5 (rendered to Chris at STAGE 7) when it needs a net-new per-lot file.
   Required fields:
   - `ticket_type: per-lot-file-registration`
   - `lot_slug: <slug>`
   - `target_path:` one of the two whitelisted per-lot prefixes (see point 3)
   - `seed_content:` the **full** drafted markdown body for the file (header +
     Substrate pointers + Cross-lot framing + Related, per the existing STAGE 5 spec)
   - `source:` the close-out session id (`{kind: "session", id: "<lot_id close-out>"}`)
   - One ticket per net-new file (a lot may emit up to 2: learnings + calibrations).
2. **Patch both close-out prompts' STAGE 5** —
   [docs/prompts/close-lot.md:163](../prompts/close-lot.md) and
   [docs/prompts/one-shot-closeout.md:126](../prompts/one-shot-closeout.md): replace the
   current three-step "deferred artifact + note to Chris" recovery prose with the
   formalized ticket emission. Keep the existing auto-split-per-`(target_doc,
   section_anchor)` rule for the *registered* citations (active-lot empty-replace,
   archive, etc.) — the ticket flow is **only** for the net-new per-lot files; everything
   else still goes through `propose_doc_changes` as today.
3. **Constrain the whitelisted prefixes** to exactly:
   - `skills/roasting-historian/cluster/learnings/<lot-slug>.md`
   - `skills/roasting-historian/cluster/one-shot-calibrations/<lot-slug>.md`
   Any other net-new `skills/` path stays hard-rejected (forces the normal arbiter path
   for non-per-lot files). No brewing-side per-lot files exist today — do not add them.
4. **Add the ARBITER.md consumer section** — a new `## Per-lot file registration tickets`
   section in [ARBITER.md](../../ARBITER.md) (match the existing numbered-substep shape of
   the "Skeleton review" / "Taxonomy queue arbitration" sections). Steps:
   - **P1.** Read the pending ticket(s) from the close-out artifact Chris pastes.
   - **P2.** Validate `target_path` is under a whitelisted per-lot prefix + `lot_slug`
     matches the path; reject otherwise.
   - **P3.** Add the `SKILL_FILES` entry (`docs://<path>: <path>`) **and** the matching
     `DOC_DESCRIPTIONS` entry in [lib/mcp/docs.ts](../../lib/mcp/docs.ts).
   - **P4.** Verify [next.config.js](../../next.config.js) `outputFileTracingIncludes`
     glob (`./docs/skills/**/*.md`) covers the path, then run `npm run check:mcp-bundle`
     (must exit 0).
   - **P5.** Seed the file with the ticket's `seed_content`.
   - **P6.** Commit + PR + merge (reuse the section's standard commit step).
5. **(decide-and-proceed) Improve the `propose_doc_changes` rejection message** in
   [lib/mcp/propose-doc-changes.ts:245](../../lib/mcp/propose-doc-changes.ts): when the
   rejected `skills/` target is a well-formed path under a whitelisted per-lot prefix,
   point the operator at the ticket flow instead of a bare "Register first" — e.g.
   `"'<path>' is a net-new per-lot file under a registration-whitelisted prefix. Do not
   retry propose_doc_changes; emit a per-lot-file-registration arbiter ticket (close-lot /
   one-shot-closeout STAGE 5) so the path is registered + seeded."` This is the
   catch-time→hint-time move *within* #33's own surface (stops the empty-retry loop).
   **Default: do this** (low-risk, same Tool, no schema change). Escalate only if it
   requires a whitelist-prefix constant that complicates the file.
