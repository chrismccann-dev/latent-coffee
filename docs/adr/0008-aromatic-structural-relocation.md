# 0008 — Aromatic and structural behavior relocate from roast_learnings to cuppings

The two prose fields `aromatic_behavior` and `structural_behavior` move from `roast_learnings` to `cuppings` (migration 062). They describe what a CUP IS — per-tasting observation tied to a specific cupping event — not what a LOT TAUGHT (lot-aggregate carry-forward, which stays on `roast_learnings.cultivar_takeaway` / `general_takeaway` / etc.). The fields were misplaced on `roast_learnings` since the schema was first drafted; the 2026-05-14 grilling session flagged the misplacement explicitly and the 2026-05-19 kickoff locked the relocation as the load-bearing decision of Sprint 11. Alternatives considered: **Option B** — keep the fields on `cuppings` but write each lot's prose to every cupping on `best_roast_id` (rejected — redundant data; the lot's reference cup is a single tasting, not a multi-cupping aggregate), or **Option C** — add the fields to `cuppings` while leaving them on `roast_learnings` as lot-level synthesis with distinct semantics (rejected — two-home ambiguity invites drift between the per-cup and lot-level prose, and the lot-level framing is what the relocation is correcting). Option A wins because (i) the per-cup framing is what Chris's existing prose actually describes — the 7 closed lots' content reads as reference-cup observation, not as cross-cupping rest-curve pattern (which is what `rest_behavior` correctly captures and stays on `roast_learnings` per Sprint T5 RO-7); (ii) lifting the fields to `cuppings` unlocks the cross-source synthesis Sprint 13 (SYN-6) plans, where brewing-side and roasting-side cup observations need to share the same per-cup primitive; (iii) it removes the "cup character (currently misplaced in schema)" caveat that CONTEXT.md § Cup character has carried since the schema was drafted, closing a known gap. Relationship to ADR-0001 (Two-Axis Brewing Framework): aromatic and structural behavior are the cup-side observation primitives the brewing-side framework consumes; lifting them to per-cup correctly aligns the cup-side substrate with the brewing-side framework's per-brew granularity. Reversal cost is moderate (the data move is reversible via SQL inverse on the 7 closed-lot rows, but the substrate-wide unwind — MCP Tool descriptions on push_cupping / push_roast_learnings, the four prompt files, ResolvedView render path, CLAUDE.md + CONTEXT.md prose, this ADR — is the real cost); recommend not reversing without comparable evidence that the per-cup framing is wrong. Migration data move strategy: per closed lot, write the existing roast_learnings prose to the canonical pourover cupping on `best_roast_id` (priority `eval_method ILIKE '%pourover%'` DESC, then `cupping_date` DESC, then `created_at` DESC, LIMIT 1); preflight on 2026-05-19 confirmed all 7 lots have a deterministic pourover-priority target. Locked 2026-05-20 via Sprint 11.
