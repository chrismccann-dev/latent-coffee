# 0009 — Carry-forward learnings carry per-field scope_tags arrays

The four carry-forward fields on `roast_learnings` (`cultivar_takeaway`, `terroir_takeaway`, `general_takeaway`, `starting_hypothesis`) each gain a paired `text[]` column `*_scope_tags` (migration 064). The scope-tag array holds loose-canonical namespaced prefixes (`process:washed`, `variety:sudan-rume`, `country:colombia`, `altitude:high`, `evaluation_method:day-7-pourover`, `density:high`, or `general` as a catch-all) describing the sub-scope the takeaway applies to. Prompts (`close-lot.md` STAGE 4 - renumbered from STAGE 3 in the 2026-07-15 reorder, backlog #42 - and `one-shot-closeout.md` STAGE 4) capture the convention; write paths do NOT enforce — the registry is loose-canonical by design at the 7-closed-lot corpus size. The Sudan Rume Washed → Natural transition is the canonical case study: an FC-temp-architectural-constraint hypothesis emerged exactly because some washed-anchor carry-forwards didn't transfer cleanly to the natural variant, and the only way to surface "which takeaways apply only to washed" was to grep prose for "For washed" prefixes — three of seven closed lots had explicit scope prose ("For washed Colombians" / "For XO-fermented coffees" / "For washed Guatemalan"), four of seven had unscoped general principles. Cross-lot queries are not reliable via grep when 4/7 lots carry implicit scope at all. Alternatives considered: **Option B** (per-lot `is_process_dominant`-style scoping flag — rejected as too coarse; the same lot can carry takeaways at different scopes — e.g. Rancho Tio's variety-signal-dominates-roast-state is variety-scoped while its cooling-curve-peak hypothesis is roast-state-scoped), **Option C** (extend ADR-0005's parameter-type-conditional signal arbitration framework to handle cross-process learning transfer — rejected as over-engineered for a 7-lot corpus where the current decision is "tag the takeaways for queryability," not "build a declarative transfer engine"), **Option D** (accept prose-only and tighten prompts to mandate explicit "For X:" prefix on every takeaway — rejected because the Sudan Rume case study shows that even with explicit prose prefixes, cross-lot queries require text-matching against a free vocabulary that drifts faster than a structured tag namespace). Option A wins because (i) the per-field framing aligns with the existing 4-takeaway schema shape — the carry-forward field's name already implies the PRIMARY scope axis (cultivar / terroir / general / starting_hypothesis), and `*_scope_tags` carries SUB-SCOPING within or across axes; (ii) `text[]` storage with prefix-namespaced tag vocabulary gives SQL-queryable structure (`WHERE 'process:washed' = ANY(general_takeaway_scope_tags)`) without forcing the rigidity of an enum or FK at a corpus too small to justify it; (iii) the loose-canonical convention is cheap to grow as the vocabulary stabilizes — prefix namespaces (`process:` / `variety:` / `country:` / `terroir:` / `altitude:` / `evaluation_method:` / `density:` / `general`) document themselves and can earn `makeCanonicalLookup` wrappers later if drift bites. Relationship to ADR-0005 (parameter-type-conditional signal arbitration): scope_tags is the data substrate that a future precedence-or-weighting framework would consume — locking the tags now means the framework, when it lands, has clean input. Relationship to ADR-0007 (brewing_tolerance rename) and ADR-0008 (aromatic/structural relocation): both are workflow-language alignments on the carry-forward layer; ADR-0009 is the third in the sequence, completing the carry-forward primitive's structural hardening at the 7-closed-lot corpus inflection point. Reversal cost is low — `text[]` columns are additive; if a future framework supersedes the tag convention, the columns can be dropped or ignored without data loss elsewhere. Backfill: NONE. Going-forward only — the 7 closed lots stay with empty arrays. Cross-lot scope-tag queries gain power as new lots close with tags populated. Locked 2026-05-21 via Sprint 12.
