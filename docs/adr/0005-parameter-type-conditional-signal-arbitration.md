# 0005 — Parameter-type-conditional signal arbitration (precedence on categorical / weighting on continuous)

Roasting design uses **two distinct signal-arbitration mechanisms in the same V1 design pass**, conditional on the output parameter's type: **Signal precedence** (one signal wins outright) governs categorical / discrete decisions like anchor profile selection, end condition mechanism (dev_time vs bean_temp), and drop ceiling regime; **Multi-factor weighting** (signals contribute hedged adjustments that sum, sometimes cancel, sometimes one dominates) governs continuous numerical adjustments like peak inlet hedge, fan-curve modifications, V1 spread width, and Maillard % tolerance. Alternatives considered: pure-precedence (one mechanism, mirror the brewing side's signal-override framework), pure-additive (one mechanism, what ROASTING.md's Green-Spec → Starting-Hypothesis table currently implies), or document the two mechanisms as separate but unrelated concepts. The parameter-type-conditional model wins because (i) the two mechanisms operate cleanly on different output classes — categorical decisions are non-additive (you don't sum two anchor choices), continuous adjustments are non-categorical (peak inlet isn't binary); (ii) lived practice already uses both, just unnamed; (iii) the cross-project asymmetry vs brewing is explained — brewing's output is categorical-only (the strategy enum) so precedence-only suffices, roasting's output is mixed so both coexist. Reversal cost is moderate (the eventual ROASTING.md doc-layer fix to split into two tables will be built against this decision; reversing means redoing the doc split). The round-7 cross-party audit further clarified that **anchor confidence × lot value** is a layered structure, not a head-on tension: lot value adjusts spread width (precedence rule #6); anchor confidence's widening pressure on expensive lots redirects to **pre-V1 risk reduction** (gating, paired roasted-reference cupping, deferral). Layering rather than overriding is the meta-pattern when two rules appear to compete on a parameter; check whether they're actually adjusting different parameters before declaring precedence.
