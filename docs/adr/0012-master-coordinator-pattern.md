# 0012 — Master Coordinator pattern + lazy-loading dispatch

> **Amendment 2026-05-27 (Research Assistant Step 2):** Sub-skill enumeration updated to 17 active per [ADR-0011 Amendment 2026-05-27](docs/adr/0011-composable-sub-skills-architecture.md#amendment-2026-05-27-research-assistant-step-2-scaffolding). Research Coordinator + Research Assistant added but **explicitly NOT dispatched by Master Coordinator** per [ADR-0017](docs/adr/0017-research-assistant-architecture.md) Exception 3 — research is operator-direct in Claude Code, not claude.ai-mediated. The Master Coordinator catalog lists them for visibility; dispatch-rules.md does NOT route into them.

The Master Coordinator is **a single dispatch surface** that lives at `docs/skills/coordinator/` and routes the operator's intent to the right Workflow tier sub-skill from the 17-sub-skill enumeration ([ADR-0011](docs/adr/0011-composable-sub-skills-architecture.md)) — **one coordinator across both brewing + roasting domains, not per-domain coordinators**, because cross-domain handoffs are real and growing (POD-1's roasting → brewing → roasting cross-project workflow, CCIL synthesis across both Historians, the future cross-pollination Wölfl-design pattern — note research projects are operator-direct in Claude Code per ADR-0017, not Master-Coordinator-dispatched), and two coordinators would need a third routing layer above them which is a leaky abstraction. **Wave 1 ships markdown-only** — the coordinator cluster contains `SKILL.md` (its own job-to-be-done spec), `catalog.md` (the 18-sub-skill registry plus domain principles at ~1-2KB each absorbed from BREWING.md's "Two-Axis Framework" + ROASTING.md's "roast for brewing tolerance, brew for intensity" + adjacent framing prose), `dispatch-rules.md` (intent → sub-skill mapping), and `handoff-rules.md` (cross-domain dispatch paths including the canonical Cupping Specialist Path A → Brewing Assistant → Close-Lot Specialist chain that today lives in `close-lot.md` STAGE 3 as manual paste-by-brew_id-reference). Code-backed dispatch (`lib/agents/coordinator.ts` with deterministic routing) is deferred to Wave 2+ if natural-language reasoning over the dispatch rules proves unreliable in practice; over-engineering the first ship is the failure mode to avoid. **Lazy-loading is the architectural primitive that solves the bloat-tripwire problem** — Master Coordinator's catalog metadata is always loaded (small, ~10-15KB total: domain principles ~3KB + 18-entry sub-skill index with I/O metadata ~7-10KB), but each sub-skill's `SKILL.md` + its `cluster/` content loads **only on dispatch**, meaning a typical claude.ai session loads 1-2 sub-skills + their 1-3 knowledge clusters = ~20-40KB working set, NOT the ~360KB monolithic-master-doc working set we'd carry without decomposition; the catalog knows *what exists* without holding *what they contain*. **Entry-point invariant preserved**: claude.ai sessions still enter via `docs/prompts/<workflow>.md` files (unchanged surface — `start-lot.md`, `log-roast.md`, `start-brew.md`, `bundled-brewing-completion.md`, etc.), but each prompt becomes a thin orchestration shell that fetches `docs://coordinator/catalog.md` at session start, identifies which sub-skill(s) it needs for its workflow, and lazy-loads them via the MCP Resource catalog (already wired through `lib/mcp/docs.ts`); no operator-facing slash-command surface is added because Chris doesn't use `/slash-skill` commands today (his entry point is prompt-paste-into-claude.ai, not Claude-Code-slash-invocation). **`.claude/skills/` stays as-is** — only `grill-with-docs/` + future Claude Code dev utilities; even Master Coordinator does NOT get a `.claude/skills/coordinator/` entry because it's invoked from claude.ai via the prompts layer, not from Claude Code via slash commands. **Cross-domain handoff dispatch flow** lands first-class in `handoff-rules.md`: when Cupping Specialist executes Path A (reference roast picked + reference cup data captured), the coordinator routes to Brewing Assistant with `green_bean_id` + reference-roast metadata for optimized brew dial-in; after the optimized brew's `push_brew` lands, coordinator routes to Close-Lot Specialist with `green_bean_id + reference_brew_id` to write `roast_learnings` + set `best_roast_id` + verify cross-links. This is the resolution of POD-1's "optimized brew lifecycle states" half — what was previously invisible workflow phase (manual paste between claude.ai/projects/roasting and claude.ai/projects/brewing) becomes an explicit dispatch path. **BREWING.md + ROASTING.md transition to redirect stubs in Wave 4**, ~500 bytes each: "Content migrated to `docs/skills/coordinator/catalog.md` + individual sub-skill clusters under `docs/skills/`. See coordinator catalog § brewing-domain / roasting-domain for the per-sub-skill index." The redirect stubs preserve back-compat for memory files / past PRs / CLAUDE.md doc-index references without holding content. Re-test the markdown-only-dispatch decision if (a) override rate on coordinator dispatch proposals exceeds 25% across any quarter (auto-demote signal per [ADR-0013](docs/adr/0013-self-improvement-primitives.md)), (b) the catalog itself crosses 30KB triggering re-introduction of the domain-shard layer we just collapsed, or (c) cross-domain handoff sequencing produces ambiguous routing more than ~3 times in a Wave 3+ session.
