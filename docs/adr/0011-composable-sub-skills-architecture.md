# 0011 — Composable sub-skills architecture replacing monolithic BREWING.md / ROASTING.md

> **Amendment 2026-05-27 (Research Assistant Step 2 scaffolding):** Original enumeration of "18 sub-skills + 1 deferred" updated to "**17 active sub-skills**" — Learning Assistant + Learning Knowledge deprecated (subsumed by Research Coordinator + Research Assistant per [ADR-0017](docs/adr/0017-research-assistant-architecture.md)); Research Coordinator + Research Assistant added with three deliberate architectural exceptions documented in ADR-0017 (Claude-Code-centric, planner/executor collapse, operator-direct dispatch). See § Amendment 2026-05-27 at the bottom of this ADR for the full diff.

The Latent reference architecture decomposes into **17 sub-skills organized across 3 tiers** (Knowledge / Workflow / Cross-Coffee Insight Layer) plus a **Master Coordinator** that lives outside the tiers — replacing the prior model where BREWING.md (213KB at decision time) + ROASTING.md (147KB at decision time) carried both reference content and operational guidance in monolithic files past the 120KB CLAUDE.md § Sprint cadence § Standing tripwires for ~2-3 weeks before this ADR landed, with both docs still growing ~19KB per arbiter batch (PR #198 + #200, 2026-05-26). **The Knowledge tier (8 active post-amendment: 7 prior + Research Coordinator added; Learning Knowledge retired) authors and maintains documents + their associated knowledge clusters** — Roasting Historian (closed-lot lessons), Brewing Historian (per-brew lessons + absorbs the existing BREWING.md "Cross-Coffee Insight Layer" section as brewing-internal cross-coffee — distinct from the cross-domain CCIL above it), WBC Roasting Archivist (with Sourcing Knowledge tentatively merged today; split when dedicated sourcing research starts), WBC Brewing Archivist, Brewing Equipment Expert (consolidating 4 existing `lib/{brewer,filter,grinder,sworks}-registry.ts` + 4 existing `docs/taxonomies/{brewers,filters,grinders,sworks}.md` files under one sub-skill; 8 existing assets in one cluster), Peer-Learning Roasting Archivist (low-volume operator-curated per Chris's "roasting is too machine-specific for broad web-scraping" framing), Roest Knowledge (machine + API + quirks; symmetry-promoted from the original audio note's "Roest Expert" workflow framing because authoring/maintaining Roest L200 Ultra docs is knowledge work, not workflow execution). The **Workflow tier (9 sub-skills post-amendment — 3 planners + 6 executors) splits across plan vs execute** — a 2nd axis Chris named directly ("all of my things really split between planning and executing — similar to how we do things here in claude code") with 3 planners (Roasting Assistant, Brewing Assistant, Sourcing Workflow Planner — Learning Assistant retired per amendment) and 6 executors (Roast Recorder, Brew Recorder, Cupping Specialist, Roest API Worker, Close-Lot Specialist, Research Assistant per amendment). **The Cross-Coffee Insight Layer (CCIL) is the cross-domain synthesis layer** above both domain Historians — two-level synthesis is locked: Brewing Historian + Roasting Historian each do *internal-to-domain* cross-coffee synthesis (cross-strategy / cross-cultivar within brewing; cross-lot within roasting), while CCIL synthesizes *across* both domains ("what I learned about Sudan Rume from BOTH my brewing of Hydrangea's Sudan Rume AND my roasting of CGLE Sudan Rume Hybrid Washed"). The **architecture is intentionally asymmetric between roasting and brewing workflows** because iteration depth differs — roasting iterates AT THE SUBSTRATE LEVEL (V-sets, comparative cuppings, experiment rows, all pushed to Latent; Cupping Specialist + Close-Lot Specialist exist because substrate-visible judgment loops need workers), brewing iterates IN-THREAD ONLY (only the final optimized brew lands in Latent; intermediate iterations stay in claude.ai thread context; no Palate Evaluator sub-skill needed because Brewing Assistant absorbs both initial recipe construction and in-thread iteration on tasting notes via a sub-prompt). **18-sub-skill enumeration absorbs POD-1**: the pour-over discriminator + optimized brew lifecycle states scoping work (audit cluster 2026-05-26 produced a DRAFT scoping doc) collapses into Cupping Specialist's Wave-3 implementation rather than running as a standalone sprint, because Path A/B/C routing + simulated-pourover-as-3rd-cup-read + cross-project handoff lifecycle states are all Cupping Specialist concerns. The **`docs/skills/<sub-skill>/` directory is the canonical home** — each contains `SKILL.md` (canonical spec: job-to-be-done + inputs/outputs + calls + MCP Tools + self-improvement signal), `cluster/` (knowledge-cluster docs for knowledge tier), and optionally `resources/` for supporting registries/taxonomies; `.claude/skills/` stays as-is (just `grill-with-docs` + Claude Code dev utilities; sub-skills are NOT exposed as slash commands because Chris's entry surface is `docs/prompts/*.md` via claude.ai, not `/slash-skill` in Claude Code); `lib/agents/` adds code helpers only WHEN automation is needed (most knowledge tier sub-skills won't have entries here; most executors will). **Operator-stubs-then-integration handoff** is the codified pattern for NEW knowledge clusters (Peer-Learning Roasting Archivist, Roest Knowledge, future Learning Knowledge) — Chris does the research + initial format proposal + stub authoring; Claude Code integrates into the architecture and applies. Mirrors the taxonomy-sprint precedent. Wave plan: **Wave 1 = Master Coordinator + Brewing Equipment Expert paired ship** (proof-of-pattern; tests catalog mechanics + first sub-skill in one PR — see [ADR-0012](docs/adr/0012-master-coordinator-pattern.md)); Wave 2 = 4 consolidation plays (Brewing Historian + Roasting Historian + WBC Brewing + WBC Roasting Archivists, pure existing-file reorganization); Wave 3 = 2 operator-stub clusters + 9 workflow tier sub-skills (POD-1 lands inside Cupping Specialist here); Wave 4 = CCIL + cross-system sync rewriting BREWING.md + ROASTING.md as ~500-byte redirect stubs. Re-test the architecture if total `docs/skills/` size exceeds 300KB (Pattern F tripwire from [ADR-0013](docs/adr/0013-self-improvement-primitives.md)), or if dispatch-accuracy degrades below the autonomy graduation threshold across the Workflow tier.

### Added sub-skills

- **Research Coordinator** — Knowledge tier (with Planning collapsed in), cross-domain, one per project, persistent. Owns the research-knowledge cluster, the roadmap, the cross-track design, the substrate-fold authoring. Different from every other Knowledge-tier sub-skill: carries Planning duties too because research methodology is co-authored per-track at Step 0, not pre-baked.
- **Research Assistant** — Workflow Executing tier, cross-domain, per-track ephemeral. Conducts research with operator, captures observations, produces handoff doc, terminates. No substrate writes.

### Retired sub-skills

- **Learning Assistant** — Workflow Planning. Was ACTIVE Wave 3 PR 2. Never ran a research track. Subsumed by Research Coordinator + Research Assistant.
- **Learning Knowledge** — Knowledge tier. Was DEFERRED (no active wave assignment). Never authored. Subsumed by the research-coordinator cluster.

Hard-removed in this amendment ship: `docs/skills/learning-assistant/` + `docs/skills/learning-knowledge/` directories deleted.

### Architectural exceptions (codified in ADR-0017)

The Research pair is subject to three deliberate exceptions from the standard sub-skill pattern documented in this ADR. Each is intentional and not drift:

1. **Claude-Code-centric, not claude.ai-mediated.** No MCP Resource registration in `lib/mcp/docs.ts`. No `docs/prompts/*.md` entry surface. No claude.ai catalog refresh. 6-actor cross-system audit collapses to 3 for sub-skill changes (Operator + Claude Code + Latent app, the last only for downstream substrate-fold).
2. **Planner + Executor collapse.** Coordinator does both planning + Knowledge-tier authoring. Assistant is execution-only. Different from the brewing/roasting Planner+Recorder split because research methodology is co-authored per-track at Step 0.
3. **Operator-direct, not Coordinator-dispatched.** Master Coordinator catalog lists the pair but does not dispatch into it. Entry surface is "operator types 'I want to start a research project' into a fresh Claude Code session."

See ADR-0017 § Architectural exceptions for the full reasoning.

### Audit considerations going forward

- **MCP Resource registration audits:** Research Coordinator + Assistant SKILL.md + cluster docs are deliberately NOT registered. `npm run check:mcp-bundle` should pass without them.
- **Master Coordinator catalog:** Research Coordinator + Assistant DO appear in the catalog for visibility, but dispatch-rules do NOT route into them. Catalog discipline + dispatch-rule discipline are decoupled for this pair.
- **Sub-skill count audits:** the 17-active figure is the post-amendment ground truth. Future architecture audits should reference this amendment, not the original 18+1 framing.
