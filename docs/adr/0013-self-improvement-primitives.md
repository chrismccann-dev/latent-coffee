# 0013 — Self-improvement primitives: 10 patterns + 3-stage autonomy ladder + bloat tripwires (Pattern J added Item 16c grill 2026-05-24; Pattern J defined / light-formalized 2026-06-03 — see Amendment)

Latent self-improves when **(1) new rows sharpen interpretation of prior rows** (reuse), **(2) substrate compacts as it grows rather than monotonically accumulating** (compaction), **(3) the system updates its own questions, not only the answers to fixed questions** (prompt-revision), and **(4) operator cognitive load decreases over time on the same workflow** (operator-load) — four tests that the composable sub-skills architecture ([ADR-0011](0011-composable-sub-skills-architecture.md)) + master coordinator pattern ([ADR-0012](0012-master-coordinator-pattern.md)) collectively satisfy via **10 self-improvement patterns** that compose at the per-sub-skill level. **Pattern A — Substrate-event refresh**: new substrate row of relevant type lands (e.g. push_roast_learnings → Roasting Historian, push_brew → Brewing Historian, push_cupping → Cupping Specialist) → cluster pattern docs refresh via `propose_doc_changes` arbited by operator (existing ARBITER.md playbook unchanged); the 5 PRs that landed during this brainstorm's authoring session (PR #196 - #200, 2026-05-26) are textbook Pattern A + C demonstrations of the existing pipeline running at Stage 1 today, which the new architecture formalizes rather than replaces. **Pattern B — External-event refresh**: external event detected (new WBC year drop, peer roaster discovery, Roest firmware release, API behavior drift) → archive cluster docs refresh via operator-initiated capture + `propose_doc_changes`. **Pattern C — Registry-expansion refresh**: new canonical entry / equipment acquisition / flow-rate measurement → equipment + taxonomy clusters refresh via registry edit + `propose_doc_changes` for surrounding prose; `taxonomy_overrides_queue` + `propose_canonical_addition` MCP Tools are the existing pipeline (PR #196 + #197 are recent live examples). **Pattern D — Tier-threshold refresh**: corpus tier crossing (early → emerging → established → mature per CONTEXT.md § Corpus tier) → pattern docs that benefit from larger corpus refresh via prompt-revision proposal. **Pattern E — Workflow-execution refresh**: sub-skill invoked → output observed → success/fail signal captured → workflow prompts + dispatch metadata refresh via workflow retro (manual today; semi-auto Wave 2+). **Pattern F — Bloat-tripwire decomposition**: cluster > 120KB OR single doc > 60KB → cluster topology splits into sub-clusters via prompt-revision + operator-confirmed split; CCIL is the archetypal Pattern F user (when CCIL grows too large to be one coherent layer, it self-decomposes into domain-specific sub-CCILs via prompt-revision triggered at the bloat threshold). **Pattern G — Catalog-update refresh** (Master Coordinator only): new sub-skill ships / existing sub-skill I/O changes → `catalog.md` + `dispatch-rules.md` + `handoff-rules.md` refresh via cross-system audit at sub-skill ship time (Actor 4 in the 6-actor matrix); intentionally stays at Stage 1 indefinitely (catalog audits gate every ship). **Pattern H — Dispatch-accuracy refresh** (Master Coordinator only): operator overrides proposed dispatch → log `(intent → wrong dispatch → correct dispatch)` tuple → `dispatch-rules.md` patches via periodic review of override log. **Pattern I — Operator-initiated resource integration**: operator brings new raw material (livestream transcript, blog post, sourcing book, etc.) → specific sub-skill cluster integrates new doc via Claude Code work session (operator + Claude Code collaborate on format + integration + cross-link); stays operator-driven indefinitely (one-off events by nature; not autonomous; example: Dongzhe livestream → transcripts → Peer-Learning Roasting Archivist cluster). **Pattern J — Substrate pruning** (added Item 16c grill, 2026-05-24, audio-ratified — light-touch ship; mechanism deferred pending case studies): substrate compaction via section-level removal as the inverse-orientation companion to Pattern F — Pattern F splits when cluster exceeds threshold, Pattern J prunes stale or over-detailed sections within a cluster regardless of total size. Counterbalance to ADD-shaped Patterns A-E which compound substrate monotonically; without Pattern J, context-window performance degrades over time as load-bearing content gets buried by stale + over-detailed sections that surface at retrieval time but don't earn their keep (Chris-articulated 2026-05-24: "if you bloat too much of the context window, then performance will decrease, and there'll just be too much stuff in your memory that's kind of irrelevant"). **Four candidate mechanisms preserved as scoping options** (locked at Item 16c, 2026-05-24): (mech-a) periodic pruning pass in the arbiter (analog to skeleton review / observing-list walk — Claude Code surfaces candidates, operator decides per-item delete/archive/keep); (mech-b) per-section recency markers ("no edits in N days + no read references = prune candidate") via git-log surfacing; (mech-c) per-section hard caps as Pattern F extension (likely too aggressive — sections vary in load-bearingness); (mech-d) section-recency metadata in frontmatter to enable mechanical "last-touched > N days" queries. **Case-study-driven generalization** (Chris-articulated 2026-05-24): mechanism shape locks once 2-3-4 lived pruning candidates have been worked through operator + Claude Code back-and-forth, deriving generalized rules from the worked examples — analog to the Research Coordinator + Research Assistant research-project methodology codified post-2026-05-27 per [ADR-0017](0017-research-assistant-architecture.md) — pick a real candidate → exercise it → derive a rule → repeat across 2-3 candidates → generalize. The filter arc (2026-05-21 → 2026-05-26) is the canonical worked example. Candidate-queue landing: [§ Substrate pruning candidates in docs/grilling-queue.md](../grilling-queue.md) is the initial home; promote to standalone file analogous to CCIL observing list if case-study volume warrants. **Existing case study** (verified 2026-05-24): PRODUCT.md § Recently shipped extraction to docs/sprints/shipped.md during May 2026 doc cleanup — sprint-by-sprint shipped log moved out of bloating PRODUCT.md with a 1-line pointer left in place; this is mech-a + extract-to-separate-file sub-mechanism, successful in lived practice. Leans toward mech-a (arbiter pass) as eventual target; full mechanism design deferred until case-study volume crystallizes the right shape. **Per-sub-skill pattern mapping** is tagged in each sub-skill's `SKILL.md § Self-Improvement` section: knowledge tier sub-skills use {A,D,F} (Historians) or {B} (Archivists) or {A,C} (Brewing Equipment Expert) or {A,B} (Roest Knowledge); workflow tier all use {E}; CCIL uses {A,D,F} with F as primary; Master Coordinator uses {G,H}; Patterns I + J apply cross-cutting (Pattern I = operator-initiated resource integration; Pattern J = substrate pruning) — any sub-skill can receive either; not per-sub-skill mapped. **3-stage autonomy maturity ladder** governs how operator-in-the-loop a sub-skill's self-improvement is: Stage 1 (in-loop) is the default for Waves 1+2 — every refresh proposal arbited by operator via `propose_doc_changes` pipeline; Stage 1 → Stage 2 (override) advances when N consecutive auto-approvals land without operator override (N=5 default; N=10 for substrate-writing executors because writes are irreversible; N=3 for CCIL because synthesis judgment is lower-stakes); Stage 2 → Stage 3 (autonomous) advances when override rate < 10% across 3 consecutive quarters; **the ladder is monotonic-with-rollback** — Stage 3 → Stage 2 AUTO-DEMOTES when override rate ≥ 10% in any single quarter (per Chris's "10% or more override is good to review" framing), and Stage 2 → Stage 1 AUTO-DEMOTES when override rate ≥ 25% in any single quarter; this gives the architecture a built-in immune system where a sub-skill that drifts (Roest API surface changes, new substrate column appears, lived practice diverges) demotes itself back to operator review without manual intervention. **Architecture-level bloat tripwires** apply Pattern F to the architecture itself: if `docs/skills/coordinator/catalog.md` exceeds 30KB → propose splitting the catalog into domain shards (re-introducing the BREWING.md + ROASTING.md domain layer this brainstorm collapsed); if individual sub-skill clusters exceed 60KB → propose self-decomposition (e.g. Brewing Equipment Expert might eventually split into Drippers / Filters / Grinders / Valves sub-skills); if total `docs/skills/` size exceeds 300KB → propose Wave-5 architectural retro. The 4 self-improvement tests at architecture level: **Reuse ✅** (Patterns A + D drive Historian + CCIL refresh on substrate writes); **Compaction ✅** (Pattern F at sub-skill + architecture level); **Prompt-revision ✅** (Patterns D + F + E drive prompt-revision proposals; full autonomous prompt revision is a far-future Stage-3 state); **Operator-load ✅** (lazy-loading reduces working set per session + Master Coordinator dispatch reduces "what skill do I need" overhead + autonomy ladder gradually removes operator gates from low-stakes patterns). Re-test the 9-pattern catalog + autonomy thresholds if (a) a self-improvement event observed in lived practice doesn't map onto any pattern, (b) the N=5/N=10/N=3 graduation thresholds prove too aggressive or too conservative across the first 2-3 quarters of Wave-2+ operation, or (c) the auto-demote signals fire so frequently that sub-skills oscillate between Stage 1 and Stage 2 without stabilizing.

## Amendment 2026-06-03 — Pattern J promoted from placeholder to defined (light formalization)

The case-study-driven deferral resolved. Three lived pruning exercises across three distinct doc-shapes worked the full shape-space: **001** CLAUDE.md (reference-doc → `extract` + `split`, PR #352), **002** CONTEXT-roasting.md (living-glossary → `consolidate` + `delete`, PR #365), **003** PRODUCT.md (roadmap → `archive` + `split` + `consolidate`, PR #368). All five sub-mechanism shapes — **extract / split / consolidate / archive / delete** — now have a standalone worked example. Full scoping + the case index live in [docs/features/doc-pruning-mechanism-brainstorm-2026-06-03.md](../features/doc-pruning-mechanism-brainstorm-2026-06-03.md); the structured handoffs in [docs/sprints/pruning-cases/](../sprints/pruning-cases/).

**Decision: light formalization, not a full mechanism build.** The strongest cross-case signal is that the **interpretive prune resists automation** — in all three, the operator reading the doc from first principles and re-clustering it was load-bearing. So Pattern J splits into an automated **trigger** half and an operator-led **response** half:

- **Trigger (automated).** `npm run check:doc-sizes` ([scripts/check-doc-sizes.ts](../../scripts/check-doc-sizes.ts)) computes live sizes for every constantly-loaded surface vs. the [doc-tripwires.md](../architecture/doc-tripwires.md) caps and exits non-zero when a Tier-1 surface is over; a daily CI cron (`doc-sizes-check.yml`) is the catch-all. This replaces mech-b/mech-d (recency markers / frontmatter) and the hand-maintained size column, which drifted 3× in 3 days. Scope is **loading-profile-aware across both surfaces** (Claude Code + the claude.ai brewing/roasting projects): root + CONTEXT-* + `docs/prompts/*` (40 KB full-read) + `docs/skills/*` clusters (per the ADR-0014 tiers — `check:doc-sizes` now also enforces those, and on its first run caught `brewing-equipment-expert` over its 150 KB cap, which the 2026-05-23 hand-audit had logged as "approaching").
- **Response (operator-led, manual).** When a tripwire fires, schedule a **post-tripwire pruning exercise**: Claude Code writes a kickoff brief + size-map; the operator does the interpretive restructure; Claude Code does the mechanical prune + the fidelity checks; the pair files a structured handoff in `pruning-cases/`. Mech-a (arbiter pass) remains the eventual home if this ever needs to run unattended, but it stays manual until the kickoff-writing actually becomes tedious — it hasn't.

**Defined sub-mechanism rules derived from the cases (001-007):**
- **Six shapes** (cases 006/007 extended the original five): extract / split / consolidate / archive / delete / **re-home (re-scope)**.
- **Doc-shape predicts prune-shape** (routing): reference-doc → `extract`/`split`; living-glossary → `consolidate`/`delete` (you cannot pointer-ize a glossary term without snapping its relationship web); roadmap/log → `archive`/`split`; registry-catalog → `split` on an owned/not-owned axis (keep the `lib/*-registry.ts` validator whole); **procedure prompt → `consolidate`-to-pointer** (re-stated glossary defs / examples / backstory are pointer-able; the stage skeleton + write-contracts + halt gates + MCP call sequences stay inline — no behavior change, case 006); **compounding Historian doc at its single-doc cap → `re-home`, NOT `split`** (case 007: tighten the doc's job, push single-axis detail to capsules behind routers; don't sacrifice synthesis to a byte target).
- **`archive`'s tell:** the content already has a historical home, so the prune is *ceasing to re-state it in the live surface*, not moving it (case 003 shed 72 KB while writing only ~35 KB of new files).
- **`re-home`'s tell:** the doc's *purpose* has drifted (a cross-anchor index became a catch-all rollup); reduce it by tightening the job definition + routing out-of-scope content to its canonical home. **Cross-cluster re-home is dedupe-and-rewire, not move** — the destination usually already holds ~90% of the displaced content (case 007b).
- **Cap working surfaces, never cap archives.** Archives (`docs/sprints/*`, `docs/roasting/archive.md`, `docs/features/*`, `docs/taxonomies/*`, `docs/adr/*`) are where pruned content goes to grow freely — `check:doc-sizes` excludes them by design.
- **Recurring safety steps** on every exercise: move-never-delete verification · concept-set diff after any structural reorg (case 002 silently dropped a concept a prompt referenced) · cross-ref / dangling-pointer check · **repo-wide anchor grep** `grep -rn '<doc>.md#<anchor>'` for *inbound cross-cluster* links (case 007b — the intra-cluster check missed one) · **destinations-first, index-last ordering** when capsules point into the index being thinned (case 007a) + **rewire back-pointers** that target a deleted index section · **preserve the append target** (rewrite the routing/workflow list to name the new homes or the next compounding refresh re-bloats the doc) · the six-actor audit (case 002 caught a stale MCP Resource description; note the Historian cluster docs *are* individually MCP-registered, so a new capsule/axis needs a `lib/mcp/docs.ts` entry).
- **`delete` stays flag-only**, with one carve-out: the **provenance-class** (grill-session breadcrumbs + deferred-future TODOs — git-recoverable, zero downstream consumers) is the candidate for the *first* autonomous-delete rule, kept distinct from concept-bearing deletes, which remain flag-only indefinitely.

A heavier mechanism (a `prune-doc` skill mirroring `grill-with-docs` / `consolidate-memory`) is deferred until the manual kickoff-writing shows friction; the accumulating `pruning-cases/` corpus is the input that would feed it.
