# Latent Glossary — Canonical Registries

Extracted from [CONTEXT-shared.md](CONTEXT-shared.md) in the Pattern J pruning sprint (2026-05-25) to bring the parent glossary under the ADR-0014 session-load tier cap (40 KB). This file holds the meta-language about Latent's canonical registry system — vocabulary that applies across all 10 taxonomy axes (cultivar / terroir / process / roaster / producer / brewer / filter / grinder / roast level / flavor).

Strict glossary per [grill-with-docs](.claude/skills/grill-with-docs/SKILL.md) format — no implementation details, only term definitions + cardinality relationships. The actual canonical lists live in [docs/taxonomies/](docs/taxonomies/) (authoritative markdown sources) and [lib/*-registry.ts](lib/) (validation mirrors); implementation specifics live in [CLAUDE.md](CLAUDE.md).


**Aggregation level**:
The per-axis choice of *which layer inside a taxonomy earns knowledge aggregation* — the canonical granularity that becomes both a destination page in the Latent app AND the unit against which Chris compounds cross-coffee learning. Picked per-axis, not by system-wide rule, because the right layer is axis-specific. **Three criteria a chosen layer must satisfy**: (1) **legible** — neither so broad that signal dilutes nor so specific that meaning fragments; (2) **enough samples** — sufficient brew / lot data density at the chosen layer for any aggregated view to carry meaningful signal rather than N=1 noise; (3) **transferable** — learnings at this layer carry over to other instances of the same kind in the future (the compounding-knowledge purpose). **Terroir is the canonical worked example**: country aggregates too broadly (mixes microclimates / subsystems / mountain ranges into one bucket — signal exists but too noisy to act on); farm aggregates too specifically (high signal per farm but too few brews per farm to draw conclusions, AND the registry explodes into too many sub-classifications to be useful as a taxonomy); macro terroir sits between (legible boundary, enough brews per macro, learnings transfer to other lots from the same macro). The same trade-off recurs per axis with different vocabularies — cultivar picks the individual cultivar (after the Sub Pages 3 lineage→cultivar inversion), process picks a composable shape (base hub + per-base modifier-combos + cross-base modifier-index, gated by Rule 1/2/3 thresholds), roaster + producer pick the individual canonical, flavor picks the individual base (family palette is render-only, not aggregation), roast level picks the individual Agtron-anchored bucket. **Aggregation level is a registry-design decision** baked into the markdown source + registry mirror; changing it post-hoc requires a registry re-cut + page-redesign sprint (the Sub Pages 3 cultivar inversion is the canonical precedent — moved aggregation from lineage to individual cultivar mid-corpus). Not every axis is treated as an aggregation surface in the same way — see **Aggregation-eligibility** for the page-bearing / page-deferred / validation-only trichotomy.
_Avoid_: "canonical granularity" (close, but loses the *knowledge-aggregation* framing — granularity is a render attribute, aggregation level is the unit-of-compounding decision), "aggregation scope" (under-specifies what is being scoped — the chosen layer is a level inside the taxonomy, not a scope around it), "registry depth" (close, but depth implies a fixed property of the taxonomy itself; aggregation level is the layer of that taxonomy that gets pages and learnings), bare "level" or "granularity" (overloaded — pin the qualifier).

**Aggregation-eligibility**:
The per-axis classification of *whether the chosen aggregation level is treated as a knowledge-compounding surface, an inventory / consistency surface, or somewhere in between*. Three statuses + an axis-status assignment table:

- **Page-bearing** — the axis earns a destination page in the Latent app today; the canonical level is the unit of both validation AND knowledge aggregation; per-page synthesis runs on the axis's anchor. Today: **terroir**, **cultivar**, **process**, **roaster** (Sub Pages 2 / 3 / 4 / 5).
- **Page-deferred** — the axis is treated as a first-class aggregation surface in Chris's mental model, the rich-registry scaffolding exists, but no destination page is built yet. Today: **producer** (121 entries × 27 fields rich; page is queued, not blocking). Producer earns the deferred-page status because **growing the bean is a bigger lever than roasting or brewing** and producer is the only canonical surface that names *who is doing that growing* — knowledge aggregation will materialize when the page lands.
- **Validation-only** — the axis carries a canonical registry to enforce consistency on write but is **not** a knowledge-compounding surface. The registry's role is vocabulary discipline / equipment inventory / cross-roaster comparability, not "what I learned about this brewer / filter / roast level / flavor." Today: **brewer**, **filter**, **grinder**, **roast level**, **flavor**.

**Per-axis rationale for the validation-only camp** (each axis has a distinct "why"):
- **Brewer + Filter + Grinder** are **equipment-tracking axes** — a brew names what kit was used; "what I learned about the Hario V60" doesn't generalize the way "what I learned about Diego Bermúdez" does because the brewer is a tool, not a coffee. Filter additionally carries an unfinished content layer (real flow-rate measurements per filter — Chris's TODO; today's fast / medium / slow tagging is approximate). Grinder is intentionally **self-only** (only EG-1 in the registry, the only grinder Chris owns) and the registry's *content* is particle-size analysis (D50 / Zone / Behavior / Use Case) on Chris's own dial.
- **Roast level** is a **cross-roaster comparability anchor** — roasters use marketing language ("Nordic Light", "Modern Light") whose meaning drifts wildly between roasters (one's light is another's medium), so the objective Agtron-anchored bucket exists to project all of those marketing tags onto one shared scale. Chris's own CM200 readings on inventory anchor the bucketing for self-roasted lots.
- **Flavor** is a **vocabulary-consistency discipline** — the 3-axis composable shape (BASE + MODIFIERS + STRUCTURE_TAGS) plus 112 aliases exists precisely to enforce "same phrase = same mapping every time, the system stays clean only if the rules are followed" (flavors.md Rule 17). The discipline IS the aggregation value; per-flavor pages would not compound.

**Aggregation-eligibility is not permanent**: a validation-only axis can be promoted to page-deferred or page-bearing if Chris's mental model shifts (the roaster promotion is the precedent — early Latent treated roaster as just-an-attribute on brews until the Jeferson Motta wrong-zone trap surfaced the roaster's brew guide / house style as a load-bearing signal, after which the roaster registry was rebuilt rich and a destination page was built). Promotion costs a registry re-cut + page-design sprint + content-authoring sweep — high enough that today's validation-only assignments are stable, but the framing keeps the door open.
_Avoid_: bare "aggregation-eligible" (under-specified — the trichotomy is the unit, not a binary), "page status" (close but loses the *whether-it-earns-knowledge-aggregation* framing — a page is just one surface of the knowledge layer), "axis tier" (overloaded — producer tier is a different concept), "validation-only registry" (close but `validation-only` is one of three statuses; the umbrella term is aggregation-eligibility).

**Producer sourcing surface**:
The second-order role the **producer** canonical registry plays — distinct from its first-order role as an aggregation level. Producer is the only axis in the taxonomy whose canonical entries are *people Chris can build relationships with* in the real world, and those relationships are the **championship-mode green-bean sourcing pipeline**: in-person rapport built at coffee events and expos, Instagram contacts (the `ProducerEntry.contact` field carries the social handle), introductions through roasters who already source from the producer. The framing carries weight because **growing the bean is a bigger lever than roasting or brewing** in cup attribution — agricultural inputs (cultivar genetics / terroir / processing decisions made at the farm) define the response surface Chris's downstream work then shapes — but Chris does not own land, will not own land, and cannot directly affect agricultural input himself. The only access path is *who he knows*, which makes the producer registry a **relationship network**, not just a knowledge surface.

**Importance asymmetry across the pipeline**: producer matters less on the **brewing side** (purchased roasted beans — producer is mostly informational, nice-to-know context) and dramatically more on the **roasting side** (green-bean sourcing — producer is the access path to the unique experimental lots that competition-grade roasting requires). The asymmetry tracks with how close the work is to the agricultural product: brewing operates on roasted output (two layers removed from the farm), roasting operates on green (one layer removed), so the roasting-side workflow's value-per-producer-relationship is materially higher.

The sourcing surface is **not modeled in the schema beyond the `ProducerEntry.contact` field** — relationship state (have I met them? what did we talk about? did they send me a sample? are they responsive?) lives outside the app today. Open future-scope question: does producer relationship state warrant its own table once the championship-mode sourcing volume grows?
_Avoid_: "producer relationship layer" (close but verbose), "sourcing network" (close — but the registry IS a network; this entry is specifically about *the second-order role* the registry plays), "producer access path" (under-specified — the surface is broader than just access), bare "producer dimension" (overloaded).

**Authoritative source**:
The hand-authored markdown file under `docs/taxonomies/<axis>.md` (`varieties.md`, `regions.md`, `processes.md`, `producers.md`, `roasters.md`, `brewers.md`, `filters.md`, `grinders.md`, `roast-levels.md`, `flavors.md`) that **is the taxonomy** — canonical list, alias map, per-entry rich content, design rules, sources. Latent's source-of-truth for every canonical axis: when the markdown and the TS validation mirror disagree, the markdown wins; when a new canonical lands, the markdown is the file that records the decision (with the registry mirror following in the same PR per the 2-step deliberate edit). Each authoritative source carries a fixed header preamble — Enforcement bar / Canonical registry pointer / Last adopted / Adoption path — followed by the canonical list and (for rich axes) per-entry reference content. The markdown is served as a `docs://taxonomies/<axis>.md` Resource via `lib/mcp/docs.ts` so claude.ai can read canonical content during workflow authoring; the file is also human-readable for arbiter sessions + future Chris-as-reader sessions. **The dual-file pattern (markdown + TS registry) was originally a Claude Code organizational choice**, not a deliberate Chris design, but the split holds because it cleanly separates *what the taxonomy is* (markdown) from *how the taxonomy is enforced at write time* (TS).
_Avoid_: bare "the markdown" (axis-specific files exist for non-taxonomy docs too — BREWING.md, ROASTING.md, etc. — pin the *authoritative* framing), "taxonomy doc" (close but under-specifies — every doc is a doc; this one is the canonical content registry), "taxonomy file" (informal), "source of truth" (correct semantically but ambiguous against the application DB which is ALSO a source of truth for instance-level data), "taxonomy markdown" (descriptive — fine in prose, but "authoritative source" is the canonical noun for this glossary).

**Validation mirror**:
The TypeScript file under `lib/<axis>-registry.ts` (`cultivar-registry.ts`, `terroir-registry.ts`, `process-registry.ts`, `producer-registry.ts`, `roaster-registry.ts`, `brewer-registry.ts`, `filter-registry.ts`, `grinder-registry.ts`, `roast-level-registry.ts`, `flavor-registry.ts`) that **mirrors the authoritative source into in-line code-callable validation**. Exports the canonical name array + alias map + (for rich axes) the typed entry object + the `CanonicalLookup` bundle constructed via `makeCanonicalLookup(names, aliases?)` from `lib/canonical-registry.ts`. Consumed at every canonical-resolution write site (MCP push handlers, find-or-create helpers, the patch-route canonicalize blocks, the `/add` + `/edit` save paths' `CanonicalTextInput` save-gates). **The TS file's whole job is mirroring** — it carries no canonical decisions of its own; if the markdown changes, the TS changes in the same PR. Validation mirrors are how the system *enforces* the authoritative source at write time. **The phrase "validation mirror" is already in active use** in every authoritative source's header preamble ("Canonical registry: lib/<axis>-registry.ts (validation mirror)") and in CLAUDE.md's canonical registries section.
_Avoid_: "registry file" (under-specified — both files are part of the registry as a whole), "TS registry" (informal — works in prose, but the canonical noun pins the *mirror* relationship), "validation file" (close but loses the *mirror* framing), "enforcement layer" (overloaded — could mean the runtime validation pipeline that uses the mirror), "code registry" (close — but the markdown is also a registry; the TS distinguishing feature is being a code-callable mirror of it).

**2-step deliberate edit**:
The standing operational discipline that adding (or renaming, or aliasing) a canonical entry on any axis is exactly **two coordinated file edits in one PR** — (1) the **authoritative source** markdown under `docs/taxonomies/<axis>.md` (per-entry content + canonical list + alias map updated as needed) and (2) the **validation mirror** under `lib/<axis>-registry.ts` (canonical name array + alias map + rich entry object kept in lockstep with the markdown). A **third step is added when an existing DB row needs renaming** (a SQL migration under `supabase/migrations/`) — making it a 3-step edit for that case. The word "deliberate" is load-bearing — canonical additions are **never auto-promoted** from drift; they pass through arbiter judgment (`process pending arbitration` for queued NET-NEW values, or direct Chris-driven decision when a new producer / roaster / equipment item enters scope). The discipline prevents both **drift** (markdown changes that don't reach the validation mirror — writes still succeed but on stale canonical content) and **category bloat** (TS additions that aren't reasoned through in the authoritative source's authored content).

**Drift between the two files is the primary failure mode the discipline prevents** — it's the canonical-registries analog of the cross-system consistency problem the MCP / Sync Architecture cluster names. Today there is no automated check that the markdown and TS file agree (queued — `check:registry-sync` script shipped PR #112, pre-push hook + GH Action queued per `memory/project_check_registry_sync_followups.md`); the discipline is enforced by Chris-as-arbiter at edit time.
_Avoid_: "two-step edit" (informal; "2-step deliberate edit" is the canonical noun with the deliberate qualifier), "registry edit" (under-specified — could mean either file), "canonical add" (under-specified — adding is one of several operations the discipline covers; renaming and aliasing also pass through it), "dual-file edit" (close but loses the *deliberate* / no-auto-promotion framing).

**Alias**:
A non-canonical input value paired with the canonical it resolves to, surfaced by the `CanonicalLookup` factory's `findClosest` + `canonicalize` methods. Every authoritative source carries an `## Aliases` section + the validation mirror carries an `<AXIS>_ALIASES` map. Aliases do NOT make `isCanonical` return true — they resolve writes through to the canonical target without polluting the canonical list. Three structurally distinct shapes (the third process-specific):

- **Drift alias** — mechanical surface-form variation of the **same canonical thing**. No judgment, no collapse, no semantic interpretation. Spelling / case / diacritic / whitespace / hyphenation / word-order variants, plus group-label normalization. Examples: `Geisha → Gesha` (English-vs-Latin spelling), `Volcan Baru Highlands → Volcán Barú Highlands` (diacritic), `EG1 / Eg-1 / Weber EG-1 → EG-1` (case + hyphen + prefix drift), `Sudán Rumé → Sudan Rume` (claude.ai's default Spanish-style accent rendering), `Catimor → Catimor (group)` (label normalization). Cheap to add, surfaced as soon as the variant appears in the wild. **Drift aliases protect downstream content attachment** — every measurement, taxonomy fact, or per-entry rich-content cell hangs off the canonical name, so a write that drifts off-canonical disconnects from all of it (Chris's grinder example: "all my measurements and grind taxonomy are attached to EG-1; every Weber-EG variant has to resolve back to it").
- **Structural alias** — surface name that names a **conceptually distinct but referentially-identical** thing, collapsed to the canonical by Chris-as-arbiter judgment. The most-judgment-laden alias type; carries semantic interpretation. **Producer is where this shape concentrates**, driven by the **producer naming overload** — a roaster may print the farm name, the producer's individual name, OR the washing station name on the bag for the same canonical producer, so structural aliases collapse all three surface forms back to the producer canonical. Examples: `Alo Village → Tamiru Tadesse` (village → producer running the station), `Café Granja La Esperanza / CGLE → Rigoberto & Luis Eduardo Herrera` (farm name → producers, and the acronym), `Higuito → Nelsyn Hernández` (farm → producer), `Heirloom → Ethiopian landrace population` (catch-all bag-name → canonical for unidentified Ethiopian lots). Each structural alias is a real promote-or-collapse decision and passes through arbiter judgment, never auto-applied from drift.
- **Decompositional alias** (process-specific) — surface name that resolves to a **structured composition** (base + modifier-arrays + qualifier + signature) rather than a single canonical string. Different shape because the process axis is composable (see Composable taxonomy). Examples: `Anoxic → Anaerobic + qualifier:Anoxic` (canonical + orthogonal annotation), `Tropical Washed → base:Washed + fermentation:[Yeast Inoculated] + intervention:[Fruit Co-ferment]` (full multi-axis decomposition of a roaster's marketed phrase). **The motivation is shifting interpretation from runtime to compile-time** — process naming varies wildly across roasters, most roasters don't know the underlying base/modifier breakdown, the producer is the actual source of process truth (and is unevenly transparent), so claude.ai would have to figure each surface name out at write time without a decompositional alias. Pre-resolving the most common compound phrases in the alias map keeps the runtime experience pick-not-author.

**Origin**: aliases emerged from the pre-Latent spreadsheet + EA-Belle compilation era — Chris and Belle had recorded the same canonical thing under many surface forms over time, and when Chris ported it into the first Latent precursor app, "none of these things collapsed down correctly." The alias map was the retroactive collapse mechanism that consolidated the legacy drift; new aliases since then are forward-looking — added as new variants surface from real-world writes (mostly claude.ai's defaults differing from Chris's canonical naming).

**Aliases are additive-only** — once an alias lands in the map it stays, because new writes may still reference it. Removing an alias requires a DB-wide migration to canonicalize any existing rows using the alias as their stored value first; in practice almost no alias has ever been removed.
_Avoid_: "synonym" (close but synonym implies surface-form equivalence; structural aliases collapse conceptually-different surface names), "mapping" (under-specified), "lookup entry" (close but the lookup is the broader mechanism), bare "redirect" (overloaded with URL routing).

**Pick-not-author principle**:
The standing system-design goal that **claude.ai's role at write time is to pick from a pre-resolved canonical surface, not to author canonicals or research at runtime**. Chris's framing (2026-05-15): the registries exist so that "if we got the taxonomy down 95-96%, claude.ai is not authoring anything or coming up with stuff or having to do research at runtime — it's just going down the list and picking the correct things." Concretely: when claude.ai writes a brew or a green-bean lot, every canonical-axis field (cultivar / terroir / process / roaster / producer / brewer / filter / grinder / roast level / flavor) should resolve cleanly against an existing canonical (or canonicalize through a drift alias / structural alias / decompositional alias) without claude.ai needing to interpret, research, or reason about what category the input falls into. The remaining 4-5% of writes hit the NET-NEW branch and go through `propose_canonical_addition` or the override-flag path — those are the *exceptions* the system is built to surface explicitly, not the default mode.

**The principle is what makes everything else work**:
- **Decompositional aliases** exist because the principle says claude.ai shouldn't be parsing "Tropical Washed" into base + modifiers at runtime; the alias pre-resolves it.
- **Drift aliases** exist because the principle says claude.ai shouldn't be reasoning about whether "Geisha" and "Gesha" are the same coffee at runtime; the alias resolves it.
- **The rich-registry shape** (27 fields per producer, 29 per roaster) exists so that picking a canonical also pulls in the canonical's reference content — no runtime research needed.
- **The propose-then-apply tier of asymmetric write trust** exists because the principle implies claude.ai's direct writes should be high-trust on canonical fields — and that trust is only safe if the canonical surface is pre-resolved to the 95-96% threshold.

**The 95-96% target is asymptotic, not a fixed line** — Chris's expected saturation projection from the MCP cluster says new-canonical rate decays over time as exposure broadens; the target rises with corpus saturation. Equipment axes hit ~100% (Chris owns what he owns); cultivar / terroir are already comprehensive (~99%); producer / roaster / process / signature_method are the long-tail axes that determine whether the system holds at 95-96% in practice.
_Avoid_: "no-author rule" (close but under-specifies — the principle is about the canonical layer doing the work, not just claude.ai abstaining), "pre-resolved registry" (close — but the principle is about the discipline, not just the artifact), "canonical-completeness target" (close but loses the *picking* framing), "registry saturation" (close — saturation is a related but distinct concept, see Asymmetric write trust's saturation projection).

**Hierarchical taxonomy**:
The structural shape of two canonical-registry axes — **cultivar** and **terroir** — where canonical entries sit inside a multi-level parent-child taxonomy with one specific level designated as the aggregation level and the levels above it as **parent levels** (render context, not validation targets). Distinct from **composable taxonomy** (flavor / process — multi-axis stacking, not parent-child) and from **flat taxonomy** (the other 7 axes — one canonical name per entry, no parent structure).

Two instances today:
- **Cultivar — 4-level ladder**: Species → Genetic Family → Lineage → individual Cultivar. Aggregation level = the individual cultivar (post Sub Pages 3 inversion, 2026-05-11; previously aggregated at lineage). Parent levels = species + genetic family + lineage — surface as breadcrumb context on `/cultivars/[id]` and as section headers on `/cultivars` index (Species → Family → Lineage → linked cultivars), but they're not validation targets in `CULTIVAR_LOOKUP` — they're metadata fields on `CultivarEntry`.
- **Terroir — 3-level ladder**: Country → Admin Region → Macro Terroir. Aggregation level = macro terroir. Parent levels = country + admin region — surface as breadcrumb on `/terroirs/[id]` and as section headers on the index. Meso and Micro terroir were deliberately **demoted below canonical** (Sprint 1d.1, 2026-04-22) and live as free-text annotations on individual terroir rows — every behavioral attribute is macro-scoped, so canonicalizing the lower levels added registry weight without per-level signal.

**The hierarchy is bottoms-up discovered, not top-down imposed** (Chris's framing, 2026-05-15) — cultivar genetics and terroir geography are *naturally hierarchical in the real world* (a cultivar belongs to a species; a region belongs to a country), so Chris built each taxonomy bottoms-up from what the real-world referents actually look like, then iterated on which level was the right aggregation target. The other 8 canonical axes don't have hierarchical real-world referents at the same depth (a brewer is a brewer; a roast level is a bucket on a scale; a producer is a person), which is why none of them adopt the multi-level ladder. **Aggregation-level choice within a hierarchy was not certain at registry-build time** — clarity emerged through iteration (spreadsheet → precursor app → Latent → synthesis surfaces); the Sub Pages 3 lineage→cultivar inversion is the canonical evidence that aggregation level can shift even when the hierarchy itself is fixed.

**Parent level**:
A level *above* the aggregation level inside a hierarchical taxonomy — used for breadcrumb context, index-page section headers, and color palette grouping, but **not** as a validation target. Cultivar's parent levels = species / family / lineage; terroir's parent levels = country / admin_region. The parent levels are exposed via the rich registry entry (`CultivarEntry.species` / `.family` / `.lineage`; `TerroirEntry.country` / `.adminRegion`) so render code can read them, but they don't participate in `CanonicalLookup` validation — only the aggregation-level name is canonical-checked at write time. **Parent levels are derived from the aggregation-level entry, not separately authored** — picking a canonical cultivar pulls in its species / family / lineage automatically; there's no separate `species` field to canonicalize.
_Avoid_: "ancestry level" (close but anthropomorphic — coffee taxonomy isn't biological descent), "ancestor", "higher level" (under-specified — higher than what?), bare "level" (overloaded — could mean aggregation level), "breadcrumb level" (close but breadcrumb is one render of parent levels, not the level itself).

_Avoid (for Hierarchical taxonomy)_: "nested taxonomy" (close — but nested could imply arbitrary depth; hierarchical taxonomy in Latent is shallow + fixed-depth per axis), "tree taxonomy" (under-specified), "multi-level registry" (close but loses the *natural-real-world-structure* framing — the depth comes from the domain, not from registry design), "taxonomic ladder" (descriptive but informal).

**Composable taxonomy**:
The structural shape of two canonical-registry axes — **flavor** and **process** — where the canonical "value" is a **structured composition** drawn from canonical fragments across multiple orthogonal axes, not a single canonical name. Distinct from **hierarchical taxonomy** (cultivar / terroir — parent-child ladder) and from **flat taxonomy** (the other 7 axes — one canonical name per entry).

Two instances today:
- **Flavor — 3-axis composable**: **BASE** (182 flavors across 12 categories — Acid / Beverage / Defect / Floral / Fruit / Nut / Roasted / Savory / Spice / Sweet / Tea & Herbal / Vegetative) names *the noun in the cup*; **MODIFIERS** (43 across 10 categories with **strict priority order** Transformation > Form > State > Processing > Intensity/Sweetness/Texture/Blending/Other > Structure, **max 2 per flavor**) name *what happened to the noun*; **STRUCTURE_TAGS** (29 across 7 axes Acidity / Body / Clarity / Finish / Sweetness / Balance / Overall) name *how the cup feels system-level* (per-coffee, not per-flavor). Storage: `brews.flavors jsonb` carries `{base, modifiers[]}` objects; `brews.structure_tags text[]` carries `"Axis:Descriptor"` keys per coffee. The 3 axes are **orthogonal** — any base × any modifiers (modulo priority + max-2) × any structure tags (subject to "2-3 across axes" soft rule).
- **Process — multi-axis composable** (8 structured fields on `brews`): `base_process` (1 of 4: Washed / Honey / Natural / Wet-hulled — required, DB CHECK constraint) + `subprocess` (Honey color tiers only, 7 values — optional) + 4 stackable modifier-array axes (`fermentation_modifiers` 13 / `drying_modifiers` 5 / `intervention_modifiers` 7 / `experimental_modifiers` 4) + `decaf_modifier` (1 of 4 — optional single-value) + `signature_method` (proper-name proprietary — optional single-value). Modifiers stack **flatly within each axis** with no priority order, unlike flavor.

**The compositional shape compresses a combinatorial vocabulary into a tractable registry** — flat-listing every meaningful flavor phrase (`Strawberry Jam`, `Blueberry Muffin`, `Stewed Apricot`, `Lemon Marmalade`, ad infinitum) would balloon to thousands of canonicals and lose the compositional grammar that makes the vocabulary self-explaining. The 182 × 43 × 29 = ~225K nominal flavor recipe combinations against a ~254-row registry is the compression ratio that makes the axis maintainable. Similarly process: ~1.5M process compositions against a ~57-row registry (4 + 7 + 13 + 5 + 7 + 4 + 4 + 14 working signatures). **The decompositional alias map (process-specific) pre-resolves the highest-frequency surface phrases into their structured forms** ("Tropical Washed" / "Anoxic" / "Yeast Anaerobic Natural" / etc.) so claude.ai picks instead of decomposes at write time — the Pick-not-author principle's compositional-axis variant.

**Composable taxonomies were bottoms-up authored from real-world surface vocabulary** (Chris, 2026-05-15) — same discovery pattern as hierarchical taxonomies. **Flavor authoring sources**: things Chris has tasted, tasting notes on roaster bags (large sample), and the **Le Nez Du Café Coffee Aroma Kit** (the standard 36-vial aroma reference); the 3-axis decomposition emerged after compiling the surface vocabulary, not as a top-down design. **Process authoring sources**: producer-side process descriptions + roasted-coffee-side process labels (which often differ — producer terminology is more technical, roaster terminology is more marketing); the base + subprocess + modifier-array structure emerged from analyzing what *makes up* a process. **Compositional sub-mapping within an axis**: when a single modifier value carries enough structural variance to fragment writes (e.g. "co-ferment" could mean fruit / floral / spice / cascara), the axis splits into a sub-mapping that resolves the variants to canonical sub-types (`Co-ferment` umbrella + `Fruit Co-ferment` / `Floral Co-ferment` / `Spice Co-ferment` / `Cascara Co-ferment` specific variants — Chris's example, 2026-05-15).

**Picker shape**: a composable-taxonomy picker is fundamentally **multi-field** (`<ProcessPicker>` / `<FlavorComposer>`) — not a single typeahead — and a save-gate that validates each axis separately (`isProcessResolvable` checks base resolvability; `cleanFlavors` checks each chip's base + modifiers; `cleanStructureTags` checks each tag). The picker shape mirrors the registry shape: composable input for composable canonical.
_Avoid_: "multi-axis taxonomy" (close — but "composable" carries the *structured-composition-as-the-canonical-value* framing more cleanly), "decomposable taxonomy" (close but decomposable is the direction of analysis; composable is the direction of writes), "compound taxonomy" (overloaded with chemistry), "stacked taxonomy" (close but stacking is the within-axis modifier-array shape, not the whole-axis-set shape).

**Coverage strategy**:
The per-axis policy choice that determines how far the canonical registry **tries to reach** into the real-world domain — full coverage, partial coverage with named tail-acceptance, owned-anchored coverage, or self-only. Determines (a) how aliases vs NET-NEW are treated on that axis, (b) whether the override-flag path is wired (or not), and (c) Chris's expected new-canonical rate. **Four strategies** today, each tied to what the real-world domain looks like for that axis + Chris's relationship to it:

- **Comprehensive** — covers the full real-world domain. Strict-axis on writes; NET-NEW is rare (genuinely-new cross-bred cultivar; region missing from the macro list). **Bottoms-up authoring**:
The recurring methodology Chris uses to author every page-bearing / page-deferred canonical taxonomy — applied uniformly across **cultivar**, **terroir**, **flavor**, **process**, and **producer**. Distinct from top-down design (where the abstract structure is decided first and instances slot in). **The pattern is iterative-convergent** and runs in four phases:

1. **Collect** — bottoms-up data pull from the existing corpus: every coffee on Chris's covered roasters' bags, plus external research sources where the corpus is thin (Le Nez Du Café for flavor, World Coffee Research + Rob Hoo's *Cultivar* for cultivar, ChatGPT-assisted research for producers + terroir geography, producer-side + roaster-side process names for processes). The collection is exhaustive *within scope* — every instance the corpus surfaces gets pulled before any abstraction starts.
2. **Choose tentative aggregation layer + draft taxonomy** — given the data, propose a layer (macro terroir, individual cultivar, base + modifiers for process, base + modifiers + structure for flavor, Anchor / Signal / Experimental for producer) and a structural shape (hierarchical / composable / flat).
3. **Test against the collected data** — does everything fit cleanly, or are there **substrate gaps**? Iterate the taxonomy until most cases fit; if a substrate gap doesn't close, either expand the taxonomy (add modifier axis / split a base / introduce qualifier) or accept the gap (deliberately scope out — see meso/micro terroir demotion).
4. **Continuous refinement on new data** — once shipped, every new roaster / coffee / lot tested against the taxonomy is an update opportunity. Drift aliases land freely; structural aliases pass through arbiter; substrate-gap surfaces in real writes trigger a registry revision (e.g. the v8.4 Hybrid promotion on the brewing side is the brewing-axis analog of this iteration).

**The pattern is what produces the bottoms-up-discovered hierarchies and bottoms-up-discovered compositions** that the Hierarchical taxonomy + Composable taxonomy entries describe — the shape is never imposed top-down; it falls out of what the real-world surface vocabulary actually looks like. **ChatGPT served as Chris's research assistant for the data-collection + initial-draft phases** (especially producer-system research and terroir geography) — bounded to the taxonomy authoring side; not part of role separation (Chris doesn't use ChatGPT for building, arbitration, or input). Most authoring is now stable; the methodology is less actively used going forward as the registries reach saturation.
_Avoid_: "iterative authoring" (close but loses the *bottoms-up-from-data* framing — iterative could be top-down too), "data-driven taxonomy" (close — but the data is the input, not the entire methodology), "grounded taxonomy" (vague), "empirical authoring" (close but academic-flavored), "bottoms-up top-down" (Chris's verbal phrasing — accurate but redundant; bottoms-up is the active step, the top-down framing is the *output* once the layer is chosen).

**Substrate gap**:
A case in the bottoms-up collected data that doesn't fit cleanly into the current draft taxonomy — the diagnostic signal during iterative authoring that the taxonomy needs revision (add a modifier axis, split a base, introduce a qualifier, demote a layer below canonical, or deliberately scope out). Chris's verbatim term during 2026-05-16 grilling: "do they all fit cleanly into this thing, or is there a substrate gap?" Substrate gaps are evidence that the taxonomy isn't yet covering the real-world surface; closing them is what makes a taxonomy reach the pick-not-author 95-96% target.

**Two responses** to a substrate gap:
- **Expand the taxonomy** — add structure that closes the gap (e.g. v8.4 Hybrid promotion on brewing — the modifier+strategy gap closed by promoting Immersion to a first-class strategy with sub-forms; the process taxonomy's stackable-modifier-array axes — closed the gap that flat process names left when one process carried 2-4 simultaneous interventions).
- **Deliberately scope out** — accept the gap and exclude the case from canonical coverage (e.g. meso / micro terroir demotion — Sprint 1d.1 confirmed every behavioral attribute is macro-scoped, so meso / micro became free-text annotations; cinnamon / vienna / french trade-name roasts excluded from roast-level registry because Chris's roaster set doesn't use them).

Closing a substrate gap is **not always promotion** — sometimes the right answer is "this surface form doesn't earn a canonical and shouldn't try to." The 2-step deliberate edit makes the response decision the load-bearing one, regardless of which path is chosen.
_Avoid_: "taxonomy gap" (close — gap is the canonical noun, substrate is the load-bearing qualifier that anchors it to the real-world data), "coverage gap" (close but coverage gap is a different concept — coverage strategy gap means we deliberately don't cover; substrate gap means the data exists but the taxonomy can't yet receive it), "edge case" (under-specified — every coffee has unique attributes; the substrate gap is specifically a case that *resists structural canonicalization*).

**Qualifier**:
A process-axis-specific **orthogonal annotation on a canonical modifier** that preserves a meaningful execution distinction without forking a new modifier — enumerated (not free text), optional, paired with a specific canonical modifier rather than standing alone. Today's only qualifier: **`Anoxic` on `Anaerobic`** (sealed-container / no-headspace / near-zero-oxygen execution of the broader Anaerobic fermentation modifier). The decompositional alias map handles the surface input: `Anoxic → Anaerobic + qualifier:Anoxic`. **Storage layer pending** — `processes.md` defines the qualifier and the alias map but `brews` does not yet carry a `fermentation_qualifiers text[]` field; qualifier metadata only persists on inputs that preserve it (the legacy `brews.process` display string, `what_i_learned` prose). Flagged as a follow-up sprint.

**Aggregation level stays at the modifier, not the qualifier** (Chris-locked 2026-05-16) — explicit corrective on `processes.md`'s framing that "Anoxic Natural typically calls for Full Expression while Anaerobic Natural typically calls for Suppression." The strategy-distinction line was sharper than Chris actually wants in practice. The right framing: **qualifiers are record-when-known annotations, not strategy-decision layers**. Reasoning: (a) the qualifier vocabulary itself is gray-area in the wild — most roasters + most producers don't specify Anoxic vs Anaerobic cleanly, so the qualifier signal is sparse and noisy; (b) the underlying agricultural reality is the biggest cup-attribution lever Chris doesn't have direct access to — process information flows roaster ← producer ← agricultural reality with two layers of opacity; (c) cross-coffee learning would over-fit on a noisy signal if pulled at the qualifier layer. Conclusion: **the qualifier exists to preserve information when it's known, but the aggregation level for knowledge AND strategy stays at the modifier**.

**The honey subprocess works the same way** (same logic, different mechanism): the 7 honey color tiers (White / Yellow / Red / Black / Purple / Generic / Hydro) are structurally a sub-axis on `Honey` base, but **aggregation level stays at base Honey** — most roasters + most producers don't specify the color tier, so default writes collapse to Generic Honey; specific tiers get recorded only when explicitly known. The structural-subprocess case is implemented as a dedicated `subprocess` column rather than a `qualifier` annotation (honey's color tier is structured enough to warrant its own column), but the **aggregation logic is identical** — sub-canonical detail is informational-when-present, not a learning surface.

**The qualifier pattern is generalizable** — any future case where a canonical modifier carries a meaningful execution variant that's too sparse / too gray-area to fork a new canonical is a qualifier candidate. The signature_method axis was an earlier candidate; Sprint T1 / BR-1 (2026-05-18) resolved both halves of the original "Wave Hybrid as a qualifier on Hybrid Washed" example differently — Wave Hybrid was promoted to a full signature canonical, and Hybrid Washed was deprecated (it failed the "mechanically opaque" criterion). The pattern stays open for future cases.
_Avoid_: "sub-modifier" (close — but sub-modifier reads as hierarchical / nested; qualifier is orthogonal annotation), "modifier qualifier" (verbose; "qualifier" is the canonical noun), "modifier variant" (close but loses the *orthogonal-annotation* framing), "execution flag" (under-specified — flag is overloaded), bare "annotation" (under-specified — many things annotate; qualifier is specifically the enumerated-modifier-companion).

**Objective bucket**:
The canonical value on the **roast level** axis — one of 8 Agtron-anchored buckets covering whole-bean color from ~130 (raw-ish underdeveloped) down to 0 (carbonized): Extremely Light / Very Light / Light / Medium Light / Medium / Moderately Dark / Dark / Very Dark. **Each bucket owns a 10-unit (or wider at extremes) Agtron range**, and a measurement on Chris's Lighttells CM200 maps deterministically to exactly one bucket via the SCA color naming convention the instrument exposes. **The objective bucket is what `brews.roast_level` IS** — purely the measured (or estimated) color level, not how the roaster markets the coffee.

**Two resolution paths into the objective bucket**:
- **Measured path** (clean, objective) — Chris owns the bag at peak rest, runs the CM200 on whole-bean, the instrument returns an Agtron number that resolves to the SCA color word that maps to one of the 8 buckets. Lossless. **Today only applied to coffees that pass through the post-purchase inventory workflow** (a recent practice; not historical coffees in Chris's freezer).
- **Inferred path** (noisy, marketing-tag-resolved) — when no CM200 reading exists, two inputs combine: the bag's wording (Light / Medium / "Ultralight" / "Nordic Light" / etc.) and Chris's knowledge of the roaster's typical roast style. Best-guess resolution against the objective bucket scale. Today's most common path for non-inventoried coffees.

The 8 buckets are **closed-objective coverage** — the Agtron scale is mechanically bounded at 0-130, so the registry covers the full domain. NET-NEW for this axis is conceptually impossible; only re-bucketing of existing data would change canonical content, which would require both registry edits and a DB migration.
_Avoke_: "Agtron bucket" (close — but Agtron is the measurement scale, objective bucket is the canonical layer on top of it), "SCA color" (close — SCA is the source of the color words, but the bucketing decisions are Chris's), "roast level" (overloaded — `brews.roast_level` is the column name; objective bucket is the canonical value within that column), "color bucket" (under-specified — color is one signal, objective bucket carries the full Agtron + SCA mapping).

**Marketing tag**:
Roaster-positioning language used on bag copy / website / brand identity that **describes how a roaster brands their roast direction**, NOT a measured color value. Examples: `Nordic Light`, `Ultra Light` (or `UL` — the most common in third-wave specialty bags), `Specialty Light`, `Modern Light`, `Omni Light`, `Omni Roast`. **None of these map cleanly to the Agtron scale** because they're marketing constructs — "Nordic Light" overlaps roughly 78-88 Agtron but the center varies per roaster; "Ultra Light" overlaps roughly 88-100 Agtron but again varies; some marketing tags (`Omni Roast`) have no clean Agtron correspondence at all.

**On `brews.roast_level`, marketing tags are deliberately collapsed into objective buckets via the alias map** (`Nordic Light → Very Light`; `Ultra Light → Extremely Light`; `Modern Light → Light`; `Omni Light → Medium Light`; etc.) — each marketing tag resolves to the **single objective bucket with the greatest overlap at the tag's center of gravity**. The collapse is intentional: `brews.roast_level` is the objective signal, and routing marketing tags through it would smuggle roaster-positioning into the measurement column.

**Marketing tags belong on the roaster, not the brew** (Chris's framing) — the right home for `Nordic Light` / `Ultra Light` / etc. as a *persistent positioning signal* is a future `roaster.roastStyle` column (queued sprint 1h.3), where the marketing tag describes how a specific roaster characteristically positions their work. The alias-to-objective-bucket collapse on `brews.roast_level` is the holding pattern until 1h.3 lands.

**The pattern (marketing positioning split from objective measurement, routed via alias to the objective canonical) is roast-level-specific today** — none of the other 9 axes have a clean marketing-positioning dimension that overlays the canonical. But the pattern generalizes: any future case where a noisy positioning vocabulary obscures a clean measured / objective canonical is a candidate for the same split.

**Inventory measurement workflow** (Chris-articulated 2026-05-16, not in `roast-levels.md` today) — Chris's process for converting purchased-roasted bags into measured-path objective-bucket data: (1) at buy time, mark the peak-rest date on the bag (typically 4-6 weeks post-roast for Moonwake-style washed coffees, varies by roaster + cultivar + process per the `roaster.restCurve` field); (2) at peak rest, open the bag, run the Lighttells CM200 to capture whole-bean Agtron; (3) record the reading in Chris's **inventory sheet inside claude.ai** (distinct from the Latent app's `green_beans` inventory, which is self-roasted-only); (4) portion the remaining beans into 15g doses in scientific vials, vacuum-seal, freeze. The inventory sheet is what claude.ai cross-references at recipe-design time when working with stored coffee. The workflow started recently — many older coffees in Chris's freezer pre-date it and have no measured Agtron, which is what creates the inferred-path dependency.
_Avoid_: "roaster tag" (close but loses the *marketing-positioning* framing), "roast style tag" (close — but roast style is the persistent property; marketing tag is the language used to express it), "positioning tag" (under-specified), "bag word" (informal — useful in prose), "ultralight" / "Nordic Light" without the umbrella (specific instances, not the umbrella term).

**Auto-created provenance**:
The audit-trail enum recording **whether a canonical FK row was matched against an existing entry at write time (`canonical`) or freshly materialized by the write itself (`auto_created`)**. Today's storage: `green_beans.terroir_provenance` + `green_beans.cultivar_provenance` + `brews.terroir_provenance` + `brews.cultivar_provenance` (Phase 3, migration 045), paired with `green_beans.canonicals_updated_at` timestamp that bumps when the FK changes via patch or future canonical re-resolution. The provenance flag is set by the find-or-create write path: alias-lookup → canonical-check → if no canonical row exists, materialize one with full metadata from the registry entry AND set `*_provenance = auto_created`; if the canonical row was already present, set `*_provenance = canonical`. Either way the write succeeds — provenance never blocks the write; it just records what happened so the arbiter procedure can review later.

**Provenance + override-flag are dual mechanisms for the same concept** — both record whether a canonical-axis input matched an existing canonical vs. lazily materialized, but use different storage shapes:
- **FK-creating axes (cultivar, terroir)**: provenance enum column on the entity (`canonical` | `auto_created`), because these axes have FK rows that exist independently of any brew / green_bean
- **Text-only override-eligible axes (producer, roaster, brewer, filter, grinder, signature_method)**: `*_override: true` boolean parameter at write time + queue entry in `taxonomy_overrides_queue`, because these axes have no FK row — the canonical is the string itself. (signature_method joined Sprint 12 / MCP-1, 2026-05-21, migration 063.)

The two mechanisms achieve the same outcome (the write succeeds + the canonical-promotion decision lands in arbiter scope) but reflect the underlying axis storage shape. **Strict axes (cultivar / terroir today) have one more state**: if the find-or-create can't even resolve canonically-or-aliasably, the entity write fails fast and claude.ai's recovery path is `propose_canonical_addition` — a separate Tool that queues the canonical decision without the entity write succeeding. The strict-fast-fail behavior is what distinguishes FK-creating-strict (cultivar / terroir) from FK-creating-with-override-fallback (which no axis uses today but the architecture would support).

**The provenance flag is the operational mechanism for the Pick-not-author principle's exception path**: when pick succeeds, write proceeds with `canonical` provenance + full registry-metadata materialization; when pick fails, the write doesn't block at the claude.ai layer — it falls back to materialize-with-flag + queue for asynchronous arbitration. This is what lets claude.ai's authoring workflow remain unblocked even on imperfect canonical coverage.

**Per-axis NET-NEW behavior expected** (Chris-articulated 2026-05-16):

- **Roaster**: auto-create + arbitrate; pretty good coverage but never complete

**Skeleton entry**:
A canonical entry promoted with **minimal authored content** + an explicit `skeleton: true` flag indicating that rich-field research is pending — the resolution to "promote now with the data we have OR wait until research is done" when a NET-NEW canonical earns promotion but the data at promotion-time is incomplete. Sprint 12 / CR-10 (2026-05-21) extended the flag from producer-only to `RoasterEntry` (Latent flagged as the inaugural roaster skeleton; audit-pass for the 21 sprint 1h.1 net-new roasters deferred to incremental arbiter sessions). The flag would extend to brewer / filter when those rich-registry axes need it. Today's instances: 5 producers (Miguel Estela / Nelsyn Hernández / Jannette & Kai Janson / + 2 MCP-feedback batches) + 1 roaster (Latent).

**The flag makes under-researched promotion visible rather than silent** — a skeleton entry looks deliberately different from a richly-authored one (page render can surface "needs research" indicators; arbiter sees the flag during queue sweeps; future readers know the thin content is a known incomplete state, not assumed-done). The flag is **monotonic-then-deleted** — flip to false (and remove the flag entirely) when the rich content is backfilled; never deleted without the content being filled, because the canonical still has to exist for downstream brews to reference.

**Skeleton entries surface in the arbiter procedure as a third queue surface** (Chris-locked 2026-05-16; shipped Sprint 12 / CR-4, 2026-05-21). The arbiter procedure walks three queues now: `doc_proposals` (prose) + `taxonomy_overrides_queue` (canonical promotions) + skeleton entries (this third surface). The new `list_skeleton_entries` MCP Tool (Tool count 34 → 35) surfaces skeleton rows from `lib/producer-registry.ts` + `lib/roaster-registry.ts` during "process pending arbitration" runs; ARBITER.md § Skeleton review walks the playbook (S1–S5). The arbiter's job per skeleton entry: (a) confirm the canonical really belongs as a separate entity (vs. should-be-aliased / should-be-rejected as drift retroactively); (b) if yes, fill in the remaining rich-field content — collaboratively between Chris (domain knowledge) and Claude Code (research + drafting). **Structural distinction from the queue trio**: no DB row, no status flip — the `skeleton: true` flag flips false (or is removed entirely) by editing the registry source file. The registry edit IS the state transition. This closes the **self-improvement loop on the canonical layer** — skeleton entries flow through arbitration into full rich entries, the registry gets richer per arbitration run, and the Pick-not-author principle's runtime experience gets thicker (claude.ai's pick at runtime returns more registry metadata, less authoring required).
_Avoid_: "stub entry" (close — but stub is overloaded with code-test-stub meaning), "partial entry" (under-specified — many entries are partial in some sense; skeleton is specifically the flagged-incomplete shape), "draft entry" (close — but draft implies not-yet-canonical; a skeleton entry IS canonical, just thin), "placeholder" (under-specified — could mean many things), bare "minimal" (under-specified).

**Family palette**:
The deliberate-hue-separated color-coded parent-level grouping applied across **five canonical-registry axes** today — roaster (6 strategy families), cultivar (8 genetic families: 5 Arabica + 3 non-Arabica), flavor (8 families: Citrus / Stone Fruit / Berry / Tropical / Grape & Wine / Floral / Tea & Herbal / Sweet & Confection + Other), process base (4 hues for Washed / Honey / Natural / Wet-hulled), country (12 country hues for terroir). Each axis's palette lives in a dedicated `lib/*-colors.ts` or as part of the registry mirror's `getFamilyColor` exports; each follows the design-conventions rule **hue-not-lightness** — if a signal deserves its own color, shift hue, not lightness.

**Distinct from Parent level**: Parent level is **hierarchical-taxonomy-specific structural ancestry** (species/family/lineage for cultivar; country/admin/macro for terroir); Family palette is the **render + semantic color scheme** keyed to a parent-level grouping. They overlap on cultivar (genetic family is both a parent level AND a family-palette unit) and terroir country (a parent level AND a family-palette unit), but family palette extends to non-hierarchical axes too: roaster strategy families and flavor families have no underlying parent-child structural taxonomy, just the family-coded color grouping. **Distinct from Aggregation level**: family palette is always *above* the aggregation level (groups for visual hierarchy on index pages, never the canonical unit itself).

**Three functions of family palette**:
1. **Visual grouping on aggregation index pages** — `/roasters` groups by family with family-colored section headers; `/cultivars` groups by species → family → lineage with family-coded swatches; `/processes` groups by base color.
2. **Color-coded chips / tags on detail pages** — strategy pills, flavor-note family coloring, terroir country swatch on `/terroirs/[id]`.
3. **Synthesis-prompt anchor context** — per-entity synthesis adapters (terroir / cultivar / process / roaster) reference the family color or the family name as anchor metadata in the directed prompt.

**Origin** — the family palette emerged because the precursor claude.ai-local app rendered every brewing recipe card the same color, which became monotonous and hard to scan as the archive grew. The first color schema was **flavor-note-based** — floral → pinkish tone, fruity → brighter, tea-like → greenish, chocolatey → browner / darker. Once that was working, the same monotony problem recurred on terroir / roaster / cultivar / process indexes, and the pattern extended axis-by-axis with each axis's families getting a deliberate hue mapped to its semantic content. **Color carries semantic encoding** (Chris's framing): clarity-first → light + delicate hues; extraction push → red (aggressive); suppression → slate. The palette is *informational*, not decorative — at-a-glance index browsing surfaces "this is stronger" / "this is more tea-like" / "this is heavier" without reading the text. **The framing comes from coffee being organic material** with flavor + aromatics + sensation; the visual encoding has to carry more dimensionality than a typical app to keep scanability when the corpus grows.
_Avoid_: "color scheme" (under-specified — too generic; family palette is specifically the canonical-axis-keyed grouping), "axis palette" (close — but the per-axis framing is part of family palette's definition, not an alternative noun), "family colors" (informal — works in prose, but `Family palette` is the canonical noun for the umbrella pattern), "render palette" (close but loses the *semantic-encoding* framing), "category palette" (close but category is overloaded — flavor BASE has categories distinct from families).

**Strategy tag**:
The **roaster-axis-specific leaf-positioning label** that names how a roaster characteristically directs their roasting + brewing intent — more granular than family palette, mechanically collapses to family via `STRATEGY_TAG_FAMILY` in `lib/roaster-registry.ts`. 10 strategy tags today + SELF-ROASTED, mapping to 6 families: `CLARITY-FIRST` → Clarity-First; `BALANCED` → Balanced; `BALANCED → CLARITY` → Balanced (slight clarity skew on cooling); `BALANCED → FULL` → Extraction-Forward; `BALANCED → HIGH` → Extraction-Forward (synonymous with FULL EXPRESSION); `FULL EXPRESSION` → Extraction-Forward; `SYSTEM` → System; `VARIES` → Varies; `BALANCED / VARIES` → Varies (Strait, Switch); `SELF-ROASTED` → Self-Roasted.

The **`→` arrow syntax** in tags like `BALANCED → FULL` names a *skew direction* — the roaster's home position is BALANCED but they push toward FULL when the coffee + process calls for it. Distinct from `FULL EXPRESSION` (no skew — Full Expression is the home position). The arrow encodes positional nuance without forking a new family.

**Cross-axis vocabulary coherence rule** (Chris-locked 2026-05-16) — strategy tag vocabulary on roaster must align with extraction strategy canonical names on brews. "My Clarity-First when I brew things should match the Clarity-First on a roaster's house style — we can't have too much drift between those." Today's alignment is **partial**: roaster `CLARITY-FIRST` ↔ brew `Clarity-First` align; roaster `FULL EXPRESSION` ↔ brew `Full Expression` align; but brew `Suppression` / `Extraction Push` / `Hybrid` have no direct roaster-tag equivalent (no roaster positions as Suppression-specific; `BALANCED → HIGH` is the closest tag to Extraction Push; no `Hybrid` roaster tag because Hybrid is a structural strategy distinct from extraction-intensity positioning), and roaster `SYSTEM` / `VARIES` / `SELF-ROASTED` are roaster-only constructs with no brew equivalent (a brew always names a specific strategy, even on a SYSTEM or VARIES roaster's coffee). The coherence rule is **aspirational** today and earns a vocabulary-audit follow-up.

**Strategy tag is the input to the Roaster signal at Coffee Brief Step 1c** — claude.ai reads the roaster's strategy tag (via the `roaster.strategyTag` field on the canonical entry, propagated through the synthesis-prompt anchor) as one of the three signals (variety / process / roaster) feeding strategy-direction arbitration. **Roaster strategy = starting point, not destination** (Chris's framing): roaster house style + published brew guide give claude.ai a strong typical-brewing inference, factored into the Coffee Brief but not copied — Chris's own brewing reality may diverge through iteration. Same shape as the roasting-side "expectations vs. reality" delta — sometimes the brewed cup matches the roaster's intent, sometimes it doesn't, and the divergence is signal worth recording.

**Origin — the Motta story** — pre-Motta, Chris's strategy palette didn't include `FULL EXPRESSION` as a roaster tag. The Jeferson Motta anaerobic-washed Gesha wrong-zone trap (Moonwake-roasted, originally iterated as Clarity-First, eventually unlocked by Full Expression — already canonical in CONTEXT.md as the brewing-iteration-loop-within-strategy example) forced the strategy tag to be added retroactively. Same iterative authoring pattern as the rest of the canonical-registries cluster — strategy tags emerged bottoms-up from real brewing experience, not designed top-down. Moonwake itself carries `VARIES` rather than a fixed-direction tag — they don't roast for one characteristic, the strategy is process + coffee dependent, and that fact itself is the load-bearing signal.
_Avoid_: "house style" (close — house style is the broader concept that strategy tag is one input to; see existing Roaster signal / House style entry), "positioning label" (under-specified — many things are positioning), "strategy direction" (close — but direction is part of the tag content via the `→` syntax, not the whole tag), bare "tag" (overloaded), "roaster strategy" (informal — fine in prose; "Strategy tag" is the canonical noun for the registry field).
