# Latent Glossary — MCP / Sync Architecture

Extracted from [CONTEXT-shared.md](CONTEXT-shared.md) in the Pattern J pruning sprint (2026-05-25) to bring the parent glossary under the ADR-0014 session-load tier cap (40 KB). This file holds the cross-cutting infrastructure terminology that defines how claude.ai, Claude Code, and Latent stay in sync — vocabulary used by both roasting and brewing workflows.

Strict glossary per [grill-with-docs](.claude/skills/grill-with-docs/SKILL.md) format — no implementation details, only term definitions + cardinality relationships. Architectural decisions live in [SYNC_V2.md](SYNC_V2.md); arbiter procedure lives in [ARBITER.md](ARBITER.md); implementation specifics live in [CLAUDE.md](CLAUDE.md).

**Championship-mode**:
The disposition Latent operates in: training for **World Brewers Cup** / **World Roasting Championship**-level mastery — not cafe-repeatable, not commercial, not chasing a universal "best" roast or brew. Every coffee is treated as a unique material whose ideal expression has to be found per-coffee, and every brew or roast is an opportunity to learn something generalizable. Drives the per-coffee optimization stance, the willingness to push extreme axes (Full Expression / Extraction Push on one end, Suppression / very dark on the other), the **archive-driven brewing** + **iterative roasting** workflows, and the system-level investment in compounding knowledge rather than locking down a single recipe. Articulated across PRODUCT.md / BREWING.md / ROASTING.md / CLAUDE.md in slightly different phrasings; all of them name the same disposition.
_Avoid_: "competition-mode" (close but champion is Chris's canonical framing), "pro-mode", "championship-grade" (the discipline is championship-LEVEL, not grade), "WBC-mode" (Brewers Cup is one half — Roasting Championship is the other), "hobby-mode" (the disposition is explicitly serious, not casual).

**System self-improvement loop**:
The compounding-knowledge mechanism the whole Latent architecture is designed to make possible: every brew or roast that lands in Latent feeds the next one, both through **structured aggregation** (per-cultivar / per-terroir / per-process / per-roaster pages, carry-forward learnings on closed lots) and through **prose improvement of the living docs** (BREWING.md, ROASTING.md, per-roaster cards, taxonomy markdown) via `propose_doc_changes`. The **MCP-only input principle** keeps the loop coherent (one canonical author, one canonical taxonomy); the **propose-doc-changes pipeline + arbiter procedure** closes the loop (claude.ai stages improvements during a workflow, Claude Code arbitrates them in batch, Vercel auto-deploys the merged result, claude.ai's next session reads the improved doc). End-state: time-to-reference-roast and time-to-reference-brew shrink with every closed lot / closed brew, because the substrate is smarter than it was last week.
_Avoid_: "self-improving system" (close but the noun is the LOOP — the system is Latent), "feedback loop" (generic), "compounding-knowledge loop" (close — but compounding knowledge is the SUBSTRATE of the loop, already covered by `carry-forward learnings`), "the loop" (under-specified — there are at least 3 named loops: the brewing iteration loop, the V-set roasting loop, and this one).

**Role separation**:
The 4-way separation of labor that lets Latent function as a self-improving system. **claude.ai** (specifically the `claude.ai/projects` surface — the brewing + roasting project threads, not the bare model API) is the **input** — authors every entity that lands in Latent (brews / roasts / cuppings / experiments / roast_learnings) plus every doc-change proposal, validating each write against canonical taxonomies. **Claude Code** (running locally in a Chris session) is the **builder + arbiter** — its dominant role is building the Latent codebase (features, schema, page shapes); **arbitration is one specific operational routine within that broader builder role**, triggered by "process pending arbitration" and walking the `doc_proposals` + `taxonomy_overrides_queue` queues to keep DB / docs / taxonomies in sync. **Latent** (bare noun — the application + Postgres + the MCP server itself) is the **system** — encoded in code, lives on independent of either Claude, single source of truth for state. **Chris** is **physical execution** — roasts, brews, cups, tastes, judges, makes the judgment calls; does not author records, does not insert into the app, does not edit. The boundary between input and physical execution is enforced by the **MCP-only input principle**; the boundary between builder and arbiter is internal to Claude Code (same model, two operational modes).
_Avoid_: "the workflow" (under-specified), "the actors", "four-role model" (jargon — the canonical noun is just "role separation" or "division of labor"), "Claude" without qualifier (ambiguous between claude.ai-input and Claude-Code-builder/arbiter — always pin which surface).

**Latent MCP server**:
The single canonical input/output plane between the Latent application and every Claude system that interacts with it — **claude.ai** (the brewing + roasting workflow input), **Claude Code** (the builder + arbiter), and the **Roest L200 machine** (the external roasting device, via the Roest API integration). Hosted as Next.js API routes on the same Vercel deployment as the Latent app (co-located, not a separate process); single-tenant + single-user by deliberate choice. Exposes two atomic primitives — **Tools** (named write actions + Tool-shaped reads) and **Resources** (URI-addressable read-only catalog/context). Two auth paths land at the same `user_id` for RLS scoping: OAuth 2.1 + PKCE for the claude.ai web client, bearer token for Claude Code + desktop MCP clients.

**Primary direction is input** — claude.ai writes brews / roasts / cuppings / experiments / roast_learnings into the Latent app via Tools — with two secondary write-back surfaces layered on: **`propose_doc_changes`** (internal — claude.ai stages prose improvements to living docs in a queue Claude Code arbitrates) and the **Roest API integration** (external — push/pull profiles + inventory + roast logs against the physical machine). Read surfaces (docs:// + canonicals:// Resources, plus Tool-shaped reads) serve the input side, not analysis: they exist so claude.ai writes against the same canonical vocabulary the Latent codebase enforces. The server is the load-bearing substrate for the **MCP-only input principle** (per `feedback_mcp_only_input.md`) and for the **system self-improvement loop** (input + arbitration + doc-improvement compounding to shrink time-to-reference on every future lot / coffee).
_Avoid_: "the app's MCP server", "Latent's MCP server", "the MCP layer" (bare), "sync server" (legacy from the pre-MCP V1 sync attempt — superseded by V2 in Sprint 2.2, 2026-04-28).

**MCP-only input principle**:
The standing rule that **claude.ai writing via the Latent MCP server is the canonical input path for every entity that lands in Latent** — brews, green_beans, roasts, roast_recipes, cuppings, experiments, roast_learnings, doc proposals, inventory rows, Roest profiles. No manual DB inserts; no paste-into-spreadsheet flows; no hand-authoring; no drafting; no manual editing. The principle has two paired motivations: **(1) the opportunity** — claude.ai writes flow through canonical-validation + `propose_doc_changes`, so every input contributes to the **System self-improvement loop** rather than just landing inert in the DB; **(2) the constraint** — the canonical taxonomy + process decomposition + signature method + Hybrid sub-form picker + 4-axis modifier stack + 4-state lifecycle + per-roast / per-cupping / per-experiment / per-recipe schema have collectively gotten complex enough that no human (including Chris) can author a write that resolves cleanly against every canonical surface in their head. Giving the input role to the system component that holds the most context (claude.ai, with docs:// + canonicals:// Resources and read_doc / read_canonical Tools available) is the only tractable path.

Historical context: pre-Latent, Chris hand-entered every row on a spreadsheet; his EA Belle compiled context, Chris arbitrated. The first Latent precursor (a Claude project) mimicked that workflow with Chris-as-arbiter inserting EA-compiled records into the app. The MCP-only input principle emerged once the schema complexity outgrew the EA-arbiter workflow. **Going back is functionally not viable** (Chris, 2026-05-15) — the system has gotten complex enough that the spreadsheet path can't reproduce a valid write. Roasting-side `/add?type=self-roasted` deprecated 2026-05-13 (Sub Pages 6.6); purchased-brew `/add?type=purchased` and `/brews/[id]/edit` are queued future deprecations pending claude.ai brewing-write workflow stability — the principle is **partially enforced today** (roasting absolute, brewing aspirational).
_Avoid_: "MCP-only writes" (the principle is about input/authoring, not about all writes — physical roasting still produces writes, just via claude.ai-as-author), "claude.ai-only input" (under-specifies the path — claude.ai writes happen specifically via MCP, not any other channel), bare "MCP-only" (under-specified), "no-input rule" (Chris does provide input via voice/text to claude.ai — he just doesn't author DB records directly).

**Cross-system consistency principle**:
The standing rule that **claude.ai, Claude Code, and Latent must always share the same vocabulary, canonical taxonomy, and operational rules** — drift between any two breaks the System self-improvement loop because the logic stops working and the input starts assuming things that aren't true. Chris's framing (2026-05-15): the three system components have to stay in sync with each other and always have a shared language, or that's where the confusion starts. Concretely: every Latent-specific term defined in `CONTEXT.md` (registered as the `docs://context.md` Resource) is consumed by claude.ai during workflow authoring; every canonical taxonomy lives in `lib/*-registry.ts` + `docs/taxonomies/*.md` and is exposed via both the `canonicals://*` Resource family and the `read_canonical` Tool (the **dual-surface pattern**, a separate entry); every operational rule (lifecycle state machine, one-shot constraints, signature method canonical list, Hybrid sub-form picker, modifier types, lever vs variable post-hoc promotion) is encoded somewhere consumed by both the input side (claude.ai authoring) and the validation side (Latent rejecting non-canonical writes). Sync is maintained by four mechanisms: **(1) live Resource reads** — claude.ai pulls the canonical state at write time, never from a frozen copy; **(2) validation on push** — Latent rejects writes that don't resolve canonically; **(3) `propose_doc_changes`** — claude.ai stages doc improvements during workflows; **(4) the arbiter procedure** — Claude Code applies proposals, redeploys, and the next claude.ai session reads the updated doc. **The principle is why CONTEXT.md exists, why it's a Resource, why every taxonomy has both a markdown source and a registry mirror, and why the arbiter procedure was built.**
_Avoid_: "single source of truth" (Latent / the repo IS the SOT — the principle is about KEEPING THE OTHER COMPONENTS aligned with it, not establishing what's authoritative), "consistency rule" (under-specified), "vocabulary sync" (too narrow — it's vocabulary + taxonomy + rules), bare "shared language" (close but doesn't carry the operational sync mechanisms), "drift prevention" (close — drift is the failure mode the principle prevents, not the principle itself).

**Tool**:
A named atomic action surfaced by the Latent MCP server — either a **write** (push_brew, push_roast, patch_*, propose_doc_changes, resolve_queue_entry, push_roast_profile) or a **read** whose results need to be model-callable from claude.ai (read_canonical, read_doc, get_brew, list_recent_brews, list_taxonomy_queue, get_bean_pipeline, pull_roest_log). Each Tool carries an input schema (validated at call time via `zod`), a description string used by claude.ai's `tool_search` for ranking + discoverability, and a typed return shape. Tools are the **primary discoverability surface** for claude.ai — its MCP client lists Tools as actions the model can invoke; Resources are catalog/context but not always model-surfaced. **34 Tools live as of 2026-05-13** (post Sub Pages 6.1).
_Avoid_: bare "action", "MCP function", "API endpoint" (Tools are HTTP-transported but not REST endpoints — they're JSON-RPC over MCP), "command", "method", "RPC".

**Resource**:
A **URI-addressable read-only artifact** surfaced by the Latent MCP server — content endpoints any MCP client can list and fetch. URIs use scheme-prefix conventions: `docs://` (BREWING.md, ROASTING.md, per-roaster cards, WBC reference + recipes, per-taxonomy markdown, operational prompts, CONTEXT.md) and `canonicals://{axis}` (one JSON registry per taxonomy axis). Resources are **catalog + context**, not actions; client behavior varies — claude.ai's MCP client surfaces them less reliably than Tools, which is why Tool-shaped reads (`read_doc` / `read_canonical`) exist alongside the Resource equivalents (the **dual-surface pattern**). Resources stay registered for catalog/discovery and for clients that DO surface them.

Retired Resources (footnote for traceability): `brews://recent` + `brews://by-id/{uuid}` were live Resources in Sprint 2.3 but were retired in Sprint 3.0.5 (2026-05-03) when their Tool equivalents (`list_recent_brews` + `get_brew`) absorbed all use — surfaced as the canonical example of the Resource → Tool promotion path.
_Avoid_: "asset", "blob", "static file" (Resources are served dynamically — source may be a repo file, a DB query, or a registry export), "read-only API" (under-specified), "endpoint", "document" (Resources include canonicals:// JSON, not just docs).

**Dual-surface pattern**:
The deliberate architectural decision to expose the same payload via **both a Resource AND a Tool** when the payload is something claude.ai's model needs to call on-demand. Discovered in Sprint 2.4 dog-food (2026-04-29): claude.ai's MCP client treats Tools as the on-demand model surface but Resources as catalog-only — without Tool-shaped reads, `propose_doc_changes` citations couldn't be drafted against verbatim live-doc text and `push_brew` callers had to read full taxonomy markdown (133 anchors for producers!) to validate one canonical name. Concrete instances: `read_doc` / `read_doc_section` / `list_docs` / `list_doc_sections` Tools alongside `docs://*` Resources; `read_canonical` / `list_canonicals` Tools alongside `canonicals://{axis}` Resources. Brews followed the same path — `brews://recent` + `brews://by-id/{uuid}` Resources were **promoted to Tools** (`list_recent_brews` + `get_brew`) in Sprint 3.0.5 (2026-05-03) and the Resource side was retired. The pattern is the operational form of an architectural rule: **claude.ai surfaces Tools to the model; Resources are catalog only**. **Self-reflective sub-case** (Chris's framing, 2026-05-15): `propose_doc_changes` is the **inverse** of dual-surface — it's a Tool that targets a Resource for write-back, rather than a Tool that mirrors a Resource's read shape. The same architectural rule produces both — model-callable surfaces must be Tools, regardless of whether they read or write Resource content.
_Avoid_: "Tool/Resource parity", "mirror surface", "promoted Resource" (promotion is a path that produces a dual-surface — not the pattern itself), "Tool-shaped reads" (the sub-shape, not the pattern), "Resource shadow".

**Redirect stub**:
The shape of a master-doc after its operational content migrated to sub-skill clusters per [ADR-0011](docs/adr/0011-composable-sub-skills-architecture.md). Post-Wave-4-PR-4b (2026-05-21) [BREWING.md](BREWING.md) is ~3KB (was ~213KB) and [ROASTING.md](ROASTING.md) is ~6KB (was ~147KB); both retain only an `# h1` + a pointer table mapping each former section to its new cluster location. Same shape applies to `docs/taxonomies/sworks.md` after Wave 1 migration (3-line stub pointing at `brewing-equipment-expert/cluster/sworks.md`). The stub preserves file-path back-compat for any external link / cached claude.ai memory / prompt reference that points at the old path (the file still resolves; deep `#section` anchors do not survive — the pointer table is the way back in); new content lands at the cluster path, the stub never re-bloats. Permanent state, not transitional — every future master-doc → cluster migration produces another redirect stub.
_Avoid_: "deprecated master doc" (overstates — anchors still resolve), "redirect file" (technical-sounding; stub is the lighter word), "pointer doc" (under-specifies the back-compat function).

**Cluster path**:
The URI shape used to address a sub-skill cluster file: `docs/skills/<sub-skill>/cluster/<file>.md`. Used in two places: (a) the `docs://` MCP Resource catalog — e.g. `docs://skills/brewing-assistant/cluster/operational-guide.md`; (b) the `target_doc:` field on `propose_doc_changes` proposals — post-PR-4b ARBITER.md allow-list accepts `skills/<sub-skill>/cluster/<file>.md` shape via `isKnownDoc()`. The cluster directory under each sub-skill holds the operational/reference content; the `SKILL.md` at the sub-skill root holds the prompt-shaped instructions. Naming convention at the leaf file is tier-dependent — see **Operator-guide / operational-guide / onboarding-protocol / operational-reference**.
_Avoid_: "sub-skill path" (ambiguous — could mean the directory or the file), "skills path" (under-specifies the cluster sub-structure), "deep doc" (descriptive but uncanonical).

**Operator-guide / operational-guide / onboarding-protocol / operational-reference**:
Four cluster-leaf file-name conventions, each scoped to a tier + a domain. The four-way distinction matters because the names look similar but address structurally different content; conflating them confuses claude.ai's fetch paths.
- **operator-guide** — Coordinator tier, ONE file at `docs/skills/coordinator/operator-guide.md`. Cross-domain operator how-to: canonical lookups + MCP server how-to + per-coffee threads + Session Debrief template. The single file every workflow sub-skill (brewing or roasting) returns to for "how do I work with the MCP server."
- **operational-guide** — Workflow sub-skill (brewing), `brewing-assistant/cluster/operational-guide.md`. Domain-specific workflow steps (brewing Steps 1-4) — the whole brewing process fits in one document because brewing is sequential and short-cycle (Chris's framing: "the brewing master needs less state in its head; mostly a pointer to references with one iteration loop").
- **onboarding-protocol** — Workflow sub-skill (roasting), `roasting-assistant/cluster/onboarding-protocol.md`. Domain-specific FIRST-STAGE protocol only — green-bean intake → first experiment set → first 3 roasts. **NOT the full roasting workflow** (which distributes across `log-roast.md` + `log-cupping.md` + `close-lot.md` prompts + roest-knowledge + roasting-historian sub-skills). The naming asymmetry vs brewing's operational-guide reflects a real structural difference: roasting is operational-centric, state-heavy, multi-sub-skill, long-cycle (V-sets, iteration between roast and cupping until reference), so no single doc consolidates the whole process. Chris's framing: "the roasting master needs to keep stage state in its head."
- **operational-reference** — Knowledge sub-skill (brewing-equipment-expert), `brewing-equipment-expert/cluster/operational-reference.md`. Equipment lookup reference: dripper specs + filter flow rates + grinder particle distribution + brewer rotation history. Workflow sub-skills CONSULT Knowledge sub-skills (brewing-assistant ↔ brewing-equipment-expert) to surface lived inventory + usage-history context when constructing recipes.

Chris's mental-model umbrella: **"the roasting process" / "the brewing process"** — each has multiple stages (intake / first-roast-design / iteration / close on roasting; coffee-brief / recipe / iteration / archive on brewing). Onboarding-protocol is stage 1 of the roasting process. **Open rename question (OBSERVING in grilling queue):** should the workflow sub-skill files be renamed to `roasting-process.md` / `brewing-process.md` for cross-domain consistency, with onboarding-protocol becoming a stage section? Deferred pending more lived practice with the multi-stage roasting workflow. **Promotion trigger** (Item 2a, Sprint R Phase 4 Step 4 Group 5 grill, 2026-05-24, audio-ratified): **≥2 additional roasting-stage cluster docs** land in the `roasting-assistant/cluster/` directory (candidates: cupping-protocol.md / iteration-protocol.md / close-protocol.md). Single-stage doc today gives the umbrella nothing to umbrella over; second additional stage doc is the structural threshold where the umbrella becomes load-bearing. Count-based mechanical trigger is observable in substrate; rejected lived-speech-based trigger as harder to verify across sessions.
_Avoid_: confusing operator-guide (Coordinator, cross-domain) with operational-guide (brewing, domain-specific) — they differ by one letter; treating onboarding-protocol as the full roasting protocol (it's stage 1 only); treating operational-reference as a workflow doc (it's a Knowledge-tier lookup that workflow sub-skills consult).

**Sub-skill tiers + Master Coordinator + Chris's vocabulary aliases**:
The Latent reference architecture (full enumeration in [ADR-0011](docs/adr/0011-composable-sub-skills-architecture.md), amended 2026-05-27 per [ADR-0017](docs/adr/0017-research-assistant-architecture.md)): **17 sub-skills across 3 tiers** + a **Master Coordinator** router outside the tiers.
- **Knowledge tier (8 active)** — authors + maintains reference docs and their associated knowledge clusters. Roasting Historian / Brewing Historian / WBC Roasting Archivist / WBC Brewing Archivist / Brewing Equipment Expert / Peer-Learning Roasting Archivist / Roest Knowledge / Research Coordinator (the last with Planning collapsed in per ADR-0017 Exception 2; cross-domain; operator-direct per Exception 3).
- **Workflow tier (9 sub-skills, plan-vs-execute split)** — 3 planners (Roasting Assistant / Brewing Assistant / Sourcing Workflow Planner) + 6 executors (Roast Recorder / Brew Recorder / Cupping Specialist / Roest API Worker / Close-Lot Specialist / Research Assistant). Plan-then-execute is the Chris-named 2nd axis ("planning step first, then execution — similar to how we do things in claude code"); brewing collapses to 1 plan / 1 execute (sequential), roasting iterates plan/execute multiple times across V-sets + cuppings (state-heavy). Research Assistant is execution-only (no separate planner — Research Coordinator does both per ADR-0017 Exception 2) and operates Claude-Code-centric (no MCP Tools per ADR-0017 Exception 1).
- **Cross-Coffee Insight Layer (CCIL)** — single tier, single sub-skill, cross-domain synthesis above the two domain Historians. **Temporal constraint** (Chris-locked Item 3, 2026-05-23): fires only AFTER complete cycles (lot close on roasting, brew archive on brewing), never during in-flight workflows. Compounds learnings from roasting + brewing + selectively external sources. See **Cross-coffee insight layer** entry for the rest.
- **Master Coordinator** — router that lives outside the tiers, plus the cross-zone meta-synthesizer at dispatch/coordination level. Catalog at [docs/skills/coordinator/](docs/skills/coordinator/). Chris's mental model: "the upper meta layer that knows about all processes, master pointer list." Analog to CCIL but at the coordination layer rather than the post-cycle synthesis layer — Coordinator dispatches in-flight; CCIL synthesizes after the fact.

"Roasting expert" = Roasting Assistant + roasting-zone executors collectively; "brewing expert" = Brewing Assistant + Brew Recorder; "equipment expert" = Brewing Equipment Expert (clean match); "cupping expert" = Cupping Specialist; "Roest expert" = Roest API Worker + Roest Knowledge collapsed. The alias entry here is the bridge; substrate stays formal.

**Machine-layer naming vs. human-operator legibility** (Item 16b grill, 2026-05-24): the general principle behind why the lived-speech ↔ substrate-name aliasing works the way it does. Substrate naming (sub-skill names, Tool descriptions, Resource URIs, cluster file paths, `propose_doc_changes` target_doc identifiers, schema column comments) optimizes for **inter-machine coherence** — Claude Code + claude.ai + MCP server + Latent app routing all need to land cleanly on the same vocabulary. **Human-operator legibility is a non-goal** at this layer; Chris doesn't reference substrate names in his lived workflow speech (he gives claude.ai the prompt and lets the system route). The alias entries above (Expert / Zone / "the roasting process" / Cross learnings) are the bridge for the rare cases Chris reads or speaks substrate by name. The principle generalizes the [ARBITER.md § Tool description writing convention](ARBITER.md) "Opening principle" (Tool descriptions are claude.ai-facing, not human-readable docs — Group 4 / PR [#237](https://github.com/chrismccann-dev/latent-coffee/pull/237)) from Tool descriptions to substrate naming broadly. Operationally this means: substrate-renaming sprints need a real machine-layer motivator (routing confusion, ambiguity in MCP resolution, friction at session time), not a Chris-side legibility motivator — Chris-side reads route through CONTEXT.md aliases, which can adapt without touching the machine layer.

_Avoid_: collapsing tiers with Master Coordinator (Coordinator is OUTSIDE the tiers, not a fourth tier); treating CCIL as an active-workflow surface (post-cycle only); using "Expert" or "Zone" in substrate code / SKILL.md / MCP catalog (lived aliases only — substrate names remain canonical); assuming substrate names need to be Chris-readable (they don't — CONTEXT.md aliases are the bridge); proposing substrate renames motivated by Chris-side ergonomics alone (real machine-layer motivator required).

**Per-lot directory taxonomy** (roasting zone):
Three directories under [docs/skills/roasting-historian/cluster/](docs/skills/roasting-historian/cluster/) hold per-lot artifacts, mapped to lifecycle state from [lib/lifecycle-state.ts](lib/lifecycle-state.ts):
- **`active-lots/<lot>.md`** — V-set lots currently iterating (`waiting_for_next_roast` / `waiting_for_next_cupping`). Holds the working hypothesis: interim learnings + per-V-set cupping notes + next-experiment design context. Updated at each V-set close-out via `log-cupping.md` STAGE 3 and at lot reframing. Interim learnings are real (they shape the next V-set design) but transient — they crystallize into the resolved learnings at lot close.
- **`one-shot-calibrations/<lot>.md`** — one-shot lots in process (`green_beans.is_one_shot=true`, single-batch, no V-set iteration). Different shape from active-lots because there's no "next session" iteration loop; the file captures the single-point question + calibration parameters + status. Up-front planning matters more here because tolerance is lower (see **Three big levers** for the lever-mix rationale).
- **`learnings/<lot>.md`** — closed lots (`resolved`), V-set OR one-shot alike. Holds the crystallized post-close synthesis: reference roast + winning cup + roast_learnings carry-forward content + per-roast reflections. Merges all interim active learnings into the final lot-level record.

**Close-out convention** (Item 4 grill, 2026-05-23 — revises README's prior "delete" rule): when a lot resolves via `close-lot.md` (V-set) or `one-shot-closeout.md` (one-shot), the originating file (`active-lots/<lot>.md` or `one-shot-calibrations/<lot>.md`) becomes a **redirect stub** pointing at `learnings/<lot>.md` rather than being deleted. The stub carries: title + Status + close date + one-line reference-roast summary + redirect link (~6 lines total). Maintains the lifecycle audit trail (which lots closed recently visible in the dir listing) while keeping bloat per-stub minimal. Lived precedents: `active-lots/cgle-srume-natural-2026.md` + `active-lots/cos-hig-bor-2026.md`, both closed 2026-05-23.

Brewing zone does not use this taxonomy — `docs/skills/brewing-historian/cluster/` has only `patterns/` (per-cultivar / per-strategy / per-coffee-family + cross-coffee-insights), because brewing has no V-set iteration concept; each brew is a single-iteration-then-archive workflow that lands directly in the relevant patterns/ sub-cluster.

**Slug convention** (Item 11 grill, 2026-05-23): all 3 directories use the **canonical lot code** as the file slug (lowercase-kebab of the lot code from Chris's inventory sheets — e.g. `CGLE-SRUME-WASHED-2026` → `cgle-srume-washed-2026.md`; `GV-OMA-25-035` → `gv-oma-25-035.md`; `ECU-TD24-RANCHOTIO-TM-WASHED` → `ecu-td24-ranchotio-tm-washed.md`). Same slug across active-lots/ ↔ learnings/ ↔ one-shot-calibrations/ means the redirect-stub at close just keeps the filename — no rename. **Three big levers**:
The three primary levers that shape how a coffee tastes in the cup (Chris-locked Item 4 grill, 2026-05-23), in descending order of operational reach for Chris:
1. **Production lever** (biggest) — how the coffee is grown + processed at origin: cultivar selection, growing conditions, harvest timing, fermentation, drying, milling. **Chris can't touch this** — Latent's domain starts after the bag arrives. The lever IS the biggest but Chris doesn't pull it.
2. **Roasting lever** (medium) — Chris's curve choices on the Roest L200: peak inlet, fan curve, drop temp, charge / hopper / end-condition. Has the most ITERATIVE reach because V-sets allow 3+ comparative batches per lot; the roasting workflow is the most state-heavy precisely because the lever payoff is high.
3. **Brewing lever** (smaller) — the final-polish lever: brewer + filter + grinder + grind + temp + ratio + pour structure + modifiers + extraction strategy. Smaller reach because it operates downstream of a fixed roasted bean — but real reach within its band.

**One-shot lots compensate via the brewing lever** — when the roasting lever is single-shot (`green_beans.is_one_shot=true`), Chris must pull the brewing lever harder to shape the end cup; up-front planning matters more because tolerance on the single-shot is lower. The roasting / brewing lever asymmetry on one-shot lots is the design rationale for keeping `one-shot-calibrations/` separate from `active-lots/` in the per-lot directory taxonomy — different lever mix → different working-hypothesis shape.
_Avoid_: "production / roast / brew = three equal levers" (lever sizes are NOT equal); "Chris controls all three levers" (production is locked at the producer); "one-shot lots are just V-sets with N=1" (the lever-mix shift is the structural distinction, not just iteration count); "the brewing lever is small so it doesn't matter" (within its band, brewing reach is real — and on one-shot lots it carries compensating load).

**Formalization tax**:
The governing principle behind the anti-bloat half of the self-improvement architecture ([ADR-0022](docs/adr/0022-formalization-tax-and-self-improvement-counterbalance.md), named 2026-06-05). Every formalization — a rule, a doc section, a registry row, a skill instruction — carries a **standing cost** (it's loaded into context every time its surface is served; on the claude.ai surface, context is the binding constraint) plus **fluke / second-order risk** if minted from too few observations. So: **formalize as late as you defensibly can**; a rule earns its place only once enough lived edge cases have been seen that it's shaped right. Chris's framing (2026-06-05): like product design / bureaucracy / law — "rules are good, but every one instills a tax on the system, so it has to deserve to be there." Enforced by a **healthy forcing function** — claude.ai's limited context window, the same constraint-as-forcing-function principle as "mobile as forcing function" ([ADR-0018](docs/adr/0018-per-surface-mobile-pattern.md)): constrained resources produce the first-principled right design. The tax is paid down by two mechanisms: **graduation threshold** (delay) + **arbiter-shaped clawback** (cleanup).
_Avoid_: "context bloat" (that's the *failure mode* the tax framing prevents — and "Category bloat" / "State bloat" are already separate failure-mode entries; the tax is the *principle*, not the symptom), "minimalism" (under-specified — the tax permits formalization, it just makes it earn its place), "less is more" (loses the standing-context-cost mechanism), "KISS".

**Graduation threshold (N=3)**:
The delay mechanism of the formalization tax ([ADR-0022](docs/adr/0022-formalization-tax-and-self-improvement-counterbalance.md)). An observation may be **acted on** at N=1 but may not graduate into a **reusable abstraction** (a rule / CONTEXT term / canonical registry entry / main-doc or main-skill section) until it has recurred **three times** — "once is a fluke, two is a pattern forming, three is graduate it." Three sightings reveal the rule **plus its edge cases**, so the formalization is shaped right and doesn't spawn unconsidered second-order effects. N=3 is the universal dial across every surface (case-study generalization, CCIL n≥3 capsule graduation, canonical-registry promotion, grilling-queue accumulate-then-grill, skeleton/observing lists); only the **trigger unit** (what you count) and **graduation target** (where it lands) vary. **Two-tier**: N=1 is for *acting* on a cost-grows-if-you-wait thing (an obvious bug; an edge case around an *existing* rule; a structural doc fix — missing headings / not section-readable), never for *minting a new abstraction*. **Provisional, not terminal**: a graduated rule is itself taxed — keep collecting examples 4-8, and edit / prune / delete the rule if later evidence diverges.
_Avoid_: "rule of three" (an AI-writing cliché + means something else in prose; this is a recurrence count, not a triadic structure), "N=3 rule" (close, but it's a *threshold* with a two-tier act-vs-abstract split, not a flat rule), "three-strikes", "promote-on-recurrence" (loses the provisional-not-terminal property).

**Arbiter-shaped mechanism** (the clawback meta-pattern):
The **prescribed shape every clawback / counterbalance mechanism MUST follow** ([ADR-0022](docs/adr/0022-formalization-tax-and-self-improvement-counterbalance.md)): **(1) a cheap mechanical trigger** that only *surfaces* candidates (`check:doc-sizes`, `check:hotspots`, recurrence counts, a tripwire) + **(2) multi-perspective arbitration** (the judgment, debated between actors from different principles — not an operator-as-higher-power gate) + **(3) stops-at-report** (never auto-commits the irreversible — it hands over packets). Rationale is pure formalization-tax: an auto-acting mechanism is *itself* a heavy formalization with big second-order risk, so the mechanism keeps its own footprint cheap by splitting. Not new with Cluster B — ARBITER, the skeleton review, and the observing-list walk were already this shape; **doc-pruning** + **architecture-review** made it explicit. Autonomy is gated on **intent, not execution**: principal direction + language agreed together, then execution autonomy scales with technical-vs-subjective; narrow low-risk mechanical classes (provenance shrapnel, dead anchors, over-cap detection) may graduate up the [ADR-0013](docs/adr/0013-self-improvement-primitives.md) autonomy ladder, the abstraction-level judgment stays multi-perspective.
_Avoid_: "operator-led" (Chris explicitly rejected the higher-order/lower-order framing — it's multi-perspective debate, the *combination* produces the result), "human-in-the-loop" (generic + implies the human is a gate, not a perspective), "surface-and-judge" / "trigger-and-steer" (coined synonyms — reuse the existing "arbiter" vocabulary instead; minting a new word is itself a formalization tax), "review step".

**Prototype-as-input-not-canon**:
The recurring practice that runs the multi-perspective arbitration ([ADR-0022](docs/adr/0022-formalization-tax-and-self-improvement-counterbalance.md)). When Chris hands over a prototype (a refactored doc, a design system, a CCIL reorganization), the standing instruction is: **"this is my interpretation — do NOT copy it; treat Latent as the source of truth, apply your own judgment, and pull what's useful from the prototype into what we already have."** The prototype is an *input*, never *canon*; the result is the operator's prototype + Claude Code's independent judgment + the lived examples, argued against the existing system as the canonical baseline. (Done 3+ times — the design-system reconciliation, the PRODUCT.md refactor, the CCIL re-architecture — so it clears its own graduation threshold.)
_Avoid_: "prototype-first" (loses the load-bearing *not-canon* clause — the whole point is it's NOT to be copied), "mockup", "reference implementation" (implies it IS the target shape), "draft" (a draft becomes the thing; a prototype-as-input is pulled-from, not finished).

**Pruning shapes** (the six clawback ops):
The vocabulary of what a doc-pruning / clawback exercise actually *does*, one of six named ops (doc-pruning arc, cases 001-007; [ADR-0013 Amendment](docs/adr/0013-self-improvement-primitives.md)): **extract** (move a block verbatim to a sub-doc + leave a pointer), **split** (divide one doc into multiple zone/topic/ownership-axis siblings), **consolidate** (merge sections into something more concise — synthesis, not move), **archive** (cease re-stating content that already has a historical home — its tell), **delete** (true removal of content existing nowhere else — flag-only except the provenance/dead-anchor low-risk class), **re-home / re-scope** (tighten a doc's *job definition* and route now-out-of-scope content to its canonical home behind routers — case 007). Doc-shape predicts the shape: reference-doc → extract/split; living-glossary → consolidate/delete; roadmap/log → archive; registry-catalog → ownership-axis split; procedure prompt → consolidate-to-pointer; compounding Historian at single-doc cap → re-home.
_Avoid_: "pruning" as a catch-all (it's six distinct ops with different safety profiles — delete is irreversible, archive/extract are moves), conflating **archive** with **extract** (archive = stop duplicating into the live surface; extract = move to an on-demand reference the live surface still points at), conflating **re-home** with **split** (re-home re-scopes the doc's job; split makes parallel siblings of the same content).

**Asymmetric write trust**:
The architectural decision that claude.ai's MCP writes are trusted differently depending on **what's being written**. Two trust tiers + one variant:
- **Direct write** (high trust, no intermediate review) — structured field-picking against canonical registries: brews, green_beans, roasts, roast_recipes, cuppings, experiments, roast_learnings, inventory rows, Roest profiles. Validated at push time against the canonical taxonomy (cultivar / terroir / process / roaster / brewer / filter / grinder / flavor / signature method); FK-enforced where applicable; rejected outright on canonical resolution failure. claude.ai is reliable here because every field has a known set of legal values and the validation pipeline catches everything else.
- **Propose-then-apply** (lower trust, Claude Code arbitrates) — free-form prose mutations to living docs (BREWING.md, ROASTING.md, `docs/brewing/roasters.md` per-roaster cards, `docs/taxonomies/*.md` source files). claude.ai authors **proposals with verbatim citations**; proposals land in `doc_proposals` (status='pending'); Claude Code walks the queue via the **arbiter procedure** and applies citation-by-citation. Multi-write contention is real — Chris often touches one coffee through 3-4 workflows the same day and may propose edits to the same doc section from different angles; the arbiter handles batching + supersede logic.
- **Canonical-promotion writes** (variant — same propose-then-apply shape, different substrate) — when claude.ai encounters a value not in the canonical taxonomy (a new producer, a new roaster, a new region, a new proprietary process, a new cultivar — override is broad, not axis-specific), the path is: direct-write the entity with an `*_override:true` flag (override-flag axes: producer / roaster / brewer / filter / grinder), OR fail fast and call `propose_canonical_addition` (strict axes: cultivar / terroir). Either way the net-new value lands in `taxonomy_overrides_queue` and Claude Code arbitrates registry promotion via the Phase 3 arbiter playbook (same procedure file, second queue type).

Why asymmetric: claude.ai's strong suit (validated across V2 dog-food) is canonical-registry field-picking — every field has a bounded set of legal values; the validation pipeline catches everything else; direct-write is safe. Free-form prose mutation is where **drift** and **bloat** happen (separate entries), so Chris explicitly chose Claude Code as the arbiter for that path. **The asymmetry is the load-bearing call of SYNC_V2 (Decision #5)** — it's what lets the system be both fully MCP-only-input AND prevent the doc + registry layers from degrading.

**Promotion-discipline rule** (Chris, 2026-05-15): when a canonical-promotion write does result in a registry addition (e.g. a new producer survives arbitration), the **full canonical entry must be populated at promotion time** — not just the bare-minimum required fields. The promotion is the only moment the entry's context is fresh; deferred fill-in is forgotten fill-in. Producer entries get 27 fields; roaster entries get 29; the discipline applies across all rich-registry axes.

**Saturation projection**: the volume of canonical-promotion writes is expected to **decay over time** as Chris's coffee exposure saturates the registries — at some point ~99% of new coffees encountered are already canonical, and promotion becomes rare. The registries are converging assets, not endlessly-growing ones.
_Avoid_: "two-tier writes" (loses the trust-is-the-variable framing), "trust split" (close but doesn't carry *what determines the tier*), "structured-vs-prose" (under-specified — canonical-promotion is a third shape that isn't strictly prose), "MCP write modes".

**Drift**:
The slow-motion failure mode where the same canonical thing gets referenced under slightly-different names across writes — "Volcan Baru Highlands" vs "Volcán Barú Highlands" (terroir), "Geisha" vs "Gesha" (cultivar), "Diego Bermudez" vs "Letty Bermudez + Finca El Paraiso" (producer drift across sources). Each reference is plausible in isolation; in aggregate they fragment what should be one canonical bucket and dilute the per-axis aggregation pages. **Real-world fuzz is the source** — coffee is an agricultural product, regions don't have clean boundary lines, cultivar identifications aren't always genetically confirmed — so taxonomies have to do active work to project messy reality onto a clean canonical surface. Chris experienced drift extensively in the pre-Latent spreadsheet era even working alone; it compounded when his EA Belle (no coffee context) joined the input loop, requiring Chris to act as arbiter on every batch. The MCP-only input principle + canonical-validation pipeline + alias-resolution maps are the architectural prevention of drift at the data layer; the **propose-then-apply tier** is the prevention of *prose drift* (same failure mode at the doc layer).
_Avoid_: "canonical drift" (drift IS the unit; adding "canonical" implies there's a non-canonical kind), "naming inconsistency" (under-specified), "taxonomy fragmentation" (this is the consequence of drift, not drift itself), "string drift".

**Category bloat**:
The opposite failure mode from drift — the canonical registry growing too many entries that should have been **aliased** or **rejected**, fragmenting what should be a unified canonical bucket. Without an arbiter step every `propose_canonical_addition` or `*_override:true` write would land verbatim in the registry; over enough writes the registry would become indistinguishable from the raw input string set, defeating the canonical aggregation purpose. The propose-then-apply variant for canonical-promotion writes (`taxonomy_overrides_queue`) exists precisely so Claude Code can decide **promote / alias / reject / duplicate** per queued entry. Chris's framing (2026-05-15): "you don't want to split categories too much" — there's an inherent tension between **fine-grained enough to be specific** and **coarse-grained enough to aggregate over**, and the arbiter is the human-in-the-loop that calibrates the boundary per axis. Terroir aggregates at the macro level (not country, not farm); cultivar aggregates at the lineage/family level (not species, not specific origin); the right scope is axis-specific and was deliberately chosen.
_Avoid_: "registry bloat" (close but the registry is the substrate, not the failure mode), "canonical fragmentation" (this is what bloat produces, not the bloat itself), "taxonomy inflation".

**State bloat**:
The doc-layer analog of category bloat — living docs (BREWING.md, ROASTING.md, per-roaster cards, taxonomy markdown) accumulating un-curated edits that fragment what should be coherent narrative sections. Without the arbiter every `propose_doc_changes` write would append-or-replace verbatim; over enough writes the docs would degrade into a sprawl of redundant bullets, contradictory framings, and stale references. The arbiter procedure looks at proposals in batch (1 proposal can have N citations across N sections; M proposals can touch the same section across a multi-day write window) and decides per-citation **apply / reject / edit**, plus enforces house style (hyphens-not-em-dashes per `feedback_hyphens_not_emdashes.md`) at write time. Chris's framing: "somebody has to be the one looking through the docs to make sure they stay clean" — without active curation the docs degrade as input volume grows.
_Avoid_: "doc bloat" (close but state is the better generalization — covers per-roaster cards + taxonomy markdown + CONTEXT.md), "narrative drift" (close but state bloat is the un-curated-accumulation problem, narrative drift is a sub-case), "edit sprawl".

**Arbiter**:
The role Claude Code plays when keeping Latent's docs + canonical registries clean against accumulating proposals — one of the two operational modes of the Claude Code half of Role separation (the other being builder). The arbiter is a **batch curator**: it doesn't run real-time after each proposal; it runs when Chris triggers it ("process pending arbitration" or aliases — see separate entry), reads everything pending, and walks the queues citation-by-citation. The arbiter's primary value-proposition is **preventing drift / category bloat / state bloat** (separate entries) — without it, every claude.ai proposal would land verbatim and the canonical layer would degrade. The arbiter is also the **multi-write contention resolver**: when Chris touches one coffee through 3-4 workflows the same day and overlapping proposals land, the arbiter batches them and decides what supersedes what.
_Avoid_: "reviewer" (under-specified — arbiter is specifically batched + queue-walking, not real-time), "curator" (close but curator implies pre-existing content; arbiter is queue-walking), "merger" (too narrow — most of the work is judgment, not merging), bare "Claude Code" (Claude Code has two operational modes; pinning which one matters).

**Arbiter procedure**:
The step-by-step playbook a Claude Code session follows when running as arbiter — codified in `ARBITER.md` at the repo root. Today **two parallel queue types**, one procedure: **prose proposals** (`doc_proposals`, migration 037) get an apply / reject / edit / orphaned-anchor disposition per citation; **canonical promotions** (`taxonomy_overrides_queue`, migration 045) get a promote / alias / reject / duplicate disposition per queue row. Both queue types are walked in the same session; if one is empty the procedure skips it and continues. The queue types are **extensible by design** — future curation surfaces (null-field reviews, missing-entity flags, other drift signals) can attach to the same trigger phrase without changing the procedure shape. Key procedural elements: read pending → group (by `target_doc` for prose, by `axis` for canonical) → present plan to Chris → Chris does a quick pass-through ("yep go for it" the typical response, ~99% of cases) → apply decisions → flip queue row status → commit / PR / merge → Vercel redeploys → next claude.ai session reads fresh state. **House-style normalization happens at apply time** (em-dashes / en-dashes → plain hyphens, per `feedback_hyphens_not_emdashes.md`). **Auto-supersede semantics**: when a newer prose proposal targets the same `(target_doc, section_anchor)` as an older pending one, the older row is auto-marked superseded by the `propose_doc_changes` Tool handler; the arbiter only sees the newer.
_Avoid_: "arbitration playbook" (close, but procedure is the canonical noun), "doc proposal workflow" (under-specified — covers both queue types and future extensions), "ARBITER.md" (that's the file; the procedure is the abstract concept the file documents), "arbiter flow".

**Process pending arbitration**:
The canonical **trigger phrase** that initiates an arbiter procedure run, plus its functional aliases ("process pending doc proposals", "arbitrate", "apply pending docs", "review the proposal queue", "review canonical proposals", "process pending taxonomy queue", etc.). All aliases route to the same playbook (ARBITER.md). The trigger is intentionally **Chris-initiated** — not auto-run on a schedule and not real-time per proposal — because batched curation is the right cadence: multi-day write windows produce contention the arbiter needs to see all at once. The phrase is documented in `ARBITER.md` § Trigger phrase and is a fixed protocol element of the system; the trigger surface is open to extension as new queue types are added.
_Avoid_: bare "process pending" (under-specified), "kick off the arbiter" (informal), "run the arbiter" (close but doesn't match the protocol phrasing), "arbitration command".

**`propose_doc_changes`**:
The Tool claude.ai calls when its current workflow surfaces a doc-improvement opportunity worth carrying forward — a new flavor pattern, a roaster-specific brewing lesson, a process-axis nuance, a taxonomy refinement, a cross-coffee insight. The propose-then-apply tier of asymmetric write trust (separate entry) — claude.ai writes the proposal as **N citations against specific section anchors** with verbatim `current_text` + `proposed_text` + `rationale` per citation; the proposal lands in `doc_proposals` with status='pending'. **Polymorphic Tool** — one schema covers all four target families (BREWING.md, ROASTING.md, per-roaster cards in `docs/brewing/roasters.md`, per-axis taxonomy markdown in `docs/taxonomies/*.md`). Citations can target **different files within a single proposal** (per-citation `target_doc` overrides the proposal-level default) because one brew debrief often touches a roaster card + a BREWING.md archive bullet + an Open Questions section simultaneously. **Auto-supersede semantics at insert time**: a newer proposal targeting the same `(target_doc, section_anchor)` as an older pending row marks the older row `superseded` automatically — the arbiter only ever sees the newest version.

**Proposal-quality discipline** (Chris, 2026-05-15): the Tool is **silent-and-optional** — claude.ai proposes only when a workflow has surfaced **net-new system improvement**, never speculatively. If a brewing completion or roasting close-out produced nothing the living docs don't already capture, **no proposal is the right answer**; writing a proposal for the sake of writing one is state-bloat at the source. The discipline is enforced at the proposer (claude.ai), not at the arbiter — by the time something lands in `doc_proposals` it should already pass the system-improvement bar.
_Avoid_: bare "propose doc changes" (informal — the literal Tool name with backticks is canonical), "doc-change Tool" (under-specified), "PR proposal" (these aren't PRs — they're queue rows the arbiter ultimately turns into PRs), "edit suggestion" (under-specifies the system-improvement bar).

**`doc_proposals` queue**:
The Postgres table (migration 037, 2026-04-29) holding pending prose-doc-change proposals between the moment claude.ai writes via `propose_doc_changes` and the moment Claude Code applies via the arbiter procedure. **Status lifecycle**: `pending` → `applied` / `rejected` / `superseded`. One row per proposal; each row carries an array of citations + a summary + a source (`source.kind` + `source.id`, e.g. `brew:uuid`). RLS-scoped to Chris's user_id. The substrate of the propose-then-apply tier — the table IS the queue. Status flips happen at the arbiter (Claude Code runs `UPDATE doc_proposals SET status = ... WHERE id = ...` per resolved proposal during the arbiter procedure). **Multi-day write windows are normal**: Chris may not run "process pending arbitration" for days after proposals land; the queue tolerates batching by design.
_Avoid_: "doc_proposals table" (technically correct but under-specifies its role — it IS a queue, with status semantics), "proposal queue" (under-specified — there are at least two queue types, this is the prose one), "BREWING.md queue" (wrong — covers all 4 target families).

**`taxonomy_overrides_queue`**:
The Postgres table (migration 045, Phase 3, 2026-05-05) holding pending canonical-promotion proposals across **8 taxonomy axes** (Sprint 12 / MCP-1, migration 063, 2026-05-21 widened from 7): `producer`, `roaster`, `brewer`, `filter`, `grinder`, `terroir`, `cultivar`, `signature_method`. The substrate of the canonical-promotion variant of asymmetric write trust — structural parallel to `doc_proposals` (same arbiter procedure walks both). One row per net-new canonical proposal; carries `axis`, `raw_value` (verbatim), `submission_path` (`override_flag` for override-eligible axes or `manual_propose` for strict axes), `source_kind` + `source_id` (the brew / green_bean / roast that triggered it), `evidence` jsonb. **Status lifecycle**: `pending` → `promoted` / `aliased` / `rejected` / `duplicate`. **EXCLUDE constraint on `(user_id, axis, lower(raw_value)) WHERE status='pending'`** auto-collapses duplicate pushes — a second push of the same string returns the existing pending row id; the arbiter sees the value once even if 5 brews triggered it. RLS-scoped. Sprint 12 closed the long-standing signature_method enforcement-tier mismatch — net-new proprietary processes (Alchemy / TIM / Enzyflow / etc.) now flow through `push_brew(signature_method_override: true)` and surface in the queue rather than failing fast.
_Avoid_: "the canonical queue" (under-specified), "override queue" (close but only one of two submission paths uses override flags), "taxonomy queue" (informal short-form — fine in prose; the canonical noun is `taxonomy_overrides_queue`).

**`propose_canonical_addition`**:
The Tool claude.ai calls when introducing a canonical value on a **strict axis** where direct-write-with-override-flag isn't available. Strict axes (cultivar / terroir today) fail fast on canonical resolution failure — the entity write is rejected outright at validation — so claude.ai's recovery path is to call `propose_canonical_addition` explicitly with `axis` + `raw_value` + `evidence`, landing the row in `taxonomy_overrides_queue` with `submission_path='manual_propose'`. claude.ai then re-attempts the entity write once Chris-as-arbiter has promoted the canonical via the arbiter procedure (or supplies an alias resolution). Distinct from the override-flag path (`*_override:true`) which lands the entity AND queues the canonical in one call — strict axes don't get the one-call shortcut because Chris deliberately wants the entity write blocked until the canonical is resolved.
_Avoid_: "propose new canonical" (close but the literal Tool name with backticks is canonical), "canonical proposal" (ambiguous — could mean a queue row), "strict-axis fallback" (informal).

**Override flag**:
The `<axis>_override: true` parameter (`producer_override`, `roaster_override`, `brewer_override`, `filter_override`, `grinder_override`) that lets claude.ai bypass canonical-resolution validation on an **override-eligible axis** when it encounters a net-new value. The entity write succeeds with the raw_value persisted verbatim; the net-new value is auto-queued to `taxonomy_overrides_queue` via `submission_path='override_flag'`. **Override is broad, not narrow** (Chris, 2026-05-15) — every override-flag axis can produce a queued canonical promotion; the flag is the **escape hatch** that keeps direct-write strict for normal writes but tractable for the long tail of new-coffee discovery. The flag exists precisely **because cascading-failure was the failure mode it prevents** — pre-override, claude.ai could not push an entity with a net-new value at all, breaking the workflow mid-write.

**Canonical-strictness spectrum** (Chris, 2026-05-15) — the per-axis policy choice is calibrated to the expected net-new rate for that axis:
- **Strict / static** (cultivar, terroir): Chris built comprehensive registries; net-new is rare (genuinely new cross-bred cultivar; region missing from the macro list). Strict-axis path only.
- **Self-owned** (brewer, filter, grinder): net-new only when Chris buys new equipment. Override-flag path (purchases happen mid-workflow).
- **Closed-objective** (roast level): Standardized canonical set; mostly about alignment, not extension. Not in the queue today.
- **Open / loose** (producer, roaster, signature_method, flavor, process modifiers): High net-new probability — producers and farms are too numerous to fully canonicalize; roasters are too numerous; signatures emerge constantly; flavor vocabulary is subjective. Producers + roasters + signature_method are override-flag axes (signature_method joined Sprint 12 / MCP-1, 2026-05-21); flavor + process modifiers remain outside the queue today.

_Avoid_: "override parameter" (under-specified), "bypass flag" (could mean any validation bypass), "escape hatch" (descriptive but not canonical), bare "override" (ambiguous against general programming meaning).

**find-or-create**:
The canonical-resolution write-path pattern every `push_*` and `patch_*` Tool runs for each canonical-axis field on the entity. Ordered steps for a single field: **(1) alias lookup first** — the `CanonicalLookup` bundle (built by `makeCanonicalLookup(names, aliases?)` in `lib/canonical-registry.ts`) is checked for an alias match (e.g. `"Geisha"` → `"Gesha"`); aliases resolve to a canonical without any further work; **(2) canonical check** — if no alias match, the input is checked against the canonical names directly (case-correcting via `canonicalize`); **(3) FK materialization** — if found canonically AND the axis is FK-resolved (cultivar / terroir), look for the canonical row, create it lazily with full metadata from the registry entry (`getCultivarEntry()` / `getTerroirEntry()`) if absent, FK-link the entity. Records provenance on the entity (`*_provenance` column flips `canonical` vs `auto_created`); `green_beans.canonicals_updated_at` timestamps the moment; **(4) NET-NEW branch** — if neither alias nor canonical resolves AND the axis is override-eligible, persist the raw_value verbatim + queue to `taxonomy_overrides_queue` for arbiter review; **(5) fail-fast** — if no override path is available (strict axes), the entity write is rejected and claude.ai recovers via `propose_canonical_addition`.

**Two flavors**:
- **FK-creating** (cultivar / terroir): canonical rows are materialized lazily on first reference, full metadata flows from the registry entry, FK-linked + provenance-tracked.
- **Text-only** (roaster / producer / brewer / filter / grinder): canonicalize-then-persist; no FK, no auto-created row. Override-eligible.

The pattern is **lazy by design** — canonical FK rows are only materialized when a write references them, never pre-populated speculatively. Shared cores (`validateCanonicalText` for text-only, `findOrCreate*` for FK) live in `lib/brew-import.ts` + `lib/canonical-registry.ts` to avoid re-implementing across axes.
_Avoid_: "upsert" (close but specifically write-with-update-on-conflict; find-or-create is read-then-conditional-create), "lookup-or-insert" (under-specified), "lazy canonical creation" (only covers the FK-creating half), "validate" (too generic — validation is a separate, broader concept).

**Canonical validation**:
The umbrella process every MCP write runs at push time — each canonical-axis field is resolved against its `CanonicalLookup` bundle and the outcome routes to one of three buckets: **resolves canonically** (write proceeds with canonical value), **resolves via alias** (write proceeds with the canonical target — raw input was just an alias for an already-known canonical), or **fails to resolve** (write either fails fast on strict axes or persists verbatim + queues to `taxonomy_overrides_queue` on override-eligible axes — the NET-NEW branch). Validation surface includes `isCanonical(v)`, `findClosest(v)`, `canonicalize(v)` (title-case canonical), `isResolvable(v)`, and `isOverridableValid(v)` used by UI save-gates. **Same validation logic runs in three places**: MCP push handlers (the canonical write surface), `/add` + `/edit` save paths (the legacy UI inputs, partially deprecated per the MCP-only input principle), and the patch route's per-field canonicalize blocks — single source of truth via the `makeCanonicalLookup(names, aliases?)` factory in `lib/canonical-registry.ts`.
_Avoid_: bare "validation" (too generic — canonical validation is specifically about canonical-registry alignment), "registry lookup" (lookup is one step within validation, not the whole), "canonical resolution" (close but resolution is the success path; validation includes the failure-routing decision).

**NET-NEW**:
Chris's verbal + memory-file shorthand for **the net-new-canonical branch of find-or-create** — the case where a write references a value that resolves neither canonically nor via alias. Triggers the override-flag path on override-eligible axes (entity persists verbatim with the canonical-promotion row landing in `taxonomy_overrides_queue` for arbiter review) or `propose_canonical_addition` on strict axes (entity write fails, queue row lands, claude.ai re-attempts the entity write after arbitration). NET-NEW writes are the **canonical-promotion entry point** — every queue row was a NET-NEW write at some moment. The arbiter's job on a queued NET-NEW entry is the **promote vs alias decision**: **promote** if the value is truly a new canonical taxonomy record (and at promotion time the full registry entry — 27 producer fields, 29 roaster fields, etc. — is populated, never left partial); **alias** if the value should resolve to an existing canonical (the alias map gets extended); **reject** if the value was drift / typo / not truly net-new; **duplicate** if it collapses to another pending queue row. Memory + followup files use the all-caps form `NET-NEW`; the glossary lemma matches.
_Avoid_: "new canonical" (under-specified — could mean a promoted-to-canonical), "missing canonical" (implies the registry SHOULD have it; NET-NEW is neutral about that), "unknown value" (too generic), bare "new" (overloaded).

**Bearer token auth**:
The long-lived `Authorization: Bearer <KEY>` auth path on the Latent MCP server — used by **MCP clients that don't speak OAuth**, principally Claude.ai's desktop app and Claude Code's local config. Keys are generated once via Supabase admin SQL into the `api_keys` table (`SUPABASE_SERVICE_ROLE_KEY` for the lookup), stored in the client's MCP config, and validated server-side on every call. Single Chris-owned api_keys row covers all bearer-auth clients; rotation = invalidate row, generate new, paste into client config. **Not a trust-tier distinction** — bearer-token and OAuth resolve to the same `user_id`, the same RLS scope, the same Tool surface; the split is purely about client protocol requirements. Bearer was the only auth path in **V1 sync** (the original MCP server ship in Sprint 2.3) because the desktop MCP client family only supported it; OAuth came in Sprint 3.0 specifically to unlock the web + mobile clients.
_Avoid_: "API key auth" (close — the literal mechanism is a bearer-token-API-key — but bearer token is the canonical MCP-spec phrasing), "static key", "long-lived key" (descriptive but not canonical), "service auth", "auth tier" (no tier distinction — bearer + OAuth carry equal trust).

**OAuth 2.1 + PKCE auth**:
The claude.ai web + mobile auth path on the Latent MCP server — added Sprint 3.0 (2026-05-03) specifically because the claude.ai web / mobile clients require OAuth; the desktop client's bearer-token path didn't extend to them. Uses MCP-spec discovery endpoints (`/.well-known/oauth-protected-resource` + `/.well-known/oauth-authorization-server`), a 5-minute auth code at `/api/mcp/authorize` (stored in `oauth_authorization_codes`, migration 043), an access token exchange at `/api/mcp/token`, and a `WWW-Authenticate` 401 trigger for re-auth. Static client credentials live in Vercel env (`MCP_OAUTH_CLIENT_ID` + `MCP_OAUTH_CLIENT_SECRET`). The single-tenant + single-user shape **collapses OAuth's heaviest costs to ~nothing** — no consent UI, no scopes, no identity surface — leaving only the spec-required discovery + token-exchange machinery. **The dominant auth path in active use** — Chris's primary input surface today is the claude.ai web + mobile app (mobile especially), so OAuth carries most production write traffic. Resolves to the same api_keys row as the bearer-token path, same trust, same Tool surface.
_Avoid_: bare "OAuth" (under-specified — version + PKCE matter), "web auth", "claude.ai auth" (descriptive but not canonical), "auth tier" (same reason as bearer-token entry).

**Roest API integration**:
The **external-device I/O surface** on the Latent MCP server — bidirectional integration with the Roest L200 Ultra ([roestcoffee.com/products/sample-roasters/l200-ultra](https://www.roestcoffee.com/products/sample-roasters/l200-ultra)), Chris's physical counterflow roaster. Six Tools span the surface: `push_roast_profile` (push a profile spec to the machine), `push_inventory` + `patch_inventory` (push and update inventory entries), `pull_roest_log` (pull one roast graph by ID — temperature / fan / power curves), `list_roest_logs` + `list_roest_inventory` (list available logs + inventory).

**The killer capability is the roast recipe push** — pre-integration, claude.ai handed Chris recipes as prose and Chris typed each V-set into the machine batch-by-batch; with V1a / V1b / V1c carrying minute differences between slots, the manual typing was error-prone and tedious enough that the integration was one of the highest-value MCP additions. With `push_roast_profile` live, claude.ai sets up the V-set AND writes the per-batch recipes into the machine in one step. **Reverse-engineered write surface**: Roest's published API documentation only covers read access; the write capability was discovered via a third-party GitHub project. Latent acts as a client to Roest's cloud API, brokering through credentials in Vercel env. Shipped 2026-05-06 (Roest API write Phase 1+2). One of the **only output exceptions** to the MCP server's primary-input direction (the other being `propose_doc_changes`); both write OUT of Latent rather than IN.

**Two distinctions to keep crisp**:
- **Roest L200 Ultra** is the **physical machine** (the device Chris roasts on, exclusively in counterflow mode); **Roest API** is the cloud surface the integration talks to. The MCP server is a client of the cloud, never directly connected to the machine.
- The integration is **not part of asymmetric write trust** — Roest writes are direct (no propose-then-apply), but the round-trip is mechanical (push a profile spec, the machine accepts it), not interpretive — there's no drift surface for an arbiter to manage.

**Adjacent equipment that is NOT integrated today**: the xBloom (manual one-recipe operation, no API need); kettle / brewer / scale / grinder (no API on any of them — all manual). Roest is the only piece of physical equipment in Chris's setup that exposes an API surface, and Latent uses all of it.
_Avoid_: bare "Roest integration" (under-specified — could mean machine, cloud, or MCP surface), "Roest sync" (close but integration is canonical), "external API integration" (too generic), conflating "L200 Ultra" with "Roest API".
