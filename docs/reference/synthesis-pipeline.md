**Humanizer pass**:
The terminal stage of the synthesis pipeline â€” a separate Sonnet 4.6 LLM call that takes the raw directed-synthesis output and rewrites it to **strip AI-tells + inject voice** while **preserving Latent domain vocabulary verbatim**. Uses the vendored `lib/synthesis/humanizer-skill.md` (the third-party `blader/humanizer` MIT skill, kept resync-able â€” do not edit in place) as its `system` prompt, wrapped with a `USER_OVERRIDE` ([humanizer.ts:23](lib/synthesis/humanizer.ts:23)) that (a) suppresses the skill's default Draft â†’ Audit â†’ Final multi-step output so the polished text returns in one shot, and (b) pins the preservation list: extraction-strategy names (the 6 canonical strategies) + gear (SWORKS / Melodrip / Hario Switch / Sibarist / Orea) + proper nouns (producer / farm / roaster names) + cultivar names + process terms + all temp / grind / ratio numbers + markdown bullet structure + paragraph breaks. **Why a separate call, not tighter raw-call prompting**: the humanizer is a cognitively distinct task (rewrite-for-voice on already-drafted prose vs. extract-patterns-from-corpus), the upstream skill is vendored verbatim and resync-able (folding it into the raw prompt would couple Latent's directed-question structure to upstream's vocabulary), and the humanizer is **reusable beyond synthesis** â€” Chris applies it to LinkedIn posts, emails, and other long-form writing where AI-tell stripping pays off. The separation is permanent and correct.
_Avoid_: "polish pass" (loses the AI-tell specificity), "humanizer step" (close â€” pass captures the LLM-call discreteness more precisely), "voice pass" (loses the strip-and-preserve duality), "post-processing" (too generic), "second call" (mechanical, doesn't carry the job).

**Knowledge capsule**:
The human-facing prose output of the synthesis pipeline â€” `4-6 short paragraphs in a natural human field-note voice followed by a bulleted list of 4-6 practical takeaways`. **The pipeline's terminal artifact, designed to be read by Chris, not consumed by any downstream workflow.** Code-side noun is `capsule` (per `EntityAdapter.capsuleNoun` â€” "living roaster knowledge capsule" / "living terroir knowledge capsule" / etc.). **This is what makes the humanizer pass load-bearing**: everything else in Latent is structured input feeding back into claude.ai / MCP workflows; the synthesis capsule is the rare surface where machine-aggregated corpus signal terminates in human-readable prose. Distinct from carry-forward learnings (which are *also* claude.ai-authored prose but feed forward to V1 design on a future lot â€” machine-consumable). Capsule is read-only by Chris; carry-forward learnings are read by both Chris and claude.ai.

**Living-at-time-of-creation** (Chris-explicit, 2026-05-16): the capsule represents the full synthesized knowledge of all data points about the anchor *at the moment of regeneration*. Not a static record â€” **regenerates from scratch every time a new data point lands**, so each generation pulls in the latest corpus signal multiplicatively, not additively. Chris's verbatim: "every entry shouldn't be static. That entry should be multiplicative every time I'm adding a new data point." Mirrors the *living reference docs* (BREWING.md / ROASTING.md) framing â€” same temporal disposition (continuously revised as the corpus grows), different scope (capsule = one anchor; living docs = cross-anchor general knowledge). **Codified as a SHARED_RULES directive Sprint T4** (2026-05-18) â€” the multiplicative-not-additive principle is now the top-of-rules directive in [buildPrompt.ts SHARED_RULES](lib/synthesis/buildPrompt.ts), paired with an explicit anti-pattern example to prevent large-corpus capsules from degrading into appended per-coffee notes.

**Cross-source by intent + by current implementation** (Sprint 13 / SYN-6, 2026-05-22): the capsule encapsulates **all underlying experiences about the anchor â€” both brewing and roasting lessons together** ("after you've tried fifty Caturra coffees... all the aggregated sum of knowledge of that particular cultivar based on all the experiences I've ever had, both roasting and brewing this specific thing"). 3 of 4 adapters (terroir / cultivar / roaster) now read both `brews` AND `roast_learnings` rows joined through `green_beans.{terroir_id,cultivar_id}`; the roaster adapter pulls all `roast_learnings` on the Latent path. Process adapter is brews-only â€” deferred per [ADR-0010](docs/adr/0010-cross-source-synthesis-and-three-call-pipeline.md) Q2 (signature-method join surfaces zero rows today; modifier-level cross-source needs heavier joins). The prompt SHARED_RULES integrate both vocabularies into a single unified narrative voice (anti-pattern: appended brewing-side/roasting-side sections); humanizer USER_OVERRIDE preserves the roasting-vocab canonical list verbatim (primary lever / brewing tolerance / underdevelopment signal / overdevelopment signal / aromatic behavior / structural behavior / rest behavior / scope tags).

**Origin lineage** (2026-05-16 audio): direct ancestor is the precursor claude.ai-in-thread loop â€” Chris would finish optimizing a brew, paste the brewing doc back into the thread, and ask Claude to "read this doc and increment it to v2 / v3 / v4 based on what was learned." BREWING.md's 8.x versioning sequence is the literal scar from that era. Today's architecture splits the same work into two surfaces: **living reference docs** (BREWING.md / ROASTING.md, cross-anchor general knowledge, updated via `propose_doc_changes` + arbiter) and **knowledge capsules** (per-anchor synthesis, updated via re-synthesis pipeline on data-point change). Same pattern, automated, sharded.
_Avoid_: "synthesis output" (close but generic â€” capsule names the artifact-shape distinctly), "lessons-learned page" (the page chrome contains other content), "aggregation summary" (loses the human-facing intent), "page synthesis" (close â€” same surface, but synthesis is the operation, capsule is the artifact), "static summary" (the living property is load-bearing), "snapshot" (snapshot is static â€” capsule is regenerated, not snapshotted).

**Synthesis pipeline**:
The multi-stage operation that takes `(adapter, entity, entityName, brews)` as input and produces a polished **knowledge capsule** cached in the per-axis synthesis table. Five stages: (1) **corpus query** â€” the synthesize route reads brews scoped to the anchor (e.g. `brews WHERE roaster = ?`); (2) **prompt assembly** â€” `buildSynthesisPrompt` composes anchor + **corpus tier** confidence hint + weighting block + tier-aware output-format block + shared-rules + per-adapter extra-rules ([buildPrompt.ts:58](lib/synthesis/buildPrompt.ts)); (3) **directed synthesis call** â€” Sonnet 4.6 generates the raw paragraph + takeaway output, paragraph + takeaway counts and `max_tokens` budget driven by the corpus tier ([runSynthesis.ts:61](lib/synthesis/runSynthesis.ts)); (4) **humanizer pass** â€” second Sonnet 4.6 call strips AI-tells + injects voice while preserving Latent vocabulary; (5) **cache write** â€” upsert into the per-axis synthesis table (`terroirs.synthesis` / `cultivars.synthesis` columns for FK-stored axes; `roaster_syntheses` / `process_aggregation_syntheses` tables for the rest). Total latency ~10-15s combined. **Regenerated from scratch every time** â€” never delta-applied; new data points trigger full re-synthesis (see knowledge capsule's living-at-time-of-creation property).
_Avoid_: "synthesis flow" (close but vague), "directed synthesis" (close but names only step 3), "two-call pattern" (mechanical, doesn't carry the corpus-query + cache write stages), "synthesizer" (too generic).

**Directed-prompt adapter**:
The per-entity contract that shapes one slice of the synthesis pipeline for one aggregation axis â€” code-side: `EntityAdapter<TEntity>` in [lib/synthesis/types.ts](lib/synthesis/types.ts). Four adapters today (terroir / cultivar / process / roaster), each contributing **5 per-entity slots** that vary while the rest of `buildSynthesisPrompt` stays shared: (1) **Anchor context** â€” `renderAnchor(entity)` injects the canonical registry entry's rich content (28 cultivar fields / 27 producer fields / 29 roaster fields / 4 base-process summary fields) as markdown bullets framed as a *working hypothesis* the model must confirm-or-push-back from the corpus, never recite. **The Pick-not-author principle made operational at synthesis-output time** â€” the same rich content the registry holds for write-time validation flows into the directed prompt for read-time synthesis. (2) **Weighting block** â€” 6-8 ordered weighting tiers (`highest / high / medium / low`) telling the model how to rank corpus signals; per-adapter authored. (3) **Output format** â€” 5-6 numbered paragraph-order steps; `buildPrompt` prefixes with the shared "4-6 paragraphs in field-note voice followed by 4-6 takeaways" block. (4) **Extra rules** â€” adapter-specific addenda appended to `SHARED_RULES` (the 7 universals about hypothesis-framing / no-overfitting / hedge-language / conciseness / field-note voice); per-adapter extras lock per-entity hedges ("if process dominates cultivar, say so"). (5) **Learning-row formatter** â€” `formatLearningRow(brew)` projects each brew into a JSON dict the prompt embeds verbatim; per-adapter cherry-picks the relevant columns (terroir includes `terroir_connection`; cultivar includes `cultivar_connection`; roaster includes `producer + roast_level + extraction_confirmed`; process includes `roast_level`). Corpus is shared, projection is adapter-specific.

**Two adapter shapes today** (forward-investment pattern, not just a code-style choice):
- **Single-shape adapter** (terroir / cultivar / roaster) â€” one adapter per axis, one anchor shape, one set of slots.
- **Per-aggregation-kind adapter** (process only â€” `getProcessAdapter(kind)` dispatches to 5 internal sub-shapes: base / honey_subprocess / modifier_combo / modifier_index / signature). All 5 sub-shapes share `SHARED_WEIGHTING + SHARED_OUTPUT_FORMAT + SHARED_LEARNING_ROW`; only `renderAnchor + entityNoun + capsuleNoun + extraRules` vary by kind. Matches the **Composable taxonomy** shape of the process axis â€” the page surface itself is multi-kind (per Sub Pages 4: 4 base hubs + 8 modifier-index pages + 3 signature pages + per-base modifier-combo mini-pages), so the synthesis adapter mirrors that multiplicity. **Template for any future axis with sub-page kinds** â€” if `/producers/{name}/system/{system}` or similar lands, the 2nd pattern is the inheritance shape; do NOT inline a 5th per-entity adapter for what's structurally one axis with multiple aggregation kinds.

**Design rule (CLAUDE.md, ratified at sprint Per-entity directed synthesis 2026-05-08)** â€” *"New aggregation surface = one new adapter; do NOT inline a fifth per-entity prompt."* Per-aggregation-kind sub-dispatch **within an adapter file** is allowed (process is the precedent); inlining a sibling prompt outside the adapter shape is not.
_Avoid_: "per-entity adapter" (close â€” but adapter alone is the slot-contract noun; directed-prompt names the design discipline), "synthesis adapter" (close â€” but directed-prompt makes the prompt-design discipline explicit), "synthesis template" (loses the per-entity dispatch), bare "adapter" (overloaded â€” there are MCP adapters and other adapter patterns in the codebase).

**Bottoms-up synthesis-prompt authoring**:
The design pattern Chris used to construct each of the 4 directed-prompt adapters â€” **same bottoms-up discipline as taxonomy authoring**, applied to synthesis prompts. Three-step exercise per axis: (1) hand-author one full synthesis report ("Terroir: Lessons Learned â€” Central Andean Cordillera" / "Cultivar: Lessons Learned â€” Gesha" / "Roaster: Lessons Learned â€” Picolot" / "Process: Lessons Learned") with ChatGPT iteration, until the output reads the way Chris wants; (2) ask ChatGPT a two-part backward-design question: *"of the data points I gave you, which ones actually mattered to producing this output, and what's the percent weighting?"* + *"if you had to design a prompt that would generate output of this shape given only the data points that mattered, what would it look like?"*; (3) port the resulting prompt structure verbatim into the adapter file (the file-header comment on each adapter â€” "Prompt structure ported verbatim from Chris's 'X: Lessons Learned' sample-output file" â€” is the traceability scar). **The weighting tiers in each adapter came directly from the percent-weighting backward-design question.** The **humanizer pass exists because ChatGPT's output was "very AI feeling"** â€” Chris wrapped the humanizer as a post-processing stage rather than refactor the directed prompt to produce humanlike text natively. The 4 sample-output files live in claude.ai project context, not in the repo today.
_Avoid_: "synthesis prompt design" (close but generic â€” bottoms-up captures the iteration discipline), "sample-output authoring" (close â€” sample-output IS the artifact; bottoms-up IS the discipline), "prompt engineering" (jargon, loses the corpus-grounded specificity).

**Cross-coffee insight layer**:
The umbrella canonical noun for **cross-coffee patterns that span multiple lots / coffees and generalize forward to future ones** â€” the surface that complements per-anchor knowledge capsules. Lives in the **living reference docs / sub-skill clusters**, NOT in the synthesis pipeline. Both surfaces share the same heading post Sprint 9 (2026-05-19) and both migrated to their Historian clusters in Wave 2 (2026-05-26): roasting-side at [docs/skills/roasting-historian/cluster/patterns/cross-coffee-insights.md](docs/skills/roasting-historian/cluster/patterns/cross-coffee-insights.md) (Wave 2 PR 3) with sibling [open-questions.md](docs/skills/roasting-historian/cluster/patterns/open-questions.md) + FK-seeded by-cultivar/gesha.md + by-process/washed.md + 7 per-lot learnings/ pointer files; brewing-side at [docs/skills/brewing-historian/cluster/patterns/cross-coffee-insights.md](docs/skills/brewing-historian/cluster/patterns/cross-coffee-insights.md) (Wave 2 PR 2) with per-strategy splits into [docs/skills/brewing-historian/cluster/patterns/by-strategy/](docs/skills/brewing-historian/cluster/patterns/by-strategy/). Both master docs are now redirect-stub pointer tables post Wave 4 PR 4b (2026-05-21) â€” they no longer carry `#cross-coffee-insight-layer` (or, on [ROASTING.md](ROASTING.md), `#open-questions`) section anchors; each routes to its cluster home via a "Former section â†’ Now at" map. **Promotion rule (both sides): 2+ coffees / lots before a pattern earns a section** â€” single data points are observations; â‰Ą2 is the beginning of a pattern (BREWING.md: "Don't add a pattern until you have seen it in at least 2 coffees"; ROASTING.md: "patterns that have generalized across â‰Ą2 lots"). **Source-diversity refinement** (Item 1, Sprint R Phase 4 Step 4 grill, 2026-05-23): for self-roasted lots, count each bean as a single source regardless of how many lifecycle data points (cuppings, brews, optimized pourovers) the bean produced. Promotion-worthy patterns need 2+ DISTINCT sources (different farm, different roaster, ideally both), not just 2+ data points off the same bean. Updated at lot-close (roasting) or brew-archive (brewing); read at workflow start as informed starting hypotheses ("when next brewing X, here's what 3 prior X-coffees taught me about strategy choice"). **Structurally a different surface from knowledge capsules**: capsules are per-axis (Gesha capsule, Panama capsule), single-anchor, machine-generated, lazy-resynthesized; cross-coffee insight layer is cross-axis (anaerobic-natural-Suppression pattern spans cultivar + terroir + process), manually-authored at lot close, propose-then-apply via `propose_doc_changes` arbiter. **Where cross-entity insights live today** â€” until / unless cross-entity synthesis capsules earn first-class status (out-of-scope by design 2026-05-16; see flagged ambiguity), this is the canonical home for them. **Candidate concepts pending more N** live at [docs/skills/ccil/cluster/observing.md](docs/skills/ccil/cluster/observing.md) â€” structural concepts named but not yet promotion-ready; surfaced at standard arbitration time alongside doc proposals + taxonomy queue + skeleton entries. **Temporal constraint** (Chris-locked Item 3, Sprint R Phase 4 Step 4 grill, 2026-05-23): CCIL fires only AFTER complete cycles (lot close on roasting, brew archive on brewing), never during in-flight workflows. The post-cycle meta-synthesizer that compounds learnings across both domains + selectively external sources; distinct from in-flight Coordinator dispatch (which happens during workflows).
_Avoid_: "cross learnings" (Chris's verbal phrasing â€” fuzzy; the canonical doc heading is *Cross-Coffee Insight Layer*), "cross-coffee lessons" (same), "archive patterns" (retired BREWING.md heading-name pre-Sprint-9; the umbrella is the broader function â€” both surfaces now share the canonical heading), "pattern layer" (under-specified), "cross-cutting insights" (close â€” cross-cutting is the WBC term for unpromoted brewing patterns; cross-coffee insight layer is the umbrella for both roasting + brewing cross-coffee generalizations).

**Variety throughline**:
The shared character description that holds across both the roasting layer and the brewing layer for a given variety â€” the constant both layers' mechanics are pointed at preserving. Sudan Rume's throughline (across CGLE Hybrid Washed + CGLE Natural V5 + externally-roasted CGLE Las Margaritas) is "transparency-driven aromatic landrace with variety-intrinsic light brown-tea body and a herbal-spicy aromatic register." Washed Geisha's throughline is "delicate floral + tea-like." Both roasting and brewing decisions on the same lot are pointed at preserving the same target; the per-layer choices differ but the goal does not. **Per-layer mechanics** are the layer-specific operational details (roasting: FC window, peak inlet, fan curve; brewing: brewer/filter, grind, temperature, pour structure) that either serve or suppress the throughline. The throughline is the *what* (immutable character of the variety); per-layer mechanics are the *how* (layer-specific toolkit). CCIL pattern shape opens with the throughline, then surfaces each layer's mechanics, then flags any cross-domain tension where one layer's findings contradict another's. Confidence in a throughline characterization requires source diversity â€” see **Cross-coffee insight layer** Â§ source-diversity refinement. Distinct from **Process throughline** (a candidate concept at [observing.md](docs/skills/ccil/cluster/observing.md): the analogous shared-character pattern that holds across layers when the process character is strong enough to dominate variety / terroir signal â€” heavy co-ferments are the load-bearing case).
_Avoid_: "variety character" (too generic), "variety profile" (overloaded with roast profile), "cultivar fingerprint" (sounds like a quantitative signature).

**Resynthesize trigger**:
The runtime condition under which a knowledge capsule gets re-built. **Lazy, corpus-delta, page-visit-bound** by deliberate design â€” fires on `useEffect` mount of [components/SynthesisCard.tsx:35](components/SynthesisCard.tsx:35) when `currentBrewCount > 0 && (!existingSynthesis || existingBrewCount !== currentBrewCount)`. Two trigger paths: (1) **auto-resynthesize** â€” page-visit hook fires when the count of brews under this anchor differs from the count cached at last synthesis (catches both cold-start "no synthesis yet" and corpus-delta "new brew added since last view"); (2) **manual resynthesize** â€” `Regenerate` button bypasses the count check, always re-runs the pipeline. Both paths call the same synthesize route + `runSynthesis`; only the trigger condition differs.

**Corpus tier**:
The 4-bucket classifier that maps `learningRows.length` (the count of brews with meaningful learning signal in the synthesis input) into a tier driving (a) confidence-language hint injected into the prompt, (b) target paragraph count, (c) target takeaway count, (d) `max_tokens` budget. Replaces the pre-Sprint-T4 binary `earlyData` flag with a gradient. Lives in [lib/synthesis/buildPrompt.ts](lib/synthesis/buildPrompt.ts) `classifyTier(rowCount)`; surfaced on every `buildSynthesisPrompt` return + `runSynthesis` outcome.

Tier table:
- **Early data** (n < 3): 2-3 paragraphs + 2-3 takeaways, `max_tokens` 700. Provisional first-impressions framing ("too early to call", "first impressions suggest", "one more data point would settle this"). Short capsule by design â€” a long capsule on N=1 overstates the signal.
- **Emerging pattern** (3-7): 3-4 paragraphs + 3-4 takeaways, `max_tokens` 1100. Working-hypothesis framing ("usually", "in my archive so far", "the pattern is recurring but the sample is still narrow").
- **Established pattern** (8-15): 4-5 paragraphs + 4-5 takeaways, `max_tokens` 1500. Standard hedge language ("often", "tends to", "consistently").
- **Mature signal** (16+): 4-6 paragraphs + 4-6 takeaways, `max_tokens` 1500. Less hedging ("reliably", "consistently"); surface deep generalizations plus the specific edge-cases where the pattern breaks (cultivar override / process override / roaster override).

**Origin**: Chris-flagged 2026-05-16 ("a 1-coffee Catimor capsule shouldn't sound as authoritative or be as long as a 30-coffee Gesha capsule"); shipped Sprint T4 (2026-05-18). The boundary numbers were chosen to match Latent's current corpus distribution â€” most cultivar / process pages land tier 1-2; popular terroirs (VolcĂˇn BarĂş Highlands at N=24) + a few processes (Anaerobic at N=16) hit tier 3-4 today.

**Paired with the re-synthesis directive**: the **mature-signal** tier is exactly where appended-list degradation is most likely (large corpus â†’ temptation to write one observation per coffee). The SHARED_RULES top-of-list directive + anti-pattern example (also shipped Sprint T4) are explicitly framed to make narrative-shape non-negotiable as corpus size grows.
_Avoid_: "confidence tier" (close â€” but the tier also drives length, not just confidence; the umbrella covers both axes), "early-data flag" (the predecessor binary, retired Sprint T4), "size bucket" (mechanical, loses the prompt-shaping intent), "gradient" (close but un-named â€” the canonical noun is corpus tier).

**Three load-bearing properties** (Chris-affirmed 2026-05-16, not an implementation accident):
- **Cost-bounded** â€” each pipeline run is two Anthropic API calls (raw synthesis + humanizer pass), billed to Chris's pay-as-you-go account. Lazy regen keeps credit spend proportional to viewing frequency, not to corpus mutation rate. Pre-warming on every corpus change would burn credits on pages that never get visited.
- **Single-user-of-one** â€” staleness only matters when Chris visits; no other reader exists. "Unless I go back to the page, nobody else is looking at it, so it doesn't really matter if it's stale."
- **Terminal output, not input** â€” knowledge capsules don't feed into any downstream MCP workflow / claude.ai input / writeback path. Staleness has no propagation cost, so the lazy model is structurally sound (a feedback-loop synthesis surface couldn't tolerate the same lag).

**Content-change scenarios 2 + 3 closed Sprint 13 / SYN-7 (2026-05-22)**: alongside the existing `synthesis_brew_count` count signal, every cache surface now carries `synthesis_input_max_updated_at` â€” the max(`updated_at`) across the corpus rows at synthesis time. Page-side, the consumer recomputes the same value from the rendered corpus + roast_learnings (the `computeInputMaxUpdatedAt` helper in [lib/synthesis/inputUpdatedAt.ts](lib/synthesis/inputUpdatedAt.ts)) and passes it to SynthesisCard; mismatch triggers regen alongside the count delta. Closes the two scenarios that became load-bearing once SYN-6 shipped â€” `what_i_learned` rewritten on an existing brew, `roast_learnings` filled in for an existing lot. Scenarios 1 (anchor FK reassigned post-arbiter) and 4 (registry rich-anchor code edits) remain manual-Regenerate-button territory per [ADR-0010](docs/adr/0010-cross-source-synthesis-and-three-call-pipeline.md) Q4 â€” neither is frequent enough today to warrant tracking.
_Avoid_: "regen trigger" (close â€” mechanical, doesn't carry the multiplicative semantics; resynthesize matches Chris's verbal model and the knowledge capsule's regenerate-from-scratch property), "cache invalidation" (CS jargon â€” the cache isn't invalidated, the row is overwritten on next page visit), "stale-revalidate" (close â€” same shape as SWR, but resynthesize is single-call not double-render), "regeneration policy" (over-engineered).
