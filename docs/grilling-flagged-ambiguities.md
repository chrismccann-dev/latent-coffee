This file is the **standing ledger of Latent vocabulary ambiguities surfaced through past `/grill-with-docs` sessions**, with resolution status per item (strikethrough + resolution date = closed; live text = still open). Distinct from [docs/grilling-queue.md](docs/grilling-queue.md), which is the standing list of *concepts and framing questions for the NEXT grill*. - "Experiment set" was Chris's prior term for the same concept (used interchangeably during the transition to V notation). Resolved: **V-set** is canonical; the V prefix is already encoded in the spreadsheet labels, Roest profile names, and DB identifiers.
- Initial grilling attempt conflated **variable** and **lever** into one term. Resolved: they're distinct — variable is the design-time noun (pre-judgment about whether it matters), lever is the post-hoc promotion (variable that turned out to affect the cup), and **non-factor** names a variable that was tested but didn't matter.
- "Reference cup" was initially treated as the umbrella for "everything tasted from the reference roast". Resolved: it's specifically the xBloom day-7 pour-over cupping; the optimized pour-over recipe is the **optimized brew** (separate entry). Three distinct things: reference roast (the bean state) → reference cup (the xBloom evaluation) → optimized brew (the consumption-condition cup).
- "Best roast" / "winning roast" naming was rejected in favor of **reference roast** because tasting is subjective and there is no objective best — "reference" reflects the calibration role this roast plays for the lot.
- ~~"Elasticity" was previously a single concept conflating roast-side latitude (range of acceptable lever values) and brew-side latitude (how the cup holds up under brewing push/pull).~~ **Resolved 2026-05-19 via Sprint 10 / migration 060 / ADR-0007**: split into **acceptable roast window** (roast deviation, content on `roast_learnings.roast_window_width`, UI label "Acceptable Roast Window") and **brewing tolerance** (brew deviation, content on the renamed `roast_learnings.brewing_tolerance` column, UI label "Brewing Tolerance"). Latent's house-style motto reframed in lockstep: "Roast for brewing tolerance, brew for intensity" — the controls-both-sides posture (Chris roasts and brews) is what makes brewing tolerance the load-bearing axis worth naming explicitly.
- "Lot" cleanly handles single-cultivar single-origin purchases but is lossy for **incoming blends** (a single green-bean SKU sold as a 2-cultivar mix from the same processing station — e.g. Libertad Bourbon + Caturra). **Roasted blends** (multiple lots roasted separately then combined) aren't modeled at all yet — future schema concern if WBC-style work enters scope.

Originally surfaced during the 2026-05-14 grilling session; re-flagged in the 2026-05-17 cross-party audit (RO-CP-7).
- ~~**Existing roast_learnings underdev/overdev data may intermix cup-side signals with roast-side observations.**~~ **Resolved 2026-05-20 via Sprint 11 / RO-5 prompts-only enforcement.** Pre-flight audit of the 3 populated cells (Mandela XO + Sudan Rume Hybrid Washed + Surma) found all 3 already follow a clean **cup-first + labeled-roast-correlate pattern**: cup observation first, then optional `Roast-level signal: <Agtron threshold / FC temp / Maillard %>` correlate explicitly named. - **Phase-mapping had two competing phrasings, resolved 2026-05-15 via formal split.** BREWING.md's `saturation / body / clarity / finish` mixed mechanical role with cup-side outcome under "sensory target." Resolved: every phase carries a **mechanical role** (what the brewer does — names a WBC axis) plus a **cup-side target** (which sensory attribute the phase delivers); these are two parallel labels, both load-bearing in recipe design. **Canonical cup-side attribute set: the fixed 5** (`aroma / attack / mid-palate / body / finish`) — same vocabulary across Step 3 tasting and Step 4 resolved-brew output, locked 2026-05-15. BREWING.md doc-edit follow-up: update Phase-Mapped Hybrid + Role-Based Pulse definitions to surface both labels and drop the `saturation / body / clarity / finish` shorthand.

- **Brewing signal arbitration is pure precedence; roasting signal arbitration is precedence + weighting (parameter-type-conditional)** (resolved 2026-05-17 via cross-party roasting audit, round 5).
