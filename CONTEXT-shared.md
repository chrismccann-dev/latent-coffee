### Cross-domain Workflow

**Brewing-to-roasting handoff brief** (Cluster A, 2026-06-01):
A structured doc a brewing-project thread produces and a roasting-project thread pulls back in, carrying a brewing-side result across the claude.ai **project boundary** (claude.ai/brewing ↔ claude.ai/roasting) without loading brewing context into the roasting thread. Three instances in the lot pipeline: the **simulated-pourover recipe handoff** (ephemeral — the first-guess recipe + derivation, no brew row); the **peer-variant handoff** (the 5-field assessment of an external roaster's version — see CONTEXT-roasting § Peer-roasted reference brew); the **optimized-brew handoff** (carries the pushed `brew_id` so close-lot can link `green_beans.optimized_brew_id`). Same shape and rationale as the research coordinator ↔ assistant handoff — the boundary exists to protect the roasting thread's context window. Both projects hit the same MCP server but keep separate context windows; cross-boundary *reads* use existing Tools (`get_green_bean` / `get_brew`), so the handoff is doc-passing, not a new Tool surface.
_Avoid_: "handoff doc" (too generic — Latent has several); conflating with the research-coordinator handoff (same shape, different domain).

## Relationships

- A **V-set** belongs to exactly one green-bean lot.
- A **V-set** contains 1–4 **batch slots** (typically 3, occasionally 4, rarely 1–2 for one-shot lots).
- A green-bean lot progresses through one or more **V-sets** until a reference roast emerges.
- Each **batch slot** corresponds to exactly one roast execution and exactly one Roest machine profile (same name everywhere).
- A later **V-set** (e.g. V2) is informed by the outcomes of the earlier V-set on the same lot.
- Each **V-set** has exactly one **experiment frame**, authored before any roasting happens.
- A green-bean lot's **brew-web** (Cluster A) links up to three brew artifacts: *my brews* (`brews.green_bean_id`, many), *my optimized brew* (`green_beans.optimized_brew_id`, 0–1, the lot's canonical end-state cup), and a *peer-roasted variant* (`green_beans.peer_reference_brew_id`, 0–1, external calibration anchor). The two canonical pointers are explicit sibling FKs set at the earliest moment both rows exist, fed by a **brewing-to-roasting handoff brief** — not a join table (1:1 in lived practice), not a heuristic (the legacy `pickOptimizedBrew` fallback only). See [ADR-0019](docs/adr/0019-lot-brew-web.md).
- An **experiment frame** predicts outcomes at both the roast layer (what the curve will look like) and the cup layer (what the brew will taste like); these predictions get compared against actuals to produce two distinct deltas per batch slot.
- A **variable** becomes a **lever** post-hoc once cross-batch-slot or cross-V-set evidence shows it meaningfully affects the cup; if the same evidence shows no effect, it becomes a **non-factor** instead.
- An experiment frame's `levels_tested` enumerates the values a **variable** takes across batch slots (e.g. fast / medium / slow, or 240°C / 250°C / 260°C peak inlet).
- A **roast→cup trace** runs per batch slot — each slot has its own predicted-roast, actual-roast, roast-delta, predicted-cup-initial (on the recipe), updated-cup-prediction (post-roast), taste-for, actual-cup, and cup-delta.
- A V-set's **taste-for** for slot `n` is informed by what was tasted in the corresponding slot of prior V-sets on the same lot, plus the producer tasting notes, plus the adjustment being tested this round.
- All taste-based terms in this glossary are **single-palate** (Chris's). Producer tasting notes are an external ballpark anchor, not a target.
- Each V-set produces a **leading slot** (the V-set-scoped winner; lives in `experiments.winner`). The leading slot of the final V-set is typically (but not always) promoted to the lot-level **reference roast** at close-out.
- A **reference roast** emerges from a V-set when the cup matches expectations + diminishing returns set in, and is typically locked via one final **control experiment** (which replicates the prior V-set's leading slot with two slight adjustments to confirm).
- A **reference cup** is the xBloom day-7 pour-over of the reference roast — one cup, one method, one row.
- An **optimized brew** is dialed in *after* the reference cup, by feeding the reference roast into the brewing-side workflow (same as a purchased roasted bean).
- The end target of the full roast→cup trace is the **optimized brew**, not the reference cup. Post-hoc attribution flows backward: optimized brew → reference cup → reference roast → variables / levers / non-factors.
- Green-bean inventory is finite: each lot supports a bounded number of V-sets (typically ~10 100g batches per pound), making the trace time-constrained. Faster lessons-learned compounding = fewer V-sets needed to reach a reference roast.
- A **reference roast** has a **roasted bean characteristic** that the brewing-side workflow reads when dialing in the optimized brew.
- The "Roast Character" mini-card on the resolved page surfaces three attributes side-by-side: **primary lever**, **acceptable roast window** (roast-deviation latitude), and **brewing tolerance** (brew-deviation latitude). They're three different facets, not three names for the same thing.
- Chris's roasting/brewing style **deprioritizes brewing tolerance** in favor of expressiveness — he intentionally pushes extremes (very light + Full Expression on one side, very dark + Suppression on the other), so a low-tolerance lot is acceptable provided it doesn't fully collapse.
- A green-bean lot moves through lifecycle states in this order: **In inventory** → **Waiting for next roast** → **Waiting for next cupping** → (loop back to Waiting for next roast for V_{n+1}, or progress to) **Resolved**.
- The loop between Waiting for next roast and Waiting for next cupping repeats N times (one per V-set) until a reference roast is declared.
- The transition from a winning cupping to Resolved compresses two sub-events: (1) reference roast designation immediately after a winning xBloom cupping, (2) optimized-brew dial-in by handing the reference roast into the brewing-side workflow (same flow as a purchased roasted bean). Both must complete before the lot is marked Resolved.
- Concurrent multi-lot mode is the normal operating state: Chris typically has several lots in different lifecycle states simultaneously, visible side-by-side on the /green index.
- **Division of labor across the workflow**: Chris does all physical work (roast, cup, taste, judge); **claude.ai** designs experiment frames, writes hypotheses, and inserts/updates app rows via the MCP server; the **Latent app** archives the resulting state. claude.ai is the canonical author for every roasting-side entity.
- A **lot** is uniquely identified by `green_lot_id`; same producer + same cultivar across two purchases = two distinct lots that behave and taste differently (agricultural variability).
- A lot is **typically single-cultivar single-origin**, but green-bean SKUs are occasionally sold as **incoming blends** (e.g. Libertad: Bourbon + Caturra from the same processing station). The current schema (one cultivar / terroir per `green_beans` row) is lossy for these — known limitation.
- **Intentional roasted-blending** (WBC-style: same bean roasted three ways and blended, or two varieties blended) is NOT in current scope but is on the long-term roadmap as Chris's interests move toward competition-style work.
- A resolved lot produces both **lot-specific learnings** (backward-looking audit-trail) and **carry-forward learnings** (forward-looking compounding-knowledge primitive). Two cards, two roles, two audiences.
- **Carry-forward learnings are the compounding-knowledge primitive** — claude.ai reads them when designing V1 on a new lot with overlapping cultivar/terroir/process. Faster compounding = fewer V-sets needed to reach a reference roast on future lots. This is the *whole point* of Latent as a journal.
- **Non-intervention principle**: Chris does not hand-author or edit any roasting-side records. claude.ai writes via MCP based on the cupping/roast trace; Chris's role is physical execution + judgment. Manual intervention is reserved for "glaringly wrong" cases (and even then he asks claude.ai to fix it rather than editing directly). The goal is a self-sustaining knowledge system.
- **Underdev signal** and **overdev signal** key off **development** (whole-curve completeness, including WB→Gnd Agtron uniformity), not classical post-FC time alone.
- The xBloom gate exists precisely **because cup-side signals must be isolated from brewing variance** — bowl cupping's variance smuggled too much brewing-tolerance noise into what looked like cup-side observations.
- **Diagnostic signals** (underdev / overdev) describe *failure modes* — what goes wrong when development is pushed off-target. **Character descriptors** (aromatic / structural / rest behavior) describe *what the cup is* when roasted well. Two different jobs, two different homes.
- Every brew names exactly one **extraction strategy**, drawn from a strict 6-value canonical enum. The strategy is declared at the Coffee Brief Step 1d *before* equipment, grind, temp, or pour structure are decided — those parameters fall out of the strategy choice.
- **Extraction strategy is intent-defined, not mechanics-defined.** Two brews with identical recipes can name different strategies if the intent differs (Suppression vs Clarity-First on the same coarse-grind / low-temp / low-agitation recipe). Strategy reclassification across schema versions (e.g. the v8.4 Hybrid promotion that moved Sebastian Ramirez White Honey + Janson 1010 from `Balanced Intensity + Immersion modifier` to `Hybrid (Sequential)` without changing the recipe) is therefore a frame change, not a recipe change.
- The 6 strategies split 5+1: five **intensity** strategies (one logic running throughout the brew) + one **structure** strategy (`Hybrid`, where phase boundaries change the brew's mode mid-stream). When the brewer changes mode (immersion ↔ percolation, or pours have explicitly different sensory jobs) the strategy is Hybrid; otherwise pick from the five intensity strategies.
- The **two-axis framework** is a deliberate **learning-stage compression** of the WBC five-axis taxonomy, chosen because (a) many WBC subtypes are not in Chris's active practice yet, (b) the compression keeps the canonical taxonomy + picker surface tractable, and (c) the modifier system gives extensibility without committing to first-class status. The shape is expected to evolve, not to remain at 2 axes forever.
- A brew may carry zero or more **modifiers**; modifiers are stackable and operate on mechanical axes orthogonal to the chosen extraction strategy. Default empty — a "no modifiers" answer at Step 1d is a positive signal that modifiers were considered.
- An Axis 2 **modifier** can be **promoted** to a first-class Axis 1 **extraction strategy** when sustained usage + cross-coffee pattern + a coherent distinct intent justify the schema cost. Promotion churns existing brew records (reclassification at promotion time); the bar is deliberately high.
- The v8.4 Hybrid promotion (Immersion modifier → Hybrid strategy) is the canonical promotion precedent — driven by SWORKS Bottomless Dripper office usage reaching threshold plus selective Hario Switch home use. The three pre-promotion brews (Sebastian Ramirez White Honey, Janson 1010, Finca La Reserva) reclassified to Hybrid (Sequential) at promotion without recipe changes.
- Cross-cutting WBC moves not promoted to canonical modifiers (water strength / agitation taper / filter behavior / pre-brew conditioning / time-distribution templates beyond Role-Based Pulse) persist via free-text `strategy_notes` on the brew row and live as doc-layer playbook patterns in [docs/skills/wbc-brewing-archivist/cluster/wbc-reference.md](../../docs/skills/wbc-brewing-archivist/cluster/wbc-reference.md).
- WBC's **Physical System** axis (Structural Systems family) is explicitly out of scope — blend-engineering, layered beds, dual-chamber extraction don't apply to Latent's single-origin home brewing context.
- "Promotion" is overloaded across roasting and brewing — the roasting-side **variable → lever** promotion is post-hoc per-V-set evidence-driven; the brewing-side **modifier → strategy** promotion is sustained-usage / schema-evolution-driven. Distinct mechanisms, distinct domains, same word.
- **Hybrid is the only extraction strategy with a mandatory second-level pick** (`hybrid_subform`, 5-value enum). The other 5 strategies are self-contained — their name carries the recipe intent. Hybrid is a pointer that becomes meaningful only once the sub-form names what kind of structural compositing the brew is doing.
- Hybrid sits on a different axis from the other 5 strategies: **structural composition** (phase boundaries change brew mode mid-stream) rather than **extraction intensity** (one logic running throughout). This is why Hybrid's labels are about *kinds of compositing* rather than *amounts of extraction*.
- The Phase-Mapped sub-form ties each pour to a cup-side attribute it primarily extracts; the canonical mapping is bloom → aromatics, Pour 1 → acidity, Pour 2 → sweetness, last pour → finish. The brewer's per-phase mechanics (valve / agitation / flow) are then tuned to emphasize the targeted attribute.
- Selective Bloom Hybrid and the Output Selection modifier both involve treating a fraction of the extraction separately, but the architectural split is intentional: **Selective Bloom = role assignment** (the bloom participates differently in the cup design), **Output Selection = fraction discarding** (kept some, threw some away). The wbc-reference.md doc explicitly preserves this split.
- Every **phase** in a brew carries **two parallel labels** (formal split, 2026-05-15): a **mechanical role** (what the brewer is doing — which WBC axis is engaged) plus a **cup-side target** (which sensory attribute the phase is meant to deliver). The two are orthogonal and both load-bearing during Phase-Mapped Hybrid + Role-Based Pulse design.
- The mechanical role per phase **names which WBC foundational control axis is being engaged** — extraction-intensity (grind / temp / ratio), time-distribution (pulse / hold / drain), physical-system (valve / immersion / percolation), or external-control (Melodrip / chilling ball / kettle off-base). This is the through-line that ties Latent's per-phase recipe vocabulary back to the WBC 5-axis lineage.
- BREWING.md's prior compressed phrasing `saturation / body / clarity / finish` (used in both Phase-Mapped and Role-Based Pulse definitions) **collapsed mechanical role and cup-side target into one mixed vocabulary** — it's been formally retired in favor of the two-label split. BREWING.md will need a doc-edit pass to update both definitions; flagged for sprint.
- **Cup-side targets use the fixed 5-attribute set** (`aroma / attack / mid-palate / body / finish`) — same vocabulary across recipe design, Step 3 iteration, and Step 4 archive output. Locked 2026-05-15.
- **Phase → cup-side-target pairing is soft.** Typical defaults exist (bloom → aroma, last pour → finish) but recipes vary in pour count (single-pour through 5+) and a phase often touches multiple attributes; design-time pairings are aspirations, not guarantees. Number of pours is itself a lever — more pours = more extraction into the cup, fewer pours = less.
- Bloom and double-bloom variants are explicit **future-scope experiments**, not currently in routine practice. They're queued in `wbc-reference.md` Tier 1 experiments but not yet shaping the canonical phase vocabulary.
- The **Coffee Brief** is the brewing-side analog of the roasting-side **experiment frame** — both are pre-execution, claude.ai-authored design artifacts that lock intent before physical work happens. Differences are real and load-bearing: Coffee Brief covers one brew (cheap to re-do — re-brew from the same bag), experiment frame covers one V-set of typically 3 batch slots (expensive — burns ~10% of green-bean inventory per V-set). Coffee Brief has an explicit confirmation gate ("yep go for it"); experiment frame is typically implicit (claude.ai writes, Chris doesn't push back). Both are claude.ai-authored on Chris's behalf — neither is hand-edited at runtime.
- The **brewing iteration loop** is the brewing-side analog of the roasting V-set cycle, but **simpler and sequential**: one recipe per turn, ~3 brews average, no parallel batch slots, cup-side feedback drives recipe adjustment. **Brewing only touches the last segment of the cup pipeline** — the bean is fixed input from the roaster, only the brewing recipe varies; roasting touches the full chain (green selection → roast recipe → cupping → optimized brew).
- The **end-state** of the brewing iteration loop is the **reference brew** (covered in next round). Like the roasting reference roast, it's not designated via a control experiment — it's recognized when iteration deltas flatten and the cup matches the targeted profile. One-shot successes get a single confirmatory brew before locking.
- **Diminishing returns** is the canonical iteration-stop signal on both sides — same word, same shape. On the roasting side it triggers reference-roast designation; on the brewing side it terminates the iteration loop and triggers reference-brew commit.
- **claude.ai is the canonical input author across the entire Latent workflow** — explicitly stated by Chris: "All of the things that end up inside the Latent app are not pre-written by me, not edited by me, not drafted by me." This applies to brewing-side workflow now (Coffee Brief authoring, Step 4 resolved-brew sync) the same way it already applies to roasting (per [feedback_mcp_only_input.md](~/.claude/projects/-Users-chrismccann-latent-coffee/memory/feedback_mcp_only_input.md)).
- **CONTEXT.md is registered as a Resource in the Latent MCP server** ([lib/mcp/docs.ts](../../lib/mcp/docs.ts) at `docs://context.md`) precisely because the claude.ai input side needs to be familiar with the same vocabulary the codebase uses — Chris stated this directly during 2026-05-15 grilling. Glossary integrity is workflow-load-bearing, not just developer-facing documentation.
- **Brewing iteration budget is bound by roasted-bean inventory** — typical purchase is 125-500g of roasted coffee, allowing ~8-30 physical brews per bag. The ~3-brew average iteration loop leaves headroom but not infinite — narrower than the roasting side's ~10-V-set budget per green lot. Drives the brewing workflow's bias toward one-shot success and tight iteration.
- Brewing is **subjective like roasting** — there's no objective "best" brew, only the recipe Chris commits to when iteration deltas flatten. Same single-palate framing as the roasting side.
- The brewing iteration loop **converges within a strategy zone, not across strategies**. A wrong-zone strategy choice at Coffee Brief Step 1d traps the loop in suboptimal territory regardless of how many turns it runs — iteration tightens within a strategy, it does not transit between strategies. **Canonical example: the Jeferson Motta anaerobic-washed Gesha** — pre-framework, Chris ran clarity-first iteration to no avail (sour / flat / grassy) before realizing Full Expression was the right zone. Tasting the coffee at Moonwake (where it was roasted) surfaced the candied strawberry / hibiscus profile that Full Expression unlocked. This is why the Coffee Brief confirmation gate matters: it's the only opportunity to catch a wrong-zone strategy before burning iteration budget.
- **Reference brew (purchased context) and optimized brew (self-roasted context) are the same artifact-shape** — same brewing iteration loop, same Step 4 resolved-brew row, same canonical-validation pass. The naming distinction reflects which side of the pipeline you're looking from: reference brew is the terminal output for a coffee that started as a purchased roasted bean; optimized brew is the terminal output for a coffee that started as a green lot, was roasted to a reference roast, and then handed into the brewing workflow.
- **The xBloom reference cup and the optimized brew are intentionally separate cup-tasting moments on the self-roasted pipeline**. Reference cup is **mechanically constrained** (same xBloom recipe every time — designed to eliminate brewing-side variance so cup-side signals can be isolated for picking the reference roast). Optimized brew is **mechanically unconstrained** (full lever range — real dripper, real filter, full strategy + modifier toolbox — designed to make the bean shine). They do different jobs: reference cup is calibration, optimized brew is consumption-condition.
- A **resolved** state applies to entities on both sides of the workflow: a green-bean lot is **resolved** when reference roast + optimized brew are both locked (roasting lifecycle terminal state); a brew row is **resolved** when it has been written via Step 4 and synced to the app. Same word, different scope — lot-level on the roasting side, per-row on the brewing side.
- **Latent records only the resolved brew, not the iteration trace.** Intermediate brews + tasting deltas + adjustment reasoning live in the claude.ai conversation thread. This is asymmetric with the roasting side (which records every batch slot within a V-set) and is a flagged ambiguity Chris is open to reconsidering.
- **The geography mental model: strategy = country, sub-form / modifier = state, iteration target = city.** Strategy choice picks the country; Hybrid sub-form (when applicable) and modifier stack pick the state inside that country; the iteration loop dials within state/city granularity. Pre-Coffee-Brief, Chris frequently ended up in the wrong country (Clarity-First when the coffee needed Full Expression) and could only iterate within that wrong country. Post-Coffee-Brief, the country is usually right by Step 1d and iteration is purely state/city dialing.
- **The Coffee Brief subsumed Extraction Confirmed's original purpose**. Pre-Latent, Chris added the `extraction_confirmed` field to catch wrong-zone strategy divergence after the fact (Jeferson Motta lesson). Today, prior-archive lookup + WBC corpus check + roaster brew guide cross-check + the BREWING.md doc-based reasoning catch wrong-zone calls at Step 1d, *before* iteration starts. Extraction Confirmed remains useful when it does fire — that signal means the upstream-prevention layer missed something specific worth carrying forward — but it's expected to fire rarely.
- **Extraction Confirmed and Modifiers Confirmed are asymmetric for historical-vs-experimental reasons**, not binary-vs-interpretive reasons. Extraction Confirmed is a **legacy field** whose original wrong-zone-gut-check role has shifted upstream into the Coffee Brief; Modifiers Confirmed is **active experimental territory** Chris is leaning into more, since modifiers are slight-variation add-ons whose effect needs per-brew validation. Different histories drive different populate-rules.
- **Three signal sources feed the Coffee Brief**: **variety signal** (from cultivar genetics, read at Step 1c), **process signal** (from processing intervention depth, read at Step 1b via the Process/Variety Signal table), and **roaster signal** / **house style** (from the roaster's persistent extraction direction + published brew guide). All three converge at Step 1d when the extraction strategy is proposed.
- **Default = variety-driven**; **signal override** is judgment-based at Step 1c, not strict precedence. Three reliable override patterns to remember: (1) intervention-heavy process overrides variety (anoxic naturals, heavy co-ferments, Moonwake Lord Voldemort); (2) strong roaster house style overrides variety + process (DAK, Sey, Picolot Garrido naturals); (3) transparency-driven variety overrides mild process (Sydra TyOxidator, washed Geshas).
- **Strong house styles name specific strategy zones**, not generic preference strength. Sey is Extraction-Push-leaning (light + high extraction + low agitation = clean but pushed); DAK is Full Expression (push to get the cup out); Picolot raised-bed Garrido naturals are Full Expression with fast/fast/slow valve discipline. The override target is encoded in the roaster's strategy tag in [docs/brewing/roasters.md](../../docs/brewing/roasters.md).
- **The heaviest-process-wins-everything pattern is canonical**: Moonwake Lord Voldemort (yeast-inoculated Ombligon, Colombia) is the reference example where the process is intervention-heavy enough that it overrides cultivar + terroir + the roaster's own published brew guide. When a coffee is "its own beast," all other signals defer to the process-driven extraction strategy choice.
- **The WBC Thermal Systems family maps to four distinct surfaces in Latent**: the **Inverted Temperature Staging** modifier (low→high temp across phases), the **Aroma Capture** modifier (chilling-ball early-phase volatile lock-in), the **Cooling-Curve Design** Named Consideration (evaluate / serve at a non-default peak temp), and the **Hybrid (Temperature-Staged)** sub-form (phase boundary coincides with temp change). All four engage temperature as a primary lever but at different architectural layers (modifier / consideration / sub-form). When a brief lands in Thermal Systems territory, the right surface depends on whether the temperature move is layered on top of a strategy (modifier), informs evaluation/service timing (consideration), or defines phase structure (sub-form).
- **Cooling-Curve Target operates at three moments** (brewing-time / evaluation-time / service-time), only two of which intersect the Latent data model. Service-time temp management (PC-fan rapid-cool to ~55°C before pouring for guests) is a real Chris-side discipline but is not logged on the brew row today.
- **Aroma Capture's observed cooling-arc artifact shifts peak warmer** (~48-50°C from a non-modifier ~45°C baseline) — the *inverse* direction from cool-window-dependent coffees. Aroma Capture is therefore the wrong tool for coffees with established cool-window dependencies (El Paraíso thermal-shock, Garrido naturals, etc.). The cooling-arc artifact was first observed on Pepe Jijón Sidra Wave Hybrid (2026-05-07).
- **`process_dominant` is an aggregation trust signal**, not a Step 1d signal-arbitration reflection. It marks brews whose learnings should NOT aggregate to cultivar / terroir / roaster pages because the process is too dominant for any non-process generalization to apply. Tightly correlated in practice with process-signal-overrode-at-Step-1d but conceptually a different surface.
- **The compounding-knowledge motivation predates the Latent app**: the precursor claude.ai-local-app that became Latent pulled three context blocks per brew (roaster / terroir / cultivar) specifically so cup-side learnings could aggregate into per-axis lesson sheets — the proto-form of today's `/cultivars`, `/terroirs`, and `/roasters` aggregation surfaces. `process_dominant` was invented to keep heavy-process brews from skewing those aggregations.
- **Roasting has no equivalent process-dominant flag** today, even though Lord-Voldemort-tier green-bean lots could in principle skew the carry-forward learnings the same way. Open follow-up: does roasting need a `process_dominant` analog on `green_beans` or `roast_learnings`? Or is the per-cultivar / per-terroir aggregation on the roasting side robust enough at current corpus sizes that it's not load-bearing?
- **Role separation has four parties**, all treated as first-class actors with shared language: **claude.ai** (input — the `claude.ai/projects` surface, authors every entity + every doc proposal), **Claude Code** (builder + arbiter — same model, two operational modes), **Latent** (the system itself — Next.js + Postgres + the MCP server, source of truth for state), **Chris** (physical execution — roast / brew / cup / taste / judge, no record authoring). The MCP-only input principle enforces the boundary between input and physical execution; the builder/arbiter boundary is internal to Claude Code.
- **The Latent MCP server is the substrate of role separation** — without it, the input surface would be the deprecated `/add` forms and the role boundary would erode. It exposes Tools (writes + Tool-shaped reads) + Resources (URI-addressable read-only catalog) under one HTTP transport on the same Vercel deployment as the app.
- **Single-tenant + single-user is permanent** (Chris, 2026-05-15): "even in the long, long term, there's nothing more than just me." No multi-tenancy plan, ever. Multi-actor framing (claude.ai + Claude Code + Latent + Chris as four parties) is the only "multi-user" sense the system has.
- **claude.ai is the dominant active client of the Latent MCP server** via the web + mobile apps (OAuth 2.1 + PKCE auth path); Claude Code is the secondary client (bearer-token auth path). The auth path split is **client-protocol-driven, not trust-tier-driven** — both paths resolve to the same `user_id`, same RLS scope, same Tool surface.
- **Tool and Resource have an asymmetric discoverability surface** in claude.ai's MCP client — Tools are model-callable, Resources are catalog-only. The **dual-surface pattern** (same payload as both a Tool and a Resource) exists because critical reads need to be model-callable; `read_canonical` + `canonicals://{axis}` + `read_doc` + `docs://*` are the canonical instances.
- **Asymmetric write trust has two tiers + one variant**: direct write (canonical-registry field-picking) trusts claude.ai fully; propose-then-apply (free-form prose mutations) routes through `doc_proposals` for Claude Code arbitration; canonical-promotion writes (net-new taxonomy values) route through `taxonomy_overrides_queue` for the same arbiter. The asymmetry is the load-bearing call of SYNC_V2 Decision #5.
- **The arbiter procedure walks both queue types in one session** — there's one trigger phrase ("process pending arbitration") and one playbook (ARBITER.md), with axis grouping per queue type. Future curation surfaces (null-field reviews, missing-entity flags) can attach to the same trigger without changing the procedure shape.
- **Drift (slow-motion canonical naming inconsistency) and category bloat (over-promotion of canonical entries)** are paired failure modes that asymmetric write trust + the arbiter procedure prevent. State bloat is the doc-layer analog (un-curated edits accumulating in living docs). All three failure modes were experienced firsthand in Chris's pre-Latent spreadsheet workflow, compounded once his EA Belle joined the loop, and motivated the MCP-only input principle.
- **The MCP-only input principle has paired motivations**: (1) opportunity — every claude.ai input flows through `propose_doc_changes` and canonical-validation, contributing to the System self-improvement loop; (2) constraint — the schema complexity has outgrown human-tractable input. Going back to manual input is functionally not viable.
- **find-or-create starts with alias lookup, not canonical lookup** — the first step is `CanonicalLookup`'s alias-resolution map (e.g. "Geisha" → "Gesha"). Only after alias resolution fails does the lookup check canonical names directly. NET-NEW is the branch hit when both resolutions fail.
- **Promotion at arbiter-time always carries a full canonical entry** — when Chris-as-arbiter promotes a NET-NEW queue row to canonical, the full rich registry entry (27 producer fields, 29 roaster fields, etc.) gets populated at that moment, not partial-now-fill-in-later. The discipline applies across all rich-registry axes; the promotion moment is the only one where the entry's context is fresh.
- **Cross-system consistency is the meta-rule that makes everything else work** — claude.ai, Claude Code, and Latent must share vocabulary + canonical taxonomy + operational rules. Sync is maintained by live Resource reads + validation-on-push + `propose_doc_changes` + the arbiter procedure. CONTEXT.md exists, is a Resource, and grows via `/grill-with-docs` sessions specifically because the principle needs a shared-language artifact to anchor on.
- **The Roest API integration is the only external-device surface** on the Latent MCP server today. Six Tools span read (logs / inventory) + write (recipes / inventory). The roast recipe push capability is the killer feature — pre-integration, claude.ai handed Chris recipes as prose and Chris manual-typed per-batch (V1a / V1b / V1c with minute differences); post-integration, claude.ai writes them straight into the machine. Roest's published API only covers read; write was reverse-engineered via a third-party GitHub project. xBloom + kettle + brewer + grinder are all manual / no-API.
- **No real-time push from app → claude.ai is planned, ever.** Roest profiles support drop-on-bean-temperature with drop rules (`drop_rule_if_fast` + `drop_rule_if_slow`); the machine handles mid-roast decision logic locally, claude.ai doesn't need real-time data pulling.
- **Aggregation level is picked per-axis against three criteria** (legible / enough samples / transferable). Different axes pick different levels: terroir at macro (between country-too-broad and farm-too-specific), cultivar at individual cultivar (post Sub Pages 3 inversion from lineage), process composable, roaster + producer + flavor + roast-level at individual canonical. **Not every axis is consumed as a knowledge-aggregation surface in the same way** — the **Aggregation-eligibility** trichotomy (page-bearing / page-deferred / validation-only) classifies the 10 axes' relationship to aggregation differently from their canonical granularity.
- **The 4 page-bearing axes today** (terroir / cultivar / process / roaster) each earn a destination page in the Latent app with per-page synthesis running on the axis's anchor. **Producer is the 1 page-deferred axis** — rich-registry scaffolding exists, no page yet built, but treated as a first-class aggregation surface in Chris's mental model. **The 5 validation-only axes** (brewer / filter / grinder / roast level / flavor) carry canonical registries for vocabulary discipline / equipment inventory / cross-roaster comparability, not for knowledge aggregation.
- **Producer is unique in carrying a second-order role** (the **Producer sourcing surface**) — producers are not just an aggregation unit but a relationship network Chris builds toward for championship-mode green-bean sourcing. **Growing the bean is a bigger lever than roasting or brewing** in cup attribution, and producer-relationships are Chris's only access path to that lever. Importance asymmetry across the pipeline: producer matters more on the roasting side (closer to agricultural input + sourcing pipeline) than the brewing side (informational on purchased roasted beans).
- **Every canonical-registry axis is authored across two coupled files** — the **Authoritative source** (markdown under `docs/taxonomies/<axis>.md`) IS the taxonomy; the **Validation mirror** (TypeScript under `lib/<axis>-registry.ts`) mirrors it into in-line code-callable validation. The pair is maintained via the **2-step deliberate edit** discipline (3-step when an existing DB row needs a rename migration). When the two disagree, the markdown wins; an automated registry-sync check is queued (`check:registry-sync` script shipped PR #112, hook + GH Action pending).
- **All authored canonical taxonomies follow the same Bottoms-up authoring pattern** — bottoms-up data collection from the existing corpus → choose a tentative aggregation layer + draft taxonomy → test against the collected data (flag **substrate gaps**) → iterate until most cases fit cleanly → continuous refinement on new data. Used across cultivar, terroir, flavor, process, producer. ChatGPT served as Chris's research assistant for the data-collection + initial-draft phases (bounded to taxonomy authoring; not part of role separation).
- **Two distinct hierarchies of structural shape exist** above the per-axis canonical entry: **Hierarchical taxonomy** (cultivar's 4-level Species/Family/Lineage/Cultivar; terroir's 3-level Country/Admin/Macro) where parent levels are render-context-only; **Composable taxonomy** (flavor's 3-axis BASE+MODIFIERS+STRUCTURE_TAGS; process's 8-field base+subprocess+4-modifier-arrays+decaf+signature) where the canonical value is a structured composition. Other axes (producer / roaster / brewer / filter / grinder / roast level) are flat — one canonical name per entry.
- **Aliases are additive-only across all axes** with three structural shapes: **drift alias** (mechanical surface variation of the same canonical thing — case / diacritic / hyphenation), **structural alias** (Chris-arbiter judgment collapsing conceptually-distinct surface names to one canonical — producer naming overload is the heaviest case), and **decompositional alias** (process-specific — resolves a surface phrase to a structured multi-axis composition).
- **Coverage strategy varies per axis** along four named strategies: **comprehensive** (cultivar / terroir / roast level / flavor-with-arbiter-caveat), **tier-scoped** (producer / roaster / signature method), **owned-comprehensive** (brewer / filter — 100% on owned + opportunistic on unowned), **self-only** (grinder + the in-BREWING.md SWORKS valve flow taxonomy + the future filter flow-rate measurements + the future water taxonomy). Coverage strategy is asymmetric because the real-world domain shape + Chris's relationship to it differ per axis.
- **The Pick-not-author principle is what the canonical-registries cluster collectively serves** — claude.ai's runtime experience should be picking from a pre-resolved canonical surface, not authoring or researching. The 95-96% completeness target is asymptotic; new-canonical rate is expected to decay as exposure broadens, and the registries are converging assets.
- **Provenance + override-flag are dual mechanisms for the same concept** — FK-creating axes (cultivar / terroir) use `*_provenance` enum columns (`canonical` | `auto_created`); text-only override-eligible axes (producer / roaster / brewer / filter / grinder) use `*_override:true` boolean parameters + queue entries in `taxonomy_overrides_queue`. Same outcome (record canonical vs lazy-materialized), different storage shapes determined by axis storage.
- **Skeleton entries are canonicals promoted with thin content + `skeleton: true`** — the resolution to "promote now vs. wait until research is done." Surface in the arbiter procedure as a third queue type (alongside `doc_proposals` + `taxonomy_overrides_queue`) so the arbiter run can sweep + backfill them. Producer is the only axis using the flag today (Miguel Estela is the lone instance); pattern extends to other rich-registry axes.
- **Family palette is the recurring color-coded parent-level grouping** across 5 axes (roaster strategy families / cultivar genetic families / flavor families / process base / country). Hue-not-lightness rule; informational not decorative — color carries semantic encoding (clarity-first → light delicate; extraction push → red; suppression → slate). Distinct from Parent level (hierarchical-only); distinct from Aggregation level (always above the canonical unit).
- **Strategy tag is the roaster-axis-specific leaf-positioning** that collapses to family (10 tags + SELF-ROASTED → 6 families via `STRATEGY_TAG_FAMILY`). The `→` arrow syntax encodes skew direction (`BALANCED → FULL` = home position BALANCED, pushes toward FULL when coffee + process call for it). **Cross-axis vocabulary coherence**: strategy tag vocabulary on roaster should align with extraction strategy canonical names on brews — partial alignment today, full coherence flagged for a follow-up.
- **Reference role + Producer system are producer-axis-specific structure** — Reference role (Anchor / Signal / Experimental + implicit Tier 4) names the producer's calibration purpose + doubles as a sourcing-accessibility ladder (T3 Experimental is the most likely sourcing target AND the most-novel-processing camp); Producer system (6 named meta-clusters) is typed metadata grouping producers by processing-system + regional context — deliberately NOT a future aggregation surface, distinct from terroir geography.
- **The aggregation-layer switch is a named pattern** Chris uses when a taxonomy's original aggregation-level choice turns out to be unbounded — pivot to a more abstract layer that can converge while accepting an implicit long tail. Producer is the canonical example (pivoted from "all producers" bottoms-up exhaustive to "Anchor producers across regions I cover" + implicit Tier 4). Signature method is a likely future candidate if NET-NEW rate stays high.
- **Marketing tag vs Objective bucket is the roast-level-specific split** — `brews.roast_level` records the measured / inferred objective bucket (Agtron-anchored, 8 buckets); marketing tags like Nordic Light / Ultra Light / Modern Light live as alias-only resolution to objective buckets, with the structural intent of relocating positioning to `roaster.roastStyle` (sprint 1h.3). Two resolution paths into the objective bucket: **measured** (CM200 reading → SCA color word) and **inferred** (bag word + roaster-typical roast style).
- **No anchor-level URL convention on `docs://context.md`** (locked 2026-05-16). The Roasting cluster open question — *should claude.ai address specific glossary entries via `docs://context.md#V-set`-style anchors?* — resolved as **no, don't ship**.
